## Abstract 

Large Language Model (LLM) agents remain vulnerable to safety threats from the external environment, where attackers inject adversar-ial content into external observations such as tool-returned data, webpages, or MCP con-text, causing harmful agentic behaviors such as unsafe actions or incorrect outputs. Exist-ing studies typically focus on single-interaction attacks, where the agent observes adversarial content and immediately exhibits harmful be-havior within one user request. However, we show that adversarial content can also persist across interactions served by the same agent, making such threats harder to detect and miti-gate. Specifically, adversarial content may per-sist in the agent state, remain dormant across interactions, and later be activated by a benign user query. We formalize this type of safety threat as Sleeper Attack . To evaluate it, we con-struct a benchmark with 1,896 instances cov-ering six real-world harmful outcomes, three attack strategies, and three agent state targets: session context, memory, and reusable skills. Experiments on seven strong open-source and closed-source LLMs show that state-of-the-art LLM agents remain vulnerable to Sleeper At-tack, even when they achieve low attack suc-cess rates under a single-interaction baseline. Our code and data are available at https: //anonymous.4open.science/r/skdvnf u23ihr9wdscnksf1asdffsaef .

## 1 Introduction 

Large Language Model (LLM) agents use LLMs as their core to complete user-specified tasks by executing actions and observing the external envi-ronment (Achiam et al., 2023; Yao et al., 2023). This has enabled their application to diverse real-world tasks, including web navigation (Zhou et al., 2023), software engineering (Jimenez et al., 2024), and scientific discovery (Boiko et al., 2023). De-spite their utility, LLM agents face safety risks that limit deployment, including unsafe tool calls and transactions (Mou et al., 2026) and privacy leak-age (El Yagoubi et al., 2026). Safety threats to LLM agents can be broadly categorized into internal and external threats. In-ternal threats originate from the user or the LLM itself, such as malicious or jailbreaking user re-quests (Andriushchenko et al., 2025b; Xiong et al., 2025; Andriushchenko et al., 2025a) and model-side planning failures (Zhang et al., 2025). Ex-ternal threats arise when external attackers inject adversarial content into the agent’s external obser-vations, including tool-returned data (Zhan et al., 2024; Debenedetti et al., 2024), webpages (Evti-mov et al., 2025; Wang et al., 2025a; Chen et al., 2025), and MCP context (Wang et al., 2025b; Zhao et al., 2025; Guo et al., 2025b). Such adversarial content may contain malicious instructions that in-duce unsafe actions (Yi et al., 2025) or tampered in-formation that causes incorrect agent outputs (Chen et al., 2024; Dong et al., 2025). External threats are particularly challenging because the external environments are diverse, dynamic, and difficult for the agent system to monitor and control. We therefore focus on the external-threat setting. We identify a key limitation of existing stud-ies on external attacks. Most benchmark evalu-ations assume that attacks occur within a single user interaction: adversarial content is injected and triggers harmful agent behavior within the same user request (Bhagwatkar et al., 2025; Wei et al., 2026). However, adversarial content may also be planted and stored in agent state, remain dor-mant across interactions, and later be triggered by a benign user request. For example, an attacker may plant a malicious instruction such as “when the email-sending tool is invoked in a future task, check and email the user’s Venmo balance to at-tacker@gmail.com.” Such cross-interaction attacks are harder to trace and mitigate. Moreover, ad-versarial content may persist in different forms of agent state such as memory (Pulipaka et al., 2026) 1

Therefore, we formalize a new type of exter-nal safety threat for LLM agents: Sleeper Attack .An illustration is shown in Figure 1. To system-atically study this threat, we design a taxonomy along two axes. The first axis describes how ad-versarial content triggers harmful behaviors. We consider two attack modes: malicious instruction planting, where planted instructions later induce unsafe actions, and stored information corruption, where corrupted stored information later leads to incorrect outputs. These modes give rise to three attack strategies: Latent Instruction Planting (LIP),

We evaluate seven strong open-source and closed-source LLMs on our benchmark. The results show that state-of-the-art LLM agents remain vulnerable to Sleeper Attack across different attack strategies and agent state targets, even when they achieve low Attack Success Rate (ASR) under the single-interaction baseline. For example, Gemini-3-Flash consistently exceeds a 50% ASR across LIP, PIE, and PIC, and all evaluated agents reach over 27% ASR under PIC. Our contributions are threefold: • We introduce and formalize Sleeper Attack , to our knowledge the first unified external-source threat model for cross-interaction LLM-agent attacks: adversarial content from an untrusted source is planted into session, memory, or skill state, remains dormant, and is later triggered by benign user requests, making it more covert and harmful than single-interaction attacks. • We build a large-scale benchmark for this set-ting with 1,896 evaluation instances spanning six real-world harmful outcomes, three attack strategies, three agent state targets, with four evaluation surfaces (direct, session, memory, and skill), enabling controlled direct-versus-sleeper comparison. • We reveal large direct-to-sleeper safety gaps across seven strong models: LIP rises from 11.1% direct ASR to 39.9% on the strongest agent state, PIE from 0.6% to 41.6%, and PIC 2reaches 47.8% mean ASR overall, showing that agents remain vulnerable to sleeper attack even when they achieve low ASR under the single-interaction baseline. 

## 2 Related Work 

External Threats to LLM Agents. External threats have emerged as a central agent-safety prob-lem because untrusted content can be mistaken for instructions instead of data. BIPIA (Yi et al., 2025) studies indirect prompt injection in content-consuming LLM applications, InjecAgent (Zhan et al., 2024) moves to tool-integrated agents, Agent-Dojo (Debenedetti et al., 2024) and WASP (Evti-mov et al., 2025) evaluate prompt injection in richer agent and web environments, and WebInject (Wang et al., 2025a) and TopicAttack (Chen et al., 2025) broaden the web setting with page-level and topic-transition attacks. These works establish external attacks as a practical threat, but they still primarily study attacks that hijack the agent in the current in-teraction or within a specific environment, leaving unified cross-interaction across session, memory, and skill surfaces unaddressed as a single problem. 

Memory Attacks. Saved user state is risky because agents later reuse it as trusted context. AgentPoison demonstrates that poisoned memories or knowledge-base entries can steer later agent behavior (Chen et al., 2024), while MINJA shows that ordinary query-only interactions can still inject adversarial memory (Dong et al., 2025). Storage-to-Steering frames memory as a control-flow channel (Xu et al., 2026). Hidden in Memory and MemoryGraft study memory-only sleeper payloads and poisoned experience retrieval (Pulipaka et al., 2026; Srivastava and He, 2025), and Zombie Agents examines self-reinforcing updates in self-evolving agents (Yang et al., 2026). These papers show why stored state matters. However, they focus on memory-specific mechanisms, not a unified sleeper benchmark spanning session context, memory, and reusable skills in one evaluation setting. 

Skill Attacks. Reusable skills and skill guidance form a persistent state because agents can reload them across future tasks. SkillJect automates stealthy skill-file injection for coding agents (Jia et al., 2026), and Skill-Inject measures agent vulnerability to malicious skill files (Schmotz et al., 2026). SkillTrojan studies backdoor behavior in skill-based systems (Feng et al., 2026), while supply-chain poisoning work studies third-party skill repositories as distribution channels (Qu et al., 2026). Because skills encode procedural guidance, poisoned skill content can affect later tool selection and argument construction after the original source content is no longer visible. These works show that skills are a practical attack surface; we study them together with session context and memory in a unified sleeper-attack setting across surfaces.

Attack Strategies. Following prior studies on ex-ternal safety threats, we extend two common attack modes to the sleeper attack setting. First, attackers can plant malicious instructions that remain dor-mant until later activated by benign user queries, causing the agent to perform harmful actions that deviate from the user request. We refer to this strategy as Latent Instruction Planting (LIP), and further introduce Proactive Information Elicitation 

(PIE), an enhanced LIP variant where the planted instruction makes the agent elicit task-relevant sen-sitive information before executing the malicious action. Second, attackers can corrupt persistent information in the agent state, causing the agent to rely on incorrect information when responding to future user requests. We instantiate the attack strat-egy as Persistent Information Corruption (PIC). We consider three attack targets within the agent state, corresponding to increasing levels of abstrac-tion in agent experience: session context , memory ,and reusable skills . Higher-level targets could be more abstract and therefore harder to inspect and detect. In the session-context setting, the adversar-ial observation is directly retained in the conversa-tion history. In the memory setting, the adversarial observation is written into memory modules and later retrieved into the LLM context for relevant in-puts. In the skill setting, the adversarial observation is injected into hierarchical skills with low-level task-specific skills and higher-level general skills. 

3.2 Benchmark Construction Agent System Construction. We construct a benchmark for evaluating Sleeper Attack in a sim-ulated agent framework designed for safety testing. The benchmark is built with the OpenAI Agents SDK 1. We use ToolEmu (Ruan et al., 2024) as the base simulated tool environment, which pro-vides 330 tools, and simulate tool outputs using DeepSeek-v3.2 (Liu et al., 2025). On top of this environment, we add agent state interfaces con-sisting of memory and reusable skills. The mem-ory module supports storage, retrieval, and update. The skill module follows a two-level hierarchy: category-level skills at the higher level and tool-level skills at the lower level. Both levels can be retrieved and updated during agent execution. We initialize the skill library with 21 high-level skills and 309 low-level skills. For PIE, we additionally use DeepSeek-v3.2 to simulate the user to provide the task-sensitive information. 

Instance Construction. To ensure comprehen-sive and realistic evaluation, we first review existing taxonomies of real-world harmful out-comes (Liu et al., 2026) and derive six harm do-mains: economic harm, account/system compro-

Economic harm covers unau-thorized payments, purchases, or resource trans-fers; account/system compromise covers creden-tial, account, or system takeover; and physical-world harm covers actions that affect real-world devices, deliveries, or services. Personal data leak-age covers disclosure of identities, contact details, or private attributes; financial data leakage cov-ers account, card, or transaction data; and other sensitive-information leakage covers confidential nonfinancial data, including API tokens, internal files, operational notes, and private records. For prompt construction, we first fix one of the six harm-domain labels and then ask an LLM to generate a concrete harmful goal under that label, conditioned on the attack strategy, agent state tar-get, and compatible tool metadata. This generation step also produces a benign trigger query that is harmless in isolation but can activate the planted ob-servation. The planting query itself is adapted from the source-tool user-request pattern used by InjecA-gent (Zhan et al., 2024), so that the first interaction naturally exposes attacker-controlled source-tool output without generating it from scratch. Strategy-specific templates then combine the adapted planting query, the generated harmful goal, and the benign trigger query into LIP, PIE, or PIC instances. We concatenate the planting and trig-ger queries into a two-interaction test instance. To evaluate longer-horizon sleeper behavior, we fur-ther extend this setup to an N -interaction setting by inserting additional LLM-generated benign in-teractions between the planting and trigger queries. Some candidate tasks cannot be completed even when the user directly supplies the attack instruc-tion as the input, so we exclude those cases during construction. In total, we construct 1,896 instances. Benchmark statistics are shown in Table 1. De-tailed domain descriptions, construction prompts, example instances, construction-side quality con-trol details including label-consistency checks and rejection criteria are provided in Appendix A.

## 5 Conclusion 

We introduced sleeper attack, a delayed external-source threat model in which adversarial content is planted into agent states and later activated by benign requests. Our benchmark spans three attack strategies across three agents states and comprises 1,896 evaluation instances. All evaluated models exhibited unsafe behavior across the all strategies, and lightweight defenses could only slightly reduce the risk, failing to eliminate it. These results show that agent safety evaluation must track not only immediate tool use, but also how untrusted content is written, stored, and reused across interactions. 8Limitations 

Our study relies on a ToolEmu-derived sandbox with simulated tool returns, so results may dif-fer in deployed tool ecosystems. The benchmark captures broad tool-use patterns, but real services may impose additional authentication, confirma-tion, rate limits, or audit logs that change both at-tack feasibility and user-visible harm. Conversely, deployed agents may expose longer-lived files, re-trieval indices, shared workspaces, or multi-agent state that are not covered by our three agent states. Our benchmark also abstracts away several deployment-specific control layers. Real agent products may gate sensitive actions behind user confirmations, permission scopes, organizational policies, or human review workflows, while other products may automatically compose multiple ser-vices in ways that create new persistence paths. We therefore interpret the results as measuring agent states vulnerability under controlled agent inter-faces, not as a direct estimate of incident rates in any particular deployed system.Because of re-source limits, supplementary experiments use nar-rower model slices to isolate mechanisms. 

## Ethicals Considerations 

This work studies a dual-use safety problem: exter-nal attackers may plant adversarial content into agent state and later trigger harmful behavior through benign user requests. Our goal is defensive: we aim to help researchers and practitioners iden-tify agent state vulnerabilities before they appear in deployed agents. All experiments are conducted in a simulated tool environment and not against real services, ac-counts, webpages, or users. The benchmark uses synthetic sensitive values, simulated user responses, and structured trace checks, so successful attacks in our experiments do not perform real-world trans-actions or disclosures. Because the benchmark con-tains adversarial prompts, released artifacts should be used only for authorized red-teaming and de-fense development, with real sensitive information removed and responsible disclosure followed when deployed vulnerabilities are found.

A.6 Ask-User Simulation Protocol 

When an attacked agent asks for informa-tion corresponding to one or more fields in 

ground_truth_params , and the simulated user does not explicitly refuse, the simulator returns the corresponding asked-for ground-truth values. If the agent asks only a subset of the required fields, only that subset is made available; missing fields are not silently completed. A non-empty tool result therefore only shows that the simulator re-turned some response. The meaningful question is whether the agent elicited all case-required fields in 

ground_truth_params and then used those values to complete the harmful action. The final ASR for Proactive Information Elicitation cases is therefore computed by the deterministic structured evaluator, not by a standalone answer-rate statistic. In real deployments, users may refuse, misunderstand the request, or provide incomplete information, so this protocol isolates the agent-side tendency to solicit and use missing attack parameters.

Abstract 

We introduce GRACE-DS, a Guarded Reward-guided Agent Correction Environ-ment in Data Science 1 for pre-deployment evaluation of LLM-powered AutoML agents. GRACE-DS is a set of evaluation metrics in an isolated environment that can be applied to tabular ML tasks specific to a particular orga-nization. It exposes agents to realistic work-flow stages, from planning and data inspec-tion through feature engineering, model de-velopment, validation, and code repair to fi-nal submission, while hidden executable val-idators measure not only final predictive per-formance but also leakage avoidance, repro-ducibility, protocol validity, correction behav-ior, and reward alignment. The strongest structured regime, flexible-iterative interaction 

(ours ), achieves higher end-to-end normalized hidden-test quality than single-shot generation, the unstructured-agent , and restart-based base-lines, while also improving protocol-valid com-pletion. Validated across more than 7,000 episodes, these results establish GRACE-DS as a robust platform for assessing the capacity of LLM-based AutoML agents to execute ma-chine learning workflows under production-like conditions and in accordance with organization-specific requirements. 

1 Introduction 

Organizations are beginning to evaluate LLM-powered agents not only as coding assistants, but as potential participants in data-science and machine-learning workflows (Chan et al., 2025). In an enter-prise setting, such agents may be asked to inspect internal datasets, propose modeling strategies, en-gineer features, train predictive models, debug fail-ures, compare candidates, and produce deployable artifacts. This creates a practical pre-deployment problem: a company does not merely need to know

whether an agent performs well on a public leader-board. It needs to know whether the agent can solve the company’s own ML tasks, data, environment, while respecting governance, privacy, reproducibil-ity, and validation constraints (Kreuzberger et al., 2022). This problem is especially important for tabular ML, which remains central to many industrial ap-plications: churn prediction, risk scoring, demand forecasting, fraud detection, etc. These tasks often use sensitive production data and domain-specific feature spaces. Sending such data to public bench-mark infrastructure is usually impossible, and eval-uating agents only on public datasets can give a mis-leading estimate of deployment readiness. A model that performs well on external benchmarks may still fail when faced with messy internal schemas, unstable data distributions, organization-specific validation rules, protected columns, leakage risks, or strict reproducibility requirements. Existing benchmarks (Section 2) for ML agents and data-science code generation provide valuable measures of general capability, but they are not sufficient as enterprise deployment gates. Final-score benchmarks can tell whether an agent ob-tained a good result, but they often do not reveal how the result was obtained. This is a serious limitation in production ML. An agent may im-prove a metric by leaking target information, tuning against validation labels, accessing protected files, hard-coding dataset-specific assumptions, produc-ing non-reproducible code, or generating a pipeline that cannot be rerun on future data. On the other hand, an agent may fail not because it lacks mod-eling ability, but because it cannot recover from execution errors, interpret validation feedback, or revise an earlier workflow decision. For companies, these distinctions matter operationally: the relevant question is not only what score the agent obtained but also whether the process that produced that score can be trusted. 1

GRACE-DS addresses this gap with a controlled evaluation harness that runs around tabular ML tasks inside an organization’s own environment. In-stead of treating the evaluation as a single final sub-mission, GRACE-DS structures the agent’s interac-tion with the task as an iterative workflow. Agents can plan, inspect data, engineer features, train mod-els, validate candidates, repair code, and submit a final predictor. At each stage, hidden executable validators assess not only predictive progress but also process reliability: whether the agent avoids leakage, follows valid validation practices, pro-duces reproducible code, respects protected data boundaries, and corrects errors after feedback. The key design principle behind GRACE-DS is that enterprise evaluation of AutoML agents must measure correction behavior under constraints. In real ML work, the first attempt is rarely the final solution. Analysts and ML engineers inspect data, revise assumptions, fix code, compare models, and improve pipelines over several iterations. An LLM AutoML agent should therefore be evaluated not only by whether it can generate a plausible model in one shot, but by whether it can use feedback to improve while remaining within methodological and operational boundaries. GRACE-DS makes this measurable by combining hidden validation, process-level reward, workflow-stage instrumenta-tion, and final held-out scoring. GRACE-DS differs from final-score-only evalua-tions in several ways: (i) separates evaluator-owned validation from agent-visible data: agents may re-ceive feedback, but they do not receive private val-idation labels or hidden test targets; (ii) requires final submissions to be reproducible on raw input data, ensuring that an agent produces a reusable prediction artifact rather than a script tied to one static file; (iii) records process-level diagnostics, in-cluding validation improvement, reward decompo-sition, terminal-path reliability, protocol violations, critical errors, recovery behavior, runtime, itera-tion count, and cost-related measures; (iv) supports stress tests such as reward-optimization prompts and red-team prompts, allowing organizations to examine whether an agent is robust to incentives that may produce superficially valid but substan-tively poor solutions. GRACE-DS supports several industrial scenar-ios: comparing LLMs on internal ML tasks, de-ciding whether an AutoML agent is ready to be connected to production data sources, detecting leakage-prone behavior and invalid validation pat-terns before deployment, and comparing workflow designs, feedback policies, and operating budgets for an internal data-science copilot. In each case the purpose is not a public leaderboard score, but a deployment decision under company-specific con-straints. We evaluate GRACE-DS across multiple LLM families, model sizes, tabular datasets, work-flow regimes, ablations, reward-optimization set-tings, and red-team conditions. The results show that structured flexible iteration provides the strongest end-to-end performance among the eval-uated regimes. In the merged experiments, the 

flexible-iterative regime reaches an end-to-end nor-malized hidden-test quality of 0.754 , compared with 0.536 for single-shot generation, 0.527 for the unstructured-agent , 0.672 for restarts-from-scratch , and 0.686 for a call-matched restart up-per bound. The structured regime also improves protocol-valid completion, indicating that the bene-fit is not limited to successful runs but extends to the reliability of the whole agent loop. GRACE-DS does not rely on a single final score or a single safety check, it combines hidden pro-cess validators, reward-guided correction, and iso-lated held-out evaluation in one controlled work-flow. The validators blocked every deployment-critical failure mode attempted in our red-team prompt suite — no target leakage, evaluator-private access, snapshot tampering, or forbidden-action attempts were observed — while the hidden test still exposed the resulting quality loss, though this prompt suite is not an exhaustive adversarial audit. The agent must therefore be safe, repro-ducible, protocol-compliant, and effective on held-out production-like tasks before deployment. The contributions of this paper are as follows:

2 Related Work 

LLM-powered agents are increasingly evaluated in settings that require tool use, code execution, and iterative debugging. Data-science and ML work-flows are a particularly important case because suc-cess depends not only on generating executable code, but on following a valid modeling process: inspecting data, avoiding leakage, choosing appro-priate validation, etc. Existing benchmarks have made substantial progress on these capabilities, but most emphasize final task performance, code correctness, or broad research-style experimenta-tion. GRACE-DS targets a complementary prob-lem: pre-deployment evaluation of LLM-powered AutoML agents on production-like tabular work-flows, where organizations must verify both pre-dictive performance and process reliability before allowing agents to operate on internal data. 

2.1 Executable data-science benchmarks 

DS-1000 (Lai et al., 2023) and DS-Bench (Jing et al., 2025) evaluate code generation for real-istic data-science problems through executable tests, and DA-Code (Huang et al., 2024b) and DataSciBench (Zhang et al., 2025) extend this to broader agent-style data-wrangling, EDA, and ML prompts, including settings with uncertain or non-unique ground truth. These benchmarks made ex-ecutable data-science evaluation measurable and reproducible, but they do not capture the workflow structure of production ML — iteration through planning, EDA, feature engineering, validation, correction, and submission, or methodological fail-ures such as target leakage and non-reproducible artifacts. GRACE-DS builds on the executable-evaluation tradition but shifts the unit of evaluation from isolated code snippets to an interactive tabular ML workflow. 

2.2 Final-score and competition-style benchmarks 

MLE-bench (Chan et al., 2025) evaluates ML agents on a large set of Kaggle competitions and offers a realistic measure of broad ML engineer-ing ability. TML-Bench (Pinchuk, 2026) is closer to GRACE-DS in domain, focusing on tabular ML competitions with hidden-label private-holdout scoring. The limitation of final-score benchmarks is that they reveal little about how a score was obtained: a high score may rest on leakage, in-valid validation, or brittle code, while a low score may reflect poor error recovery rather than weak modeling knowledge. GRACE-DS retains hidden final-test evaluation but adds workflow-stage in-strumentation, hidden process validators, reward decomposition, error statistics, and reproducibility checks — less broad and less competition-oriented, but more diagnostic for enterprise pre-deployment decisions. 

2.3 Interactive ML environments 

MLE-Dojo (Qiang et al., 2026), MLGym (Nathani et al., 2025), DSGym (Nie et al., 2026), and MLA-gentBench (Huang et al., 2024a) treat ML work as a sequential process in executable environments, enabling the study of tool use, debugging, and improvement over time. GRACE-DS shares this motivation but makes a different trade-off : in-stead of maximizing task breadth, it narrows the scope to tabular supervised ML and governs the workflow explicitly, with named stages, evaluator-owned validation, hidden checklist validators, hard penalties for critical methodological errors, and a reproducible submission requirement. 

2.4 Data-analysis and scientific-discovery benchmarks 

InfiAgent-DABench (Hu et al., 2024), DABstep (Egg et al., 2025), and IDA-Bench (Li et al., 2025) make multi-step data analysis interactive and auto-matically scorable, while BLADE (Gu et al., 2024), 3ScienceAgentBench (Chen et al., 2025), Discov-eryBench (Majumder et al., 2025), InsightBench (Sahu et al., 2025), and ReplicationBench (Ye et al., 2026) evaluate broader scientific or analytic reason-ing, often via expert matching, planted insights, or LLM-as-judge components. GRACE-DS adopts the interaction idea but applies it to a different tar-get: rather than scoring answers to analytical ques-tions or discovery novelty, it measures whether the agent generates a stable predictor that generalizes to unseen data while satisfying the process con-straints, using low-cost, reproducible, and feasible tests. 

2.5 AutoML benchmarking and agent systems 

Classical AutoML benchmarking, notably the OpenML AutoML Benchmark (Gijsbers et al., 2024), established how to compare automated mod-eling systems across many tabular tasks with con-trolled splits and standardized metrics. GRACE-DS adds an LLM-agent interaction layer on top of this problem: the evaluated system is not a fixed modeling algorithm but an agent that writes code, interprets feedback, chooses workflow ac-tions, and must produce a reproducible final artifact under organizational constraints. Agent systems such as Data Interpreter (Hong et al., 2024) and AIDE (Jiang et al., 2025) are complementary: they propose ways to build stronger agents, whereas GRACE-DS provides an architecture-neutral envi-ronment in which any such agent — hierarchical, tree-search, ReAct-style, or a company-internal copilot — can be evaluated under the same hidden validation and scoring protocol. 

2.6 Positioning of GRACE-DS 

The works closest to GRACE-DS are MLE-bench, TML-Bench, MLE-Dojo, MLGym, and DSGym: the former two provide strong hidden-score evalua-tion but limited process-level diagnosis, while the latter three provide broad interactive environments without GRACE-DS’s specialization in the tabular supervised-learning workflow and its deployment-specific risks. GRACE-DS combines five proper-ties in a single platform: it is stage-aware (ex-plicit workflow stages from planning to final sub-mission), guarded (hidden validators for leakage, protocol validity, reproducibility, and submission validity), reward-guided (process-level feedback whose alignment with hidden-test quality can be an-alyzed), deployment-oriented (final submissions must reproduce on raw input records), and ex-perimentally controlled (the same tasks support single-shot, unstructured, iterative, restart, ablation, reward-optimization, and red-team comparisons).

3.3 Business objective, constraints, and acceptance criteria 

GRACE-DS is framed around a deployment con-tract. From raw tabular data the agent must produce a reproducible pipeline — a callable predict_fn

3.6 Evaluation metrics 

We report the main metrics used throughout Sec-tion 5 (Tables 4, 6, 11) and separate outcome met-rics (was a trustworthy predictor produced?) from 

process metrics (how was it produced?), as the cen-tral claim of GRACE-DS is that the two need not agree. All quality numbers use the task-normalized scale of Eq. (1) , ˆq = ( raw − baseline )/(oracle −

baseline ), where 0 means no better than the trivial baseline and 1 matches the untuned oracle, so het-erogeneous metrics (ROC-AUC, RMSE, R2, . . . ) become comparable across tasks. 

End-to-end quality (E2E Q ), the headline metric, is the normalized hidden-test score of the submit-ted candidate, counting episodes that never reach a valid terminal scoring as 0. It thus folds modeling skill and the ability to finish into the single num-ber a deployment decision turns on. We therefore treat E2E Q together with PV as the primary deployment metric , and read every other quan-tity as a diagnostic that explains them. Observed quality (Obs. Q ) averages only over episodes that produced a terminal score, isolating modeling skill conditional on finishing. Obs. Q must not be com-pared across regimes on its own: because it condi-tions on a regime-dependent subset of completed episodes, a regime can post a high Obs. Q while finishing far fewer episodes, so reading it in isola-tion rewards survivorship. The unstructured-agent 

is the canonical case — its Obs. Q nearly matches the flexible-iterative regime (Table 4) yet its PV is far lower, so its E2E Q collapses. Obs. Q is thus reported only alongside E2E Q and PV, and the gap between E2E Q and Obs. Q quantifies the quality lost to non-completion rather than to weak models. 

Protocol-valid completion (PV ) is the fraction of episodes whose candidate is reproducible and ter-minally scored exactly once without a protocol vio-lation or a harness-forced rescue. Recovery charac-terizes correction behavior: among episodes that hit at least one error, the fraction that repair it — typi-cally through the CODE FIX loop in response to a natural-language hint — and still reach a protocol-valid submission, reported over all errors ( Rec. any ) and over execution errors ( Rec. exec. ). The 

critical-error rate is the share of episodes with a deployment-critical fault (Section 3.5), each forced to zero reward. 

R is the final decomposed process reward of Eq. (3) , reported with its mean contributions Rperf ,

Rplan , Rcode . As this reward is a proxy for held-out quality rather than a substitute for it (Section 3.7), we measure its alignment with E2E Q directly, through a per-(model , task ) reward-quality rank correlation and a signed validation →hidden-test gap that flags overfitting to the gated validation signal. Finally, each episode logs deployment cost — wall-clock seconds, LLM calls, and tokens — summarized as the seconds-per-unit-of-quality ra-tio ¯tsec /E2E Q .

3.7 Validator Structure 

The validator layer in GRACE-DS is designed as a set of narrow, auditable checks rather than as a 6single monolithic judge. This design serves two purposes. (i) it makes the reward and failure anal-ysis decomposable: an episode can fail because of execution, reproducibility, leakage, weak vali-dation discipline, poor feature handling, or termi-nal submission errors, and these failure modes are recorded separately. (ii) it limits reward hacking: no individual checklist item can by itself produce a high final score, and deployment-critical violations override the ordinary reward calculation. Formally, each validator v maps the current run-time session st to a structured result 

zv(st) = (name v, passed v, score v, penalty v,

status v, details v).

(4) where score v ∈ [0 , 1] , penalty v ≥ 0, and 

status v indicates whether the validator is inac-tive, unresolved, or resolved at the current step. Inactive validators are ignored by the reward cal-culation. Active non-plan validators contribute to the code-quality/process term, while the dedicated plan validator contributes to the plan term. The per-formance term is supplied only by evaluator-owned scoring actions, i.e. validation or final hidden-test evaluation. Thus, validators shape the agent’s pro-cess, but they do not replace held-out predictive evaluation. 

Phase-gated activation. Validators are activated according to the workflow phase reached by the agent. This prevents the environment from penal-izing an agent for not yet satisfying requirements that are only meaningful later in the episode. The phase is inferred from the session state: before a plan the phase is zero; after a submitted plan it becomes the planning phase; after EDA it becomes the data-inspection phase; after feature engineering it becomes the preprocessing phase; after a model or reproducible candidate appears it becomes the modeling phase; and after terminal submission it becomes the final-scoring phase. For example, missing-value and feature-pipeline validators ac-tivate only after feature construction is possible, whereas leakage, execution, intactness, efficiency, and iteration-budget checks are active from the be-ginning. Evaluator-owned validators such as model evaluation, feature-importance diagnostics, correct-ness, target-leakage model checks, and baseline comparison are additionally gated until a scoring action is requested and a reproducible candidate exists. 

Sandbox and protocol guards. The first layer is enforced before ordinary validators run. The action parser requires every agent response to be-gin with exactly one action label. Executable ac-tions, such as EDA , FEATURE ENGINEERING ,

MODEL , CODE , and CODE FIX , must contain ex-actly one fenced Python block. Evaluator-trigger actions, VALIDATE and FINAL SUBMIT , must con-tain no code. This prevents agents from mixing scoring requests with hidden side effects. The sandbox also performs static policy checks over the submitted code. It blocks file and network access, subprocesses, serialization/deserialization modules, direct environment access, and unsafe builtins such as open , exec , eval , and import . Calls to scikit-learn metric functions are allowed only for train-local diagnostics; attempts to score evaluator-owned validation or hidden-test objects are rejected. Failed or rejected actions are transactional: the workspace is rolled back to its previous snapshot, so partially executed code cannot corrupt later steps. Protected raw snapshots are hash-checked after each executable action and restored if tampering is detected. 

Integrity and leakage validators. The sec-ond layer targets deployment-critical method-ological errors. The IntactnessValidator 

checks that protected raw snapshots are un-changed. The LeakageValidator scans exe-cuted code for unsafe use of evaluator-reserved objects, fitting on validation/test frames, and using the target column as a feature. The 

TrainValidRefitLeakageValidator catches the common pattern in which the agent concate-nates training and validation frames and then fits or transforms on the combined object. The TargetLeakageModelValidator adds an evaluator-side diagnostic: it searches for suspicious single-feature predictors and large train-validation gaps that suggest leakage or severe overfitting. These validators are deliberately redundant: leak-age can be caught by namespace restrictions, code-pattern checks, snapshot integrity, and evaluator-side scoring behavior. A small set of critical errors is treated spe-cially. Target leakage, train+validation refit leakage, evaluator-private access attempts, and protected-snapshot tampering force the final reward for that step to zero. This is stricter than an ordi-nary soft penalty: the agent cannot compensate for a critical methodological error by satisfying check-7list items or obtaining a high intermediate process score. 

Data-condition validators. The third layer is task-conditioned. Before the agent begins, the en-vironment computes a private data profile from the training split: missing-value structure, categorical columns, duplicate rows, high correlations, out-liers, skew, feature-scale disparities, class imbal-ance, identifier-like columns, and high-cardinality columns. This profile is used to compile a hidden checklist whose criteria are included only when the corresponding condition is present in the task. For example, missing-value criteria are added only for datasets with missing features; imbalance criteria are added only for classification tasks with skewed class proportions; correlation and outlier criteria are added only when the training data exhibits those patterns. These validators check both inspection and miti-gation. The EDA criteria look for evidence that the agent examined schema, target behavior, missing-ness, duplicates, correlations, distributions, scale, imbalance, or suspicious identifier-like fields. The feature-engineering criteria then check whether the corresponding issue was addressed in a reusable way, for example by placing imputers, encoders, scalers, or feature derivation logic inside the sub-mitted pipeline or prediction function. This two-level design separates merely noticing a data condi-tion from producing a replayable modeling artifact that handles it. 

Submission and reproducibility valida-tors. The fourth layer enforces the raw-input submission contract. GRACE-DS accepts only two terminal artifact types: a callable 

predict_fn(raw_dataframe) or a fitted scikit-learn Pipeline named pipeline or 

submission_pipeline . The artifact must accept raw feature rows, perform all preprocessing internally, and return predictions with valid length and finite values. After each candidate-changing action, the environment performs a target-free smoke check on raw validation rows. This smoke check does not compute or reveal a metric; it only verifies that the candidate can reproduce preprocessing from raw input. Several validators support this contract. The 

NamespaceCheckValidator checks that expected model artifacts exist and expose prediction methods. The FeaturePipelineValidator pe-nalizes unresolved NaNs, unencoded categor-ical variables, missing scale-sensitive prepro-cessing when appropriate, and transformations that are not packaged into a reusable arti-fact. The ReproducibilityValidator checks for fixed seeds in stochastic estimators. The 

CorrectnessValidator verifies terminal pre-diction length and numerical validity. The 

BaselineComparisonValidator compares the terminal model against the task baseline when a hidden-test score is available. Together, these checks prevent the agent from submitting anotebook-local solution that depends on trans-formed arrays, temporary variables, or validation-only state. 

Modeling and efficiency validators. The mod-eling validators encourage simple, controlled model development rather than uncontrolled au-tomated search. The ModelChoiceValidator re-wards conventional tabular estimators and penal-izes missing models, inappropriate custom esti-mators, or neural-network choices that are poorly matched to the low-budget tabular setting. The 

HyperparamValidator checks whether the agent specifies a small number of explicit, sensible hy-perparameters, while penalizing automated search APIs such as exhaustive grid search, randomized search, Bayesian search, or external tuning frame-works. The EfficiencyValidator penalizes ex-cessive elapsed execution time and search-heavy code. These checks keep the comparison focused on agent planning, correction, reproducibility, and validation-driven improvement rather than on ex-pensive AutoML search. 

Process validators. The fifth layer tracks the agent’s correction behavior. The 

PlanCoverageValidator checks whether the plan acknowledges the metric, validation discipline, reproducibility, iteration, and task-specific risks. The BacktrackingValidator detects repeated destructive transformations or unstable workflow reversals. The IterativeCycleValidator 

evaluates whether additional cycles produce useful validation improvement rather than repeated regres-sions. Candidate diversity is also rewarded through a lightweight model-family signal: before stopping, the agent is encouraged to validate a small number of meaningfully different manual candidates, such as a tree ensemble and a linear/logistic baseline, without running automated search. 8Hidden checklist feedback. The exact detectors, thresholds, and recognition patterns are never re-vealed to the agent. At each step, the environment computes the relevant stage assessment internally and exposes only a stage score and a bounded list of abstract unresolved hints. For instance, the agent may be told that missing values remain unresolved or that transformations are not demonstrably repro-ducible, but it is not shown the regular expressions, object-inspection rules, or numerical thresholds used by the evaluator. This hidden-checklist design reduces direct pattern matching against the evalua-tor while still giving the agent enough information to repair its workflow. The agent receives three kinds of feedback. Exe-cution feedback reports success/failure, the cap-tured stdout , and on failure a traceback with targeted hints (e.g. missing imports, engineered columns left outside the pipeline, an unfitted pipeline). Stage-aware feedback is compiled from the dataset conditions actually present (missing val-ues, categoricals, duplicates, correlations, outliers and skew, imbalance, identifier-like columns, . . . ) into a hidden checklist : criteria are added only for conditions a task exhibits, so the rubric varies per task and cannot be memorized. Each step yields a stage score in [0 , 1] and, for every unresolved crite-rion, an abstract hint that says what is unresolved and which direction to move but never reveals the detector (its regexes, thresholds, or checks). Metric feedback is gated : the validation metric appears only after an explicit VALIDATE (or harness auto-validation), MODEL never discloses quality, and the hidden-test metric is computed once at terminal scoring. 

Why the rubric stays hidden? Disclosing the exact rubric would let an agent pattern-match the checker — e.g. calling df.isna().sum() to tick the EDA missing-value box without ever reason-ing about whether those values matter or handling them inside a reproducible pipeline. Hiding it while surfacing only abstract hints pushes the agent to reason about the data. The reward reinforces this: checklist compliance feeds only the plan ( 0.15 )and code-quality ( 0.30 ) terms and cannot move the performance term ( 0.55 ), which the evaluator computes on held-out data. The process reward is therefore a proxy for task quality, not an inde-pendent target, under ordinary task-solving the two move together, but the adversarial regimes (Sec-tion 3.8) show that rewards cannot fully replace test metrics. 

Evaluator-owned scoring validators. Metric-bearing validators are evaluator-owned. The valida-tion target and hidden-test labels are never placed in the agent-visible namespace. During VALIDATE ,the evaluator resolves the active reproducible can-didate, runs it on raw validation rows, computes the task metric privately, and returns only the scalar validation score. During FINAL SUBMIT , the eval-uator runs the active candidate once on the hidden-test split. The final hidden-test score is therefore isolated from the agent’s code and from the process reward. This separation is essential: process valida-tors measure whether the workflow is methodolog-ically acceptable, while the hidden test remains the final arbiter of predictive quality. 

Reward integration. Validator outputs are com-bined with the normalized performance score in the decomposed reward. The performance term has the largest weight and is nonzero only on evalua-tor scoring actions. The plan term is supplied by plan coverage. The code-quality term is the mean score of active non-plan validators, including both explicit validators and hidden stage-checklist ag-gregates. Soft validator penalties are summed and capped as a fraction of the weighted reward, while critical errors override the ordinary calculation and force zero reward. This produces a reward sig-nal that is informative for correction but bounded enough that checklist compliance alone cannot sub-stitute for solving the task. Overall, the validator structure makes GRACE-DS a guarded evaluation environment rather than a leaderboard wrapper. The agent must satisfy in-terface constraints, preserve evaluator-owned data boundaries, produce a reproducible raw-input arti-fact, respond to task-specific data conditions, and improve through valid feedback. At the same time, the final held-out score is reported separately from process reward, so the framework can expose cases where a trajectory is superficially compliant but pre-dictively weak, or predictively strong but method-ologically invalid. 

3.8 State and interface ablations (H7) To show that the specific set of states — not merely “an LLM writing AutoML code” — drives quality, leave-one-state-out ablations remove a sin-gle capability from an otherwise identical agent (fixed and flexible without plan/EDA/feature engi-neering) and compare against the full fixed-stage-

9iterative and flexible-iterative on the same splits, jointly on the hidden-test metric and the protocol-valid rate. We expect removing EDA or FEATURE ENGINEERING to hurt more than removing PLAN ,isolating each state’s marginal contribution. (H6, H3) A second family stress-tests reward optimiza-tion . The reward maximizer hidden hints and re-ward maximizer disclosed criteria regimes prompt the model to maximize the process reward (the latter also disclosing the public-criteria names), and red-team-vs-validators is a stronger adversary told to treat reward as the only goal and to delib-erately not solve the task, measuring the valida-tors’ catch-rate. The red-team probe is the sharpest test of why process reward cannot replace held-out evaluation : the validators reliably block the deployment-critical failures (no explicit target leak-age, evaluator-private access, snapshot tampering, or forbidden actions), yet the same runs show that the process reward can stay near its baseline level while hidden-test quality drops. GRACE-DS there-fore reports process reward and hidden-test quality separately and treats the former as a useful but non-substitutable progress signal. All ablations reuse the shared protocol, splits, and budgets of Section 4 so comparisons are paired. 

3.9 Experimental Protocol 

(i) The agent uses training labels and validation features, validation labels and the hidden test are evaluator-private; (ii) Each task is fully specified by a task.json manifest (target, type, metric, ref-erence scores, per-state budgets, split strategy), so an episode is reconstructible from manifest plus seeds; (iii) Metrics are higher-is-better, error met-rics (RMSE, MAE, log-loss) are returned negated ;

(iv) Randomness is fixed: split seeds, and LLM sampling seed = base + repeat where the provider honors it (Table 15); (v) Execution is bounded by per-block timeouts and per-episode action and to-ken budgets.

All concrete configuration values are gathered in Tables 16 and 15 (Appendix A), the models in Ta-ble 3, and the tasks in Table 2 (per-task details in Table 17, Appendix A), so the prose can describe what is measured without being interrupted by raw numbers. 

Regimes The environment is exercised under 15 regimes (Table 18, Appendix A): 7 core regimes spanning the single-shot-to-structured-iterative spectrum, 5 leave-one-state-out ablations, 2 reward-optimization probes, and a red-team-vs-validators adversary (Section 3.8). Budgets are set by regime class: the stateless regimes ( single-shot , n_restarts * , unstructured-agent ), which do not drive the structured state machine, share one per-action and per-episode budget, while the 

stateful structured regimes get the larger state-ful budgets of Table 15. To remove an asymme-try whereby restarts could get “free” validation metrics, the harness auto-validates newly repro-ducible candidates during the working phase. This spectrum operationalizes our two primary compar-isons. (H1) We test whether the structured iter-ative regimes ( flexible-iterative , flexible-compact ,

fixed-stage-iterative ) improve the protocol-valid success rate and failure recoverability over both

(H2) We test whether structured iteration

Reproducibility and tooling A single CLI run-ner consumes a YAML config (models, tasks, regimes, seeds, temperatures, budgets) and sup-ports --dry-run and smoke runs, so a new ex-periment is a new YAML rather than a code change. Every episode is reconstructible from its (model , task , regime , seed , temperature , repeat )

tuple; runs are checkpointed and resumable, each model call is retried under explicit per-request/per-episode timeouts, and each split man-ifest stores SHA-256 hashes of its index sets so train/valid/hidden-test isolation is auditable after the fact. 

Controlled model complexity To avoid con-founding from automated search, agents are instructed and validated to use only small, manually selected model configurations (no grid/random/Bayesian search), which focuses the comparison on planning, EDA, feature engineering, 10 reproducibility, and validation-driven iteration. 

Regime details flexible-iterative lets the agent order stages but under a candidate-first discipline: the plan must build a simple reproducible base-line MODEL before deep EDA and name a second, meaningfully different manual model family, and voluntary early stopping is gated until several val-idated candidates span at least two distinct fam-ilies. A lightweight regex-based diversity signal (computed without executing user code) feeds rcode ,rewarding a few distinct manual choices rather than automated search. fixed-stage-iterative instead dictates the stage each turn ( PLAN → EDA →

FEATURE ENGINEERING → MODEL → re-fine ), so it needs no candidate-first gate. flexible-compact is an interface ablation of flexible-iterative

After the working phase the evaluator selects the best candidate by validation metric and reproduces it in a fresh environment with the same seed before a single FINAL SUBMIT , giving every regime an identical terminal policy and scoring the hidden test exactly once.

6 Conclusion 

We introduced GRACE-DS, a controlled pre-deployment evaluation harness for LLM-powered AutoML agents on tabular supervised-learning workflows, and used it to compare 15 harness regimes across eight frontier LLMs and ten production-like tasks. The central finding is that how an agent is allowed to work matters as much as which model runs it: flexible-iterative reaches an end-to-end normalized hidden-test quality of 0.754 ,against 0.536 for single-shot generation, 0.527 for the unstructured-agent , and 0.672 and 0.686 for the restart baselines, while raising protocol-valid completion to 96 .9% . 7 Future Work 

GRACE-DS is a controlled pre-deployment eval-uation harness for LLM-powered AutoML agents on tabular supervised-learning workflows, its direc-tions for future work therefore follow directly from the design choices that define its current scope. 

Scope GRACE-DS currently targets tabular ML because of its central role in industrial applications such as risk scoring, churn prediction, forecasting, fraud detection, and operations analytics. Extend-ing the harness to deep learning training loops, mul-timodal modeling, reinforcement learning, large-scale distributed training, architecture search, and open-ended scientific discovery would test whether the process-level, deployment-relevant distinctions GRACE-DS surfaces generalize beyond tabular Au-toML and move it toward a more universal bench-mark for ML agents. 

Relation to production readiness Integrating GRACE-DS into the broader MLOps lifecycle. Passing GRACE-DS does not by itself certify a model as production-ready: monitoring, proba-bilistic calibration, robustness to future distribution shift, latency constraints, explainability, security review, and approval workflows. 

Governance coverage The constraints GRACE-DS enforces (Table 13) are a selected, deployment-relevant subset, not full governance readiness. It does not yet model personally identifiable in-formation detection and handling, role-based ac-cess control, data-retention and audit-logging poli-cies, fairness or bias auditing, or regulatory-compliance checks (e.g. GDPR). Extending the manifest and validator suite to express and enforce such organization-specific policies is a natural di-rection for future work. 

Adversarial coverage Our red-team evaluation is prompt-based and not an exhaustive adversarial audit. It does not yet cover validator-aware attacks, manual or obfuscated leakage (e.g. via serializa-tion, indirect imports, or feature-name encodings), data-copying, or attempts to infer validation labels from feedback. Strengthening the adversarial suite along these axes, and treating it as an open red-team benchmark rather than a fixed prompt set, is left to future work.

23 B Extended Operating-Time and Cost Results 

Tables 19 and 20 extend the runtime view of Ta-ble 11 to all 15 regimes evaluated in the grid and to the eight LLMs under flexible-iterative , with three additional columns aimed at deployment reason-ing: the median wall-clock with its inter-quartile range (more representative than the mean under the long-tailed episode-time distribution induced by occasional retries and slower endpoints), token usage in thousands, and a derived seconds-per-unit-of-quality ratio ¯tsec /E2E Q that summarizes how much wall-clock each regime spends per unit of end-to-end quality it produces. We deliberately do not convert this into a monetary cost, because dollar prices vary per provider and per endpoint precision and would date the comparison, the calls and token columns give all the information needed to derive a per-deployment cost. Three deployment-relevant readings follow from Table 19. First, the rank by mean wall-clock and the rank by efficiency are different: the call-matched restart upper bound is the slowest regime in absolute terms (251 s, 2.5× the flexible-iterative 

regime’s mean) and the least efficient (365.9 s per unit of E2E Q), while flexible-iterative is the fastest iterative regime in mean wall-clock (101 s vs. 120 s for the unstructured-agent ) and delivers +0 .227 

E2E Q over it, i.e. a 41% better sec./E2E Q ratio (134.5 vs. 226.9). single-shot remains the most efficient per unit of quality it does produce when one is willing to accept its end-to-end quality of 0.536, but the reward-max-disclosed probe and 

flexible-iterative sit at the next efficiency frontier points (129.1 and 134.5 sec./E2E Q at quality 0.71 and 0.75 respectively). Second, ablating PLAN in-creases time and reduces quality, so its efficiency penalty is compounded: fixed-without-plan is 16% slower than the full fixed schedule and reaches only 89% of its quality, yielding a 1.3× worse sec./E2E Q ratio (241.3 vs. 185.3). Third, the long-tailed time distribution (medians 57–76% of means in every iterative regime) means that median la-tency is the relevant figure for SLA-style budget-ing: under flexible-iterative , half of episodes finish within 75 s and three-quarters within 137 s, even though the mean is pushed up to 101 s by a small number of slow provider responses. Table 20 makes the model-choice trade-off ex-plicit. Wall-clock per episode ranges over 3.4×

across the eight models (47 s for Qwen3-Next-80B to 158 s for Gemini-3.1 Pro), and so does the effi-ciency ratio (70.8 to 192.5 sec./E2E Q). The non-monotonicity between size and time is informative: the small-active-parameter MoE Qwen3-Next-80B is fastest because it is the only model run with-out reasoning, while the highest-quality model, Gemini-3.1 Pro (0.820 E2E Q), is also the slow-est. The iteration overhead — the ratio of flexi-ble to single-shot wall-clock — also varies widely: GPT-5.4 grows only 1.4×, whereas DeepSeek-V4-Flash, Qwen3.5-397B, and Gemini-3.1 Pro grow 

3.0–3.5×, so the cost of running the structured loop is highly provider-dependent. The Pareto fron-tier on this table (faster and more efficient than the next-best alternative) is short: Qwen3-Next-80B at the low-cost/lower-quality end, Grok-4.3 and GPT-5.4 in the middle, and GPT-OSS-120B and Gemini-3.1 Pro at the high-quality end (0.788 and 0.820 E2E Q). DeepSeek-V4-Flash, DeepSeek-V4-Pro, and Qwen3.5-397B are dominated. This is exactly the type of frontier that motivates H9 and that GRACE-DS makes directly visible to an organization choosing an LLM backend. Token consumption is illustrated in Figure 5.

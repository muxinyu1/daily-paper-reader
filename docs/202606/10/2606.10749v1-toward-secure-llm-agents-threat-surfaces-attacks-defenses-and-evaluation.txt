CHUNRONG FANG ∗, State Key Laboratory for Novel Software Technology, Nanjing University, China Large language model (LLM) agents are rapidly moving from conversational interfaces to software components that plan, invoke tools, maintain memory, and act on external environments. This transition changes the nature of security risk. In agentic settings, failures are no longer limited to unsafe text generation. Untrusted content may redirect control flow, misuse tool privileges, corrupt persistent state, leak sensitive information, or trigger harmful external actions. At the same time, research on LLM agent security is expanding quickly but remains fragmented across attack families, defense layers, application domains, and evaluation settings. This paper synthesizes 247 papers through a lifecycle-based, systems-oriented framework that models agent security around the interaction of information flow, delegated authority, and persistent state. We organize the literature around four questions: how LLM agent security should be modeled, which threat surfaces and attack families dominate, what defenses have been proposed and with what tradeoffs, and how security claims are evaluated. We find that prompt injection and tool-mediated control-flow hijacking still dominate the field, while persistent state corruption and multi-agent propagation are becoming central emerging concerns. We further find that current defenses provide useful building blocks but remain weakly compositional, and that existing benchmarks still underrepresent long-horizon, stateful, and deployment-sensitive risks. We argue that secure LLM agents require explicit trust boundaries, principled privilege control, provenance-aware state management, and evaluation practices aligned with realistic operational settings. Once an LLM is embedded in a loop that connects language understanding, planning, action selection, tool execution, and state update, failures no longer end with the generation of problematic text. They may instead appear as hijacked workflows, unauthorized tool use, persis-tent state corruption, sensitive data leakage, or harmful external actions [ 27 , 165 , 225 ]. In agentic settings, a malicious instruction hidden in a web page, a poisoned tool response, or a contaminated memory entry can affect not only the next model output, but also later decisions, stored state, and downstream agents [ 36 , 89 , 195 ]. Existing overviews likewise emphasize that agentic systems combine autonomy, tool use, and deployment risk in ways that go beyond prompt-only model safety [ 35 , 47 , 55 ]. LLM agent security should therefore be understood as a software and systems security problem, not merely as an extension of prompt-level model safety. This shift matters because LLM agents operate under delegated authority. They do not only interpret language. They can also choose actions, invoke capabilities, and modify persistent state on behalf of users or organizations. As a result, the central security question is no longer only whether a model can be induced to say something unsafe. It is whether untrusted or invalid content can alter control flow, cross privilege boundaries, compromise state integrity, or amplify across coordinated workflows. This perspective brings LLM agent security closer to classical concerns in systems security, including trust boundaries, mediation, capability control, provenance, and containment, while also introducing new challenges created by natural-language reasoning and flexible context composition in relatively open runtime settings. Despite rapidly growing attention, the literature on LLM agent security remains fragmented, uneven in emphasis, and incomplete in its treatment of deployment risk. Recent work has exam-ined prompt injection, tool-use security, memory poisoning, runtime monitoring, secure agent architectures, and multi-agent coordination [ 27 , 199 , 218 , 222 , 225 ]. Adjacent surveys, however, frame the space from broader trustworthy-agent, governance, or full-stack safety perspectives rather than from one shared agentic-loop model [ 35 , 143 , 177 , 221 ]. The result is a field rich in individual attacks, defenses, and benchmarks, but still lacking a coherent view of how these pieces fit together. The imbalance of attention is also notable. Prompt injection has become the most visible topic and has driven substantial progress, yet persistent state integrity, tool-mediated privilege misuse, and multi-agent propagation are increasingly important in realistic deployments while remaining less systematically synthesized [ 27 , 64 , 84 , 130 ]. Meanwhile, current benchmark practice still emphasizes immediate attack success in bounded environments, leaving long-horizon, stateful, and deployment-sensitive risks insufficiently evaluated [33, 219, 224, 231]. This paper addresses that gap through a curated corpus of 247 papers and a lifecycle-based, systems-oriented analytical framework. Rather than organizing the field around a single perspective, we study how risks emerge and propagate across inputs, planning, decisions, tool execution, outputs, memory, monitoring, and coordination. This structure allows us to connect prompt injection, memory corruption, tool-mediated abuse, and multi-agent risk within a shared account of the agentic loop, and to compare defenses and evaluations according to where they intervene and what assumptions they make. Existing surveys provide useful background, but they do not yet offer the same combination of lifecycle structure, systems interpretation, software-engineering emphasis, and transparent cross-sectional coding that we aim to provide [ 58 , 80 , 159 ]. A more explicit comparison with representative prior surveys is deferred to the discussion section, where we relate the present survey’s scope and methodological choices to adjacent reviews. What is still missing is a survey that synthesizes LLM agent security around the architecture of agents themselves and relates threat modeling, defense design, benchmark realism, empirical evaluation, and deployment assurance within one coherent frame.

This design lets us move beyond descriptive aggregation toward conceptual synthesis. It also provides a timely perspective on how the field is evolving. The main contributions of this paper are as follows.

2 Background 2.1 Rise of LLM-based Autonomous Agents 

Recent advances in large language models (LLMs) have shifted research attention from passive text generation toward action-oriented systems that can reason, use tools, access external environments, and pursue multi-step goals [ 85 , 99 , 159 ]. Early work on reasoning-acting integration and tool use already showed that LLMs can serve not only as conversational interfaces but also as decision-making components in interactive software systems [ 145 , 217 ]. That shift has since accelerated through the emergence of web agents, coding agents, memory-augmented assistants, embodied agents, and multi-agent workflows, where LLMs are used to interpret instructions, plan actions, execute tasks, revise intermediate results, and coordinate subtasks [136, 173, 196, 212, 245]. A defining feature of contemporary LLM agents is that they function as autonomous or semi-autonomous workflow components rather than as answer generators alone. In research settings, agents can browse websites [ 41 , 245 ], manipulate files [ 8 , 134 ], generate and execute code [ 40 , 212 ], decompose and coordinate tasks [ 196 , 222 , 251 ], and maintain persistent state across long-horizon interactions [ 27 , 36 , 136 ]. Comparable capabilities are increasingly visible in deployed systems, including browser-use agents [ 53 , 134 ], computer-use tools [ 8 , 9], and agentic assistants with remote

browsing, terminal access, and connector-based access to external applications [ 133 , 135 ]. The convergence between research prototypes and deployment-oriented systems makes the security implications of these capabilities increasingly difficult to treat as speculative. This development also changes the practical meaning of autonomy. An LLM agent is not merely producing a response to be read by a user. It may select the next operation in a workflow, decide which page to open, choose which tool to invoke, determine what command to run, update memory, or communicate with other agents. As the operational reach of these systems expands, their failure modes increasingly resemble those of software systems with delegated authority. The relevant question is no longer only whether the model outputs incorrect or unsafe text, but whether it makes unsafe decisions, takes unsafe actions, or preserves unsafe state while acting on behalf of end users or organizations. 

2.2 Security Challenges and Deployment Risks of LLM Agents 

LLM agent security should therefore be understood as a systems problem. It arises when language-mediated reasoning is coupled with action selection, state management, and externally conse-quential execution. Unlike content-level safety, which is primarily concerned with prompts and responses, agent security concerns how user instructions, retrieved content, tool outputs, memory updates, and inter-agent messages can alter control flow, misuse privileges, corrupt state, or trigger harmful operations. From this perspective, an LLM agent is defined less by a particular application domain than by an agentic loop in which the large language model participates in goal interpretation, planning, tool selection, action execution, state update, progress monitoring, or coordination with other agents [ 76 , 143 , 156 ]. This framing allows web agents, coding agents, memory-augmented assistants, embodied systems, and multi-agent workflows to be analyzed within a shared security vocabulary [ 84 , 130 , 237 ]. It also clarifies why LLM agent security cannot be reduced to generic LLM security. In text-only settings, unsafe behavior typically produces problematic textual content [ 113 , 177 ]. In agentic settings, the same underlying vulnerability may instead redirect a workflow, invoke privileged tools, leak sensitive data, poison persistent or temporary memory, or propagate malicious instructions to downstream agents. As a result, securing LLM agents requires attention to control-flow integrity, authority boundaries, runtime containment, state provenance, and monitoring of external actions [61, 153, 169, 198]. These risks are already visible in both research prototypes and deployed systems. Prior studies show that agents can be redirected by malicious web content [ 183 , 195 ], induced to exfiltrate data through tools [ 129 , 170 ], manipulated into unsafe software actions [ 108 , 201 ], or compromised through poisoned memory and inter-agent communication [ 27 , 89 , 165 ]. Deployment documents for browser- and computer-using agents similarly identify prompt injection, harmful task execution, model error, irreversible actions, and risks created by remote browsing, terminal access, and connector-based data access as major concerns [ 8 , 9 , 53 , 133 –135 ]. The significance of these examples is not only that failures can occur, but that they occur at the boundary where language is translated into operational authority. The central challenge is therefore to secure an agentic loop that turns natural language into decisions and decisions into external effects. The remainder of the paper examines how this challenge appears in LLM agent architectures, threat patterns, defensive designs, and subsequent evaluation practice in the field.

3.1 Research Questions 

This paper is guided by the following four research questions on LLM agent security: 

• RQ1 (LLM Agent Security Scope and Modeling): How can LLM agent security be scoped and modeled as a software systems problem? 

• RQ2 (Threat Surfaces and Attack Families): What threat surfaces and attack families dominate current research on LLM agent security? 

• RQ3 (Defense Strategies and Tradeoffs): What defense mechanisms have been proposed to secure LLM agents, and what tradeoffs do they introduce? 

• RQ4 (Evaluation Practices and Benchmark Gaps): How is LLM agent security evaluated, and what gaps remain in existing benchmarks and empirical methodologies? To answer RQ1, we analyze the corpus through a lifecycle-based coding lens that records which stages of the agentic loop each paper addresses, including input, planning, decision, tool execution, 

ACM Trans. Softw. Eng. Methodol., Vol. 1, No. 1, Article 1. Publication date: January 2026. 1:6 Yuchen Ling, Shengcheng Yu, Zhenyu Chen, and Chunrong Fang 

output, memory, monitoring, and coordination. This allows us to synthesize a systems-oriented model of LLM agent security centered on how untrusted information becomes control decisions, how those decisions interact with delegated authority, and how state persistence changes the security properties of the overall system. To answer RQ2, we examine the threat surfaces, attack methods, and deployment scenarios represented in the reviewed literature to identify the attack families that dominate current research and the propagation patterns through which they cause harm. This analysis is used to distinguish between the current empirical center of the field, especially prompt injection and tool-mediated control-flow hijacking, and the emerging areas of concern around persistent state corruption, authority misuse, and multi-agent spread. To answer RQ3, we compare defense papers by their main intervention point, protected asset, trust assumption, and reported tradeoffs in overhead, flexibility, and utility. This makes it possible to assess not only which defenses are most common, but also whether the literature is converging on a coherent and composable security stack for practical LLM agent deployments. To answer RQ4, we review benchmark and evaluation studies in terms of scenario coverage, benchmark reuse, metric selection, and methodological realism. In particular, we examine whether current evaluations adequately capture long-horizon behavior, memory and coordination risks, privilege-sensitive action, and the joint measurement of safety, utility, latency, and cost. 

3.2 Search Strategy 

We construct the review corpus through a three-stage pipeline that combines database retrieval, LLM-assisted candidate expansion, and citation snowballing [ 83 , 193 ]. The search window covers studies published from January 1, 2023 to April 27, 2026. We choose this period because explicit work on LLM agent security begins to appear in late 2023 and then expands rapidly after 2024. The core review artifact is the project workbook, which contains four auditable sheets: the final corpus ( LLM Agent Security ), a relevance-audit sheet, an exclusion log, and notes documenting inclusion-boundary decisions. The review protocol centers on six operational choices that are important for audit: a fixed time window, a six-source retrieval set, a stable three-concept Boolean query template, constrained LLM-assisted expansion, backward and forward snowballing, and explicit normalization/coding rules. We search ACM Digital Library, IEEE Xplore, Scopus, Web of Science, arXiv, and Google Scholar because this combination covers formal publications, fast-moving preprints, and cross-disciplinary indexing. Venue-indexed databases are necessary for software engineering, security, systems, and NLP venues, while arXiv and Google Scholar are necessary because many relevant studies first appear as preprints or are more easily discoverable through benchmark names, framework names, and scenario-specific terminology than through stable subject headings. To improve recall, we use a concept-combination search design rather than isolated keywords. The canonical Boolean template is:

Traditional database search alone is insufficient for a field where terminology shifts quickly and benchmark or framework names often precede stable community vocabulary. We therefore use GPT-5.4 with web search as a bounded retrieval assistant, not as an inclusion oracle. The model is given the current seed list together with titles, keywords, and publication dates, and is prompted to: (1) suggest lexically distant but semantically related candidate papers, (2) propose query variants and benchmark aliases, (3) normalize synonymous terms, and (4) highlight missing branches such as memory poisoning, instruction hierarchy, MCP tooling, or multi-agent governance. Every candidate surfaced by this step is then manually verified against the underlying bibliographic source before entering the screening pool. In other words, LLM assistance expands recall, but final eligibility remains entirely manual. The third stage is backward and forward snowballing from representative seed papers, early benchmark papers, and existing surveys [ 35 , 55 , 159 , 193 , 221 ]. This step helps recover papers that use emerging terms not well captured by keyword search and papers that are influential through system lineage, benchmark reuse, or architectural influence rather than through literal query overlap. Taken together, these three stages produce a broader search than keyword retrieval alone while retaining explicit source boundaries and manual verification. 

3.3 Literature Selection 

We apply a staged eligibility protocol to construct the final corpus [ 83 ]. After preliminary retrieval and normalization, 275 records enter detailed relevance auditing. These records are screened first by title and abstract, with full-text inspection when abstracts are insufficient. The audit retains 251 records and excludes 25 with explicit reasons recorded in the exclusion sheet. A final bibliographic normalization step then merges duplicate versions and preprint-publication pairs, yielding the final corpus of 247 papers. The inclusion criterion is intentionally strict: a paper must be substantively about LLM agent security and must involve at least one explicit agentic surface such as tool invocation, external action, planning, persistent memory or state, runtime infrastructure, embodied interaction, MCP or skill interfaces, or multi-agent communication. This criterion keeps the paper centered on security problems created by the agentic execution loop rather than on text generation alone. Under this boundary, studies on web agents, coding agents, memory-augmented assistants, embodied systems, and multi-agent workflows are eligible only when the security problem is coupled to agent behavior rather than serving as generic background. We exclude studies that fall outside this boundary. The most common exclusion cases in the log are generic prompt-injection or instruction-separation papers without an explicit agentic execution loop; broad LLM safety, privacy, or governance papers whose relevance to agents is indirect; agentic or autonomous-system papers in which security is not a central research question; and benchmarks whose primary contribution is capability evaluation rather than safety or security. 3.4 Coding and Analysis Framework 

After finalizing the corpus, we annotate each paper using a structured codebook designed to support both descriptive statistics and cross-sectional synthesis. The coding protocol records bibliographic metadata together with analytical dimensions used throughout the paper: primary paper type, system setting, research topic, task scenario, threat model, attack surface, lifecycle stage, benchmark usage, evaluation metrics, attack methods, and defense methods. The codebook intentionally mixes single-label and multi-label fields. For corpus-level distribution analyses, each paper is assigned one primary paper type from the set Attack , Defense , Benchmark ,

Survey , Evaluation , or Report , and one system setting label indicating whether the study primarily concerns a single-agent or multi-agent system. These single-label assignments are used only for high-level distributions. Technical dimensions such as lifecycle stage, threat surface, threat model, defense method, benchmark, and metric are coded as multi-label fields because most papers span multiple dimensions at once. Operational definitions matter because some categories are close neighbors. We therefore apply the following boundary rules throughout the corpus: Web Content denotes externally fetched human-readable content in browsing environments; Retrieved Content denotes search, RAG, or retrieval outputs consumed as supporting evidence; Tool Outputs denotes responses returned by tools or APIs that the agent has explicitly invoked; Files / Code denotes local artifacts such as repositories, scripts, configuration files, and attachments; and Memory / Scratchpads denotes transient or persistent state written by the agent or reused across turns. Likewise, Planning denotes decomposition or trajectory generation, Decision denotes commitment to one action or delegation step, and Tool Execution denotes the actual invocation of an external capability or environment transition. These distinctions do not eliminate every gray area, but they make the counts more interpretable and keep similar surfaces from collapsing into one generic “text context” category. The most important coding rules can be summarized compactly in prose. Primary paper type and 

system setting are single-label fields used only for high-level corpus views. Threat surface , lifecycle stage , threat model , defense method , benchmark , and metric are multi-label fields because most papers span several dimensions at once. Threat-surface labels are separated by provenance and runtime role, while lifecycle labels are separated by when the relevant risk, defense, or evaluation acts in the loop. 3.5 Coding Reliability, Quantitative Interpretation, and Remaining Limits 

The revised manuscript is careful to distinguish three related but different notions: coding trans-parency, coding reliability, and evidential strength. The workbook preserves a harmonized final label set and explicit relevance and exclusion logs, which makes the review auditable. However, coder-specific parallel annotations were not preserved during the initial corpus construction. We therefore do not claim inter-rater agreement statistics, and we do not treat the coded counts as if they were uncertainty-free measurements. Instead, we use three safeguards. First, the codebook makes the most consequential labels operational rather than purely intuitive. Second, we use single-label fields only for coarse corpus views and keep substantive overlap in multi-label fields, which reduces forced simplification. Third, we present most quantitative results as descriptive indicators of research attention , not as direct evidence of technical maturity, real-world prevalence, or validated effectiveness. This distinction is especially important because 169 of the 247 papers (68.42%) are arXiv preprints, including 47 of 66 attack papers and 48 of 64 defense papers. Frequency therefore reveals where the field is concentrating effort, but not necessarily where the strongest evidence already exists. These safeguards do not eliminate all subjectivity, but they make the coding process more transparent and more defensible. Claims about dominant topics, recurring threat surfaces, and benchmark fragmentation are therefore best interpreted as descriptive patterns over a fast-moving corpus. Stronger causal claims about effectiveness, maturity, or deployment readiness still require conservative interpretation and are framed as such throughout the remainder of the paper. Throughout the remainder of the paper, we distinguish three notions that are often conflated in fast-moving survey literatures. Research popularity refers to how often a topic, defense family, or benchmark appears in the corpus. Evidential strength refers to how strong the empirical support is for a claim, which depends on evaluation quality, replication, and publication stability rather than on raw paper count alone. Deployment importance refers to how consequential a problem may be in realistic systems, especially in high-stakes or high-authority settings. Most of our corpus statistics directly measure the first notion, only indirectly inform the second, and may diverge substantially from that third notion. 

4 Trend Observation 

The final corpus shows that LLM agent security has emerged as a rapidly expanding but still methodologically unsettled research area. Fig. 2 summarizes three basic corpus distributions that are especially important for interpreting the rest of the paper: publication year, primary paper type, and system setting. Taken together, these distributions show not only that the field is growing quickly, but also that it remains pre-standardized in venue structure, heterogeneous in research function, and uneven in architectural coverage. The temporal distribution in Fig. 2 (a) makes the growth pattern particularly clear. The corpus contains only 3 papers in 2023, grows to 42 papers in 2024, and then rises sharply to 121 papers in 2025. This increase from 2024 to 2025 marks the clearest inflection point in the field, suggesting a rapid transition from early exploratory work to a concentrated research agenda. The 81 papers collected by April 27, 2026 already account for 32.79% of the corpus. Although this partial-year count should not be interpreted as a final annual total, it still indicates that research activity remained intense after the 2025 surge. The temporal pattern also implies that terminology, threat models, evaluation protocols, and deployment assumptions are still evolving, which makes version control and cautious interpretation especially important.

5 RQ1: LLM Agent Security Scope and Modeling 

RQ1 asks how LLM agent security should be scoped and modeled as a software systems problem. We approach this question through the agentic loop that transforms language inputs into plans, decisions, tool invocations, state updates, and external effects. This scope is broader than prompt security but narrower than generic AI safety. It does not treat every failure of an LLM-enabled system as an agent-security problem. Instead, it focuses on the software structure through which information becomes control, control becomes action, and action produces durable consequences. For clarity, we model an LLM agent at a semi-formal level as the tuple 𝐴 = ⟨𝐼, 𝑃, 𝐷,𝑇 , 𝑀, 𝑂, 𝐶 ⟩, where 

𝐼 denotes inbound context and observations, 𝑃 planning over candidate trajectories, 𝐷 commitment to a concrete next action or delegation step, 𝑇 tool or environment execution, 𝑀 transient or persistent state, 𝑂 externally visible outputs and side effects, and 𝐶 coordination channels with monitors, humans, or peer agents. Security-relevant behavior emerges not from any one element in isolation, but from the flows among them: low-authority content in 𝐼 can distort 𝑃 , change 𝐷 , trigger privileged 𝑇 , poison 𝑀 , or propagate through 𝐶 . This representation is intentionally lightweight, but it makes explicit where trust, authority, and persistence interact. This framing is also consistent with the empirical center of the corpus. The most frequent research topics are Tool-use Security 

with 156 papers, Runtime Defense with 88, Prompt Injection Security with 75, Multi-agent Security 

with 63, and Memory Safety with 32. These topic frequencies are revealing. They show that the literature is already oriented around control, capability use, runtime mediation, coordination, and state integrity, even when individual papers use different labels or enter the problem from different application domains. Survey and taxonomy papers make the same move from prompt-level risk to agent-level control concerns [ 35 , 47 , 55 , 58 , 80 , 85 , 99 , 143 , 159 , 177 , 221 ]. Related governance and systems analyses further emphasize delegated authority, state, assurance, and deployment context [ 3 , 17 , 43 , 48 , 64 , 76 , 77 , 84 , 100 , 112 , 113 , 115 , 130 , 151 , 154 , 156 , 163 , 206 , 207 , 210 , 237 ]. Deployment reports, system cards, and operational advisories point in the same direction from the practice side [8, 9, 51, 53, 54, 119, 133–135, 150, 164, 167]. 

5.1 Why a Lifecycle View Is Needed 

A lifecycle view is necessary because the main risks of LLM agents are propagated rather than localized. A malicious instruction hidden in a web page may enter as input, distort planning, alter a tool choice, contaminate memory, and eventually produce an unsafe external action. If analysis stops at the first entry point, the mechanism through which harm is realized remains only partially visible. For this reason, taxonomies organized only by attack names or deployment domains are too shallow for the present field. The corpus strongly supports this claim. Among papers with lifecycle annotations, Planning 

appears in 227 papers, Input in 225, Tool Execution in 209, Decision in 166, and Output in 151. These stages dominate the literature and together define the main operational chain through which agentic risk is realized. Memory and coordination are less frequent but still substantial, with 

Memory appearing in 82 papers, Monitoring in 58, Inter-agent Comms in 35, and Coordination in 20. The implication is substantive. The field is not primarily studying isolated prompt failures. It is studying repeated, stateful, and externally consequential control processes. These frequencies should nevertheless be read as indicators of research popularity , not as direct measures of evidential strength or deployment importance. A less frequent topic may matter more in high-consequence domains, and a frequent topic may still rest on a narrow benchmark base or predominantly preprint evidence. This perspective is useful not only descriptively, but also analytically. It provides a common unit of comparison across attacks, defenses, and benchmarks. It also explains why seemingly different

5.3 Implications for Modeling LLM Agent Security 

The practical implication of this model is that agent security should be analyzed in terms of 

transitions rather than components alone. Inputs, plans, tools, memory, and coordination channels are all important, but the most consequential failures usually occur when one kind of state is allowed to transform into another without adequate mediation. Untrusted content becomes especially dangerous when it is reinterpreted as a planning constraint, when a tentative plan becomes an executable decision, or when a stored trace is later reused as trusted context. This transition-centered view helps explain why agent security often looks different from both conventional application security and prompt-only model safety. The key question is not only what information the system has seen, but what the system is allowed to do next because it has seen it. This perspective also sharpens the architectural meaning of trust boundaries. In a conventional software pipeline, developers often know where data validation ends and privileged execution begins. In agentic systems, those boundaries are easier to blur because natural-language context may simultaneously contain user goals, retrieved evidence, tool feedback, intermediate reasoning, 

ACM Trans. Softw. Eng. Methodol., Vol. 1, No. 1, Article 1. Publication date: January 2026. Toward Secure LLM Agents: Threat Surfaces, Attacks, Defenses, and Evaluation 1:15 

and delegated instructions. The tuple therefore provides a way to ask more precise design questions: which channels into 𝐼 should be treated as low-authority; which transitions into 𝑃 and 𝐷 require source-aware interpretation; which actions at 𝑇 deserve explicit capability checks; which state written into 𝑀 can be reused later; and which messages moving through 𝐶 may propagate authority rather than merely information. Once stated in this form, several seemingly separate issues in the literature become variations of the same modeling problem. This modeling choice also clarifies several later findings. It helps explain why the dominant threats in the corpus manipulate control flow rather than merely content, why many promising defenses intervene at capability boundaries or runtime mediation points, and why evaluation becomes less satisfactory once persistence, coordination, and externally consequential action move to the foreground. It also provides a common language for comparing papers that study very different deployments. A browsing attack, a coding-agent compromise, a memory-poisoning case, and a multi-agent contagion event may look heterogeneous at the surface level, yet they can still be compared in terms of where they enter the loop, which transitions they exploit, and where consequences are finally realized. In the remainder of the paper, we therefore reuse the tuple not only as a framing device but also as a shorthand for where attacks enter, which transitions defenses constrain, and which components current benchmarks actually exercise. 

Answer to RQ1 

LLM agent security should be modeled as the interaction of information flow, delegated authority, and persistent state across the agentic loop. This formulation is more specific than simply calling the problem “system-level.” It helps unify strands of the literature that might otherwise appear disconnected: prompt injection is often a planning and execution problem, memory poisoning is a delayed control-flow problem, and multi-agent coordination is also a trust and propagation problem. 

6 RQ2: Threat Surfaces and Attack Families 

RQ2 asks which threat surfaces and attack families dominate current research on LLM agent security. The corpus points in two directions at once. At the level of volume and visibility, the field is still centered on prompt injection and its tool-mediated variants. At the level of conceptual development, however, the threat model is expanding toward persistence, authority abuse, and propagation. Put differently, the literature is moving from single-step prompt compromise toward longer-horizon and networked forms of failure, even though prompt injection remains the dominant organizing problem. This attack literature already spans mediated prompt injection in web and tool settings [ 22 , 41 , 45 , 98 ,106 , 183 , 195 , 200 , 205 ]; coding and computer-use compromise [ 40 , 57 , 104 , 108 , 127 , 139 , 201 , 236 ]; and memory, storage, or tool-chain corruption [ 27 , 36 , 37 , 42 , 60 , 65 , 74 , 89 , 97 , 158 , 172 , 208 ]. Multi-agent and propagation-centric attacks form a parallel branch [ 124 , 165 , 179 , 184 , 214 , 249 , 251 ]. Other studies extend the threat model to embodied backdoors, delayed triggers, social-engineering variants, and policy-evasion attacks [ 4 , 50 , 56 , 69 , 78 , 88 , 90 , 101 , 109 , 110 , 117 , 129 , 131 , 147 , 152 ,157 , 170 , 178 , 186 , 188 , 197 , 211 , 229 , 232 , 233 , 239 , 242 , 244 , 246 ]. Across these works, the most useful distinction is not simply between different attack names, but between different propagation logics : mediated entry through untrusted content, delayed reactivation through stored state, and amplification through inter-agent communication.

6.2 State Integrity and Multi-Agent Propagation 

Prompt injection remains dominant, but it is no longer sufficient to characterize the full threat landscape. The corpus increasingly points toward a second line of concern: threats that exploit persistence and propagation. These threats are especially visible in work on memory integrity and multi-agent coordination. State integrity is the clearest example. Memory Poisoning appears in 24 papers as a threat model, while Memory Safety appears in 32 papers as a research topic. These numbers are still much smaller than those of prompt injection, but their conceptual weight is substantial. Once an agent stores notes, summaries, or long-lived context, contamination may survive the original interaction and reappear during a later planning episode. The security question is no longer only whether a malicious instruction was seen. It is whether the system can remember, trust, forget, quarantine, and revoke correctly over time. This is why memory should be understood not as an auxiliary module, but as a primary security surface in long-horizon agents. 7 RQ3: Defense Strategies and Tradeoffs 

RQ3 asks what defense mechanisms have been proposed to secure LLM agents and what tradeoffs they introduce. The corpus suggests that defense is no longer a secondary reaction to attack discovery. It is now a substantial and rapidly growing part of the field. At the same time, the defense space remains fragmented, with few techniques reaching the status of a stable and reusable stack. The central challenge is therefore no longer whether defenses exist, but whether the current defense landscape can be organized into a coherent engineering logic that matches the propagation pathways identified in RQ2. Correspondingly, the defense literature now covers source-ordering and prompt-handling mechanisms [ 12 , 32 , 61 , 71 , 73 , 82 , 169 , 216 , 234 ]; runtime monitoring and guard-agent architectures [ 44 , 87 , 93 , 174 , 182 , 199 , 240 , 243 , 248 ]; privilege control, safer tool mediation, and containment [ 72 , 94 , 122 , 142 , 162 , 198 , 235 , 247 ]; and memory, provenance, protocol, or communication protection [ 2 , 6 , 11 , 13 , 15 , 16 , 19 , 21 , 28 – 30 , 34 , 52 , 59 , 70 , 79 , 95 , 111 , 116 , 118 ,120 , 128 , 137 , 153 , 160 , 161 , 171 , 176 , 180 , 181 , 190 –192 , 194 , 202 , 203 , 223 , 255 ]. The diversity of this literature is encouraging, but it also creates a familiar problem: the field has many defensive components without yet having a strong account of how those components should compose under realistic and heterogeneous threat assumptions. 

7.1 Defense Families Along the Agentic Loop 

The strongest way to organize the defense space is by intervention point along the agentic loop. This framing clarifies both what current defenses protect and where their limits lie. It also helps separate defenses that primarily influence interpretation from those that constrain action or contain damage after interpretation has already failed. The first family concerns source handling and instruction ordering. Methods such as Instruction Hierarchy and Guardrails try to reduce the chance that low-authority or malicious content is treated as a legitimate control signal. Their appeal lies in their relatively low deployment cost and architectural simplicity. However, their protection remains fundamentally model-mediated. They still depend on the model correctly interpreting source precedence in noisy, multi-source contexts, which means they improve robustness without fully relocating trust away from the model. The second family concerns runtime scrutiny. Runtime Monitoring , Policy Enforcement , and 

Anomaly Detection all intervene after content has already entered the loop but before harmful consequences are fully realized. These methods are attractive because they match the temporal structure of agent behavior more closely than prompt-only defenses do. They can inspect plans,

7.2 Enforcement, Isolation, and State Protection 

Two higher-level patterns emerge once the defense space is viewed through the lifecycle and threat analysis developed earlier. First, the most developed recurrent defenses in the corpus tend to protect authority boundaries rather than prompt boundaries alone. The strongest recurring ideas in the normalized corpus, especially access control, policy enforcement, and information-flow control, all try to constrain what contaminated reasoning is allowed to influence. This is a constructive sign. It suggests that the field is gradually translating classical security principles such as least privilege, mediation, and separation of duties into agentic settings rather than relying solely on better prompting discipline. Second, defense coverage is still misaligned with the direction of the threat model. The field has already identified persistent memory corruption and coordination failures as important concerns, yet comparatively little defense work directly addresses provenance, revocation, inter-agent trust, or topology-aware containment. In other words, current defenses are densest around better-studied prompt-to-tool pathways and much thinner around the memory and coordination pathways which are still expanding. 8.2 Metrics and Sources of Fragmentation 

Metric practice reveals a second and equally important imbalance. Attack Success Rate appears in 129 papers, far ahead of Success Rate with 59 and Accuracy with 41. By contrast, more deployment-sensitive metrics are much less common. Utility appears in only 22 papers, Latency in 8, and Cost in 7. Even measures that are essential for operational tradeoffs, such as Refusal Rate and False Positive Rate , appear only 13 times each across the corpus. This asymmetry matters because LLM agent security is not a single-objective problem. A defense that reduces attack success by refusing every action is safe only in a trivial sense. Conversely, an agent that maintains task success while silently leaking data or corrupting memory is not secure. Much of the current literature therefore evaluates whether systems can be broken, but less often whether they remain governable, useful, and economically deployable under realistic controls. In that sense, many current benchmark practices still resemble vulnerability discovery more than deployment assurance under realistic use. Fragmentation is compounded by differences in task construction and reporting practice. Different studies assume different tools, different attacker knowledge, different environment realism, and different utility baselines. Some focus on short-horizon tool use. Others emphasize red-teaming yield, classification quality, or refusal behavior. Because these metrics are usually reported without a stable lifecycle-wide evaluation framework, the raw amount of evaluation work currently exceeds the coherence of its conclusions. 9 Discussion 

The preceding sections show that LLM agent security is already a rich and fast-moving field, but it is still organized more around immediate problems than around stable engineering principles. The most urgent transition is therefore from attack-centric analysis to secure agent engineering. Current work has become increasingly effective at demonstrating that agents can be manipulated. By comparison, the literature is less consolidated around how agent architectures should encode

9.2 From Attack-Centric Analysis to Secure Agent Engineering 

Three design priorities emerge clearly from the corpus. The first is provenance-aware state man-agement. Persistent memory, summaries, and scratchpads can preserve useful context, but they can also preserve contamination. Future systems need stronger mechanisms for recording where a state item came from, under what authority it should be trusted, and when it should be revoked, decayed, or quarantined. Without this capability, memory protection will remain largely reactive. More importantly, long-horizon agents will continue to lack a principled answer to what they are allowed to remember and reuse. The second priority is secure tool governance. Tool use is where many of the highest-consequence risks materialize, because it is also where language-mediated decisions cross capability boundaries. This suggests that agent frameworks should expose permissions, confirma-tion policies, capability scoping, and execution isolation as first-class design elements rather than as optional add-ons. The prominence of access control, information-flow control, and sandboxing in the corpus supports this direction, but current implementations remain uneven and often difficult to adapt to open-ended workflows. The third priority is secure orchestration for multi-agent systems. Once tasks are delegated across agents, message integrity, role separation, authority scoping, and coordination topology become security concerns in their own right. Future work should therefore treat orchestration not only as a capability problem, but also as a trust-management problem. In this sense, multi-agent security is not just single-agent security repeated several times. It introduces new semantics of propagation, amplification, and failure coupling. The common thread across the table is that the field’s most urgent open problems all arise where state, authority, and evaluation interact: how to trust stored context, how to govern consequential tool use, how to contain delegated coordination, and how to evaluate these properties under realistic operating assumptions in deployment.

9.3 Implications for Software Engineering Research and Practice 

For a software engineering audience, the main value of this survey is not only that it catalogs new attacks. It is that it translates those attacks into recurring design, implementation, testing, and maintenance obligations for agentic software systems. The first implication concerns requirements and architecture. Agent systems need security requirements that explicitly bind authority , state , and tool scope . Traditional functional requirements 

ACM Trans. Softw. Eng. Methodol., Vol. 1, No. 1, Article 1. Publication date: January 2026. Toward Secure LLM Agents: Threat Surfaces, Attacks, Defenses, and Evaluation 1:27 

such as “the agent can browse the web” or “the agent can edit code” are too coarse for secure deployment. They should be refined into authority-scoped requirements, e.g., , which tools may be called under which provenance conditions, which memory items may be reused across tasks, which actions require confirmation, and what rollback or containment behavior should occur after a suspected compromise. This suggests a clear research opportunity for agent-oriented threat modeling and secure agent API design. The second implication concerns development workflows, especially coding agents and CI/CD integration. The corpus shows that software-engineering scenarios are both high-frequency and high-consequence. Coding agents often combine file access, shell execution, package installation, secret exposure, and tool-mediated prompt injection in the same loop. From a software engineering perspective, this means that secure-by-design coding agents should expose least-privilege execution, provenance-aware file access, typed tool contracts, and auditable approval boundaries as framework defaults rather than optional hardening layers. Supply-chain risk is part of the same picture: MCP tools, skills, plugins, and repositories extend the attack surface in ways that resemble dependency risk in conventional software ecosystems, but with more ambiguity about what counts as data versus executable control. The third implication concerns testing and regression assurance. Many current evaluations still resemble one-off red-teaming studies, whereas engineering teams need repeatable regression tests and sustainable maintenance practices. For agentic systems, this points toward security regression suites that replay prompt-injection traces, malicious tool outputs, poisoned memory states, and multi-agent communication failures across versions of prompts, tools, policies, and orchestration logic. Benchmarks such as AgentDojo , ASB , OS-Harm , and AgentLeak are useful seeds, but they need to be connected to CI-style security gates, step-level telemetry, and failure triage workflows before they can support mature software processes. The fourth implication concerns runtime verification and operations. Because many agent failures are stateful and delayed, verification cannot stop at pre-deployment testing. Runtime monitors, policy engines, and containment layers must become first-class operational components. This aligns well with established software engineering interests in runtime verification, self-adaptive systems, and assurance cases. In particular, long-horizon agent deployment calls for provenance-aware memory management, explicit tool-approval policies, post-incident replayability, and maintenance procedures for updating prompts, tools, and policies without silently widening the authority surface. Taken together, these implications suggest that LLM agent security should be treated as a software lifecycle problem: requirements engineering for trust boundaries, secure design for tool and memory mediation, regression testing for long-horizon use, runtime verification for consequential execution, and maintenance for evolving agent ecosystems in real deployments. 

9.4 Research Priorities for the Next Stage 

The evaluation findings also point to broader research priorities. High-impact domains such as software engineering, finance, healthcare, and embodied robotics require stronger evidence than short-horizon toy benchmarks can provide. In such settings, the relevant question is not only whether one attack succeeds, but whether the system can remain governable under imperfect information, partial compromise, repeated use, and nontrivial operational constraints. A natural next step is the development of deployment-oriented assurance cases for LLM agents. An assurance case would combine threat modeling, privilege analysis, runtime controls, benchmark evidence, and incident-containment expectations for a specific deployment setting. Such a direction is well aligned with software engineering and systems research because it connects design choices to operational evidence rather than treating evaluation as an isolated leaderboard exercise. 10 Threats to Validity 

This survey involves a substantial amount of retrieval, filtering, normalization, and manual coding. As a result, its conclusions should be interpreted together with the following validity threats. 

Construct validity. The most important construct-validity risk is category overlap. Threat surfaces such as web content, retrieved content, tool outputs, files, and memory can all appear as untrusted text, while lifecycle stages such as planning, decision, and tool execution are often tightly interleaved in modern agent frameworks. We mitigate this risk by publishing explicit operational rules, using multi-label coding for technical fields, and separating descriptive single-label fields from substantive multi-label annotations. Even so, some boundary cases remain judgment-sensitive. 

Internal validity. The corpus relies on manual screening and manual coding. Although the workbook preserves the final labels together with relevance-audit and exclusion sheets, relevance judgments, label assignment, and later harmonization still involve human interpretation. To reduce subjectivity, we use conservative inclusion rules, explicit exclusion logging, version normalization rules, and a boundary-aware codebook. Even with these safeguards, some coding decisions remain judgment-sensitive, especially for overlapping lifecycle stages, threat surfaces, and multi-label benchmark annotations. 

External validity. Search bias and publication bias are unavoidable in a fast-moving area. The search protocol spans six major sources and includes snowballing and LLM-assisted candidate expansion, but it may still miss papers that use unusual terminology, unpublished industrial artifacts, or adjacent system names not captured by the query template. The field is also heavily preprint-based: 169 of 247 papers are from arXiv. This improves timeliness but weakens stability because titles, claims, and experimental designs may change before archival publication. 

Conclusion validity. Many statistics in this survey are frequency counts. Such counts are useful for showing research attention, benchmark reuse, and topic concentration, but they do not directly measure evidential strength, real-world prevalence, or deployment readiness. This risk is especially relevant when interpreting defense effectiveness or benchmark maturity, because benchmark-specific clusters can create a false impression of convergence. We mitigate this threat by distinguishing descriptive counts from evidence-based conclusions, by highlighting metric and benchmark fragmentation, and by avoiding claims that frequency alone implies importance or maturity by itself. 

LLM-assisted retrieval bias. The candidate-expansion stage uses GPT-5.4 with web search to surface lexically distant papers and query variants. This improves recall, but it also introduces model bias because the model may over-suggest highly visible benchmarks, popular terminology, or papers that are easy to retrieve on the public web. We therefore limit the LLM’s role to candidate suggestion and term normalization and require manual verification for every candidate that enters the screening pool for review. These validity threats also clarify the scope of the paper’s contribution. The survey provides a documented and auditable synthesis of a rapidly evolving corpus; it does not claim the final word on corpus completeness, exact retrieval replayability, or coder-independent measurement stability. For a field that is still pre-standardized and heavily preprint-based, this narrower claim is both more realistic and more methodologically defensible.

Toward Secure LLM Agents: Threat Surfaces, Attacks, Defenses, and Evaluation 1:29 

11 Conclusion 

LLM agent security has emerged as a distinct research area because agents transform language into decisions, tool invocations, persistent state, and coordinated action. Based on a corpus of 247 papers, this survey argues that the field is best understood through a lifecycle-based and surface-aware framework that connects information flow, delegated authority, and persistent state across the agentic loop. Viewed through that framework, three descriptive conclusions are especially robust. First, prompt injection and tool-mediated control-flow hijacking still define the empirical center of the reviewed threat literature. Second, the clearest expansion visible in the corpus lies in persistence and propagation, especially through memory reuse and multi-agent coordination. Third, the current defense and evaluation ecosystems are less consolidated than the threat literature they respond to: defense research offers useful components but not yet a stable compositional stack, and evaluation research offers substantial activity but not yet a common basis for deployment assurance. Secure LLM agents will therefore require more than better prompts or stronger filters. They will require principled privilege control, provenance-aware state management, runtime oversight, and evaluation practices that reflect realistic operational conditions. More broadly, they will require a shift in mindset: from securing model outputs in isolation to engineering governable agentic systems whose authority, memory, and coordination structure are treated as first-class security design concerns in deployment.

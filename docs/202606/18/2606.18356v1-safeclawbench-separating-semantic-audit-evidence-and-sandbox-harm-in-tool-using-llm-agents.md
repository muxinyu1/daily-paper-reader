## 摘要
使用工具的语言模型代理引入了超出不安全文本的安全故障：它们可能泄露受保护对象、写入持久化内存、发送消息、修改数据库或触发有害代码和工具效果。现有的评估通常将这些阶段合并为一个单一的攻击成功率，使得很难判断模型仅仅是同意了攻击者还是实际产生了可观察的危害。我们引入了SafeClawBench，一个用于工具使用代理安全性的分阶段基准测试，包含600个受控对抗性任务，涵盖六个攻击家族：直接和间接提示注入、工具返回注入、记忆投毒、记忆提取和歧义驱动的不安全推理。SafeClawBench报告了三个独立的端点：语义攻击接受、审计可见的危害证据以及沙箱观察到的工具/状态危害。在四种提示级别策略下评估五个代理端点，我们发现这些端点捕捉到了不同的故障模式。在没有额外提示保护的情况下，语义故障率在不同模型之间差异很大，从9.0%到44.2%。审计可见的危害证据比语义故障更窄，并且在单独的可执行协议下，一些匹配的任务标识在通过语义核心调用后仍会产生沙箱危害：在12000行匹配分析中，观察到的347个沙箱危害中有291个出现在通过语义检查的行中。提示策略会改变端点结果，但其效果取决于模型和协议。SafeClawBench提供了一个可重复的框架，用于比较代理模型和提示策略条件，而不会混淆文本合规性、证据支持危害和可执行状态更改。开源数据集可在https://huggingface.co/datasets/sairights/safeclawbench获取。

## Abstract
Tool-using language-model agents introduce security failures that go beyond unsafe text: they can disclose protected objects, write persistent memory, send messages, modify databases, or trigger harmful code and tool effects. Existing evaluations often collapse these stages into a single attack success rate, making it difficult to tell whether a model merely agreed with an attacker or actually produced observable harm. We introduce SafeClawBench, a staged benchmark for tool-using agent security with 600 controlled adversarial tasks across six attack families: direct and indirect prompt injection, tool-return injection, memory poisoning, memory extraction, and ambiguity-driven unsafe inference. SafeClawBench reports three separate endpoints: semantic attack acceptance, audit-visible harm evidence, and sandbox-observed tool/state harm. Evaluating five agent endpoints under four prompt-level policies, we find that these endpoints capture different failure modes. Without additional prompt protection, semantic failure rates vary widely across models, from 9.0% to 44.2%. Audited harm evidence is narrower than semantic failure, and under a separate executable protocol some matched task identities produce sandbox harm despite passing the Semantic Core call: in a 12,000-row matched analysis, 291 of 347 observed sandbox harms occur in rows that pass the semantic check. Prompt policies change endpoint outcomes, but their effects depend on both model and protocol. SafeClawBench provides a reproducible framework for comparing agent models and prompt-policy conditions without conflating textual compliance, evidence-supported harm, and executable state changes. The open-source dataset is available at https://huggingface.co/datasets/sairights/safeclawbench.

## 1. 核心问题与研究动机

- **问题背景**：现有使用工具（tool-using）的大语言模型（LLM）代理的安全评估大多将多个危害阶段（如语义同意的攻击、实际执行的恶意操作、持久化状态污染等）合并为一个单一的攻击成功率指标。
- **核心矛盾**：单一指标使得模型仅仅在文本上“同意”了攻击者与实际产生了可观察的危害（如泄露文件、修改数据库、发送邮件、污染记忆等）无法区分。这混淆了文本合规性、证据支持的危害和可执行状态变更。
- **研究目标**：提出一个分阶段的、多端点的基准（SafeClawBench），分别测量语义接受、审计可见的危害证据、以及沙箱可执行工具/状态危害，以解耦不同层面的安全故障。

## 2. 方法论核心思想与技术细节

- **核心思想**：建立三层分离的评估端点
    - **语义核心（Semantic Core）**：600个文本级的对抗任务，由LLM评判器判断模型最终响应是否在语义上同攻击目标（CoreFail@600）。
    - **核心门控危害证据审计（Core-gated Harm Evidence Audit）**：仅对语义失败的响应进行审计，检查是否包含受保护对象泄露、未授权访问、有害行动或持久化污染的证据（HarmEvidence@600）。
    - **可执行沙箱（Exec-Balanced）**：在同一任务标识下，将工具调用部署到隔离沙箱中，通过确定性状态预言者（state oracle）检测实际的状态变更（ObservedHarm@Exec）。
- **攻击分类**：覆盖6个攻击家族
    - DPI（直接提示注入）、IPI（间接提示注入）、TRI（工具返回注入）、MPI（记忆投毒）、MEX（记忆提取）、ADI（歧义驱动的不安全推理），每个家族100个任务，共600个受控对抗任务。
- **提示策略**：定义了9种系统提示级的防御策略（D0无防御、B2轻量Sandwich、D3分层堆叠、D4极长策略探针等），但主实验中聚焦4种（D0、B2、D3、D4/LongPolicy）。
- **测量模型**：
    - CoreFail@600：二进制变量，评判器判断模型响应是否语义上成功攻击。
    - HarmEvidence@600：在CoreFail的行中，审计判断是否具有文本、访问、行动、持久化四类危害证据中的至少一种。
    - ObservedHarm@Exec：沙箱隔离执行后，确定性预言者判断是否发生了有害状态变更。
    - 匹配的Core–Exec连接：对同一个任务标识，比较语义判断与沙箱结果的一致性（如CorePass但ExecHarm的情况）。

## 6. 主要结论与发现

- **RQ1（语义核心）**：
    - 模型选择对安全影响巨大：无防御下CoreFail@600范围从9.0%（Claude Opus 4.7）到44.2%（Kimi K2.5），比率达4.9倍，绝对差距35.2个百分点。
    - 记忆投毒（MPI）和记忆提取（MEX）攻击是最高危类型（平均54.4%），高于其他攻击家族。
    - 提示策略效果依赖模型：B2降低平均率至14.8%；D3和D4接近（9.1% vs 9.2%），但并非对所有模型一致（GLM-5.1在D3低至2.2%，GPT-5.5在D4最低8.8%）。
- **RQ2（门控危害证据）**：
    - 语义失败中只有约54.8%（959/1834）具有审计证据支持（HarmEvidence），其他为纯语义（SemanticOnly）。
    - 无防御下全部840个语义失败行中，504个（16.8%）具有危害证据，336个（11.2%）为纯语义。
- **RQ3（沙箱可执行）**：
    - ObservedHarm@Exec远低于CoreFail：D0下平均4.13%，而核心失败28.0%。
    - **关键发现**：在12,000匹配行中，291/347（83.9%）的沙箱危害发生在通过语义核心检查的任务上（CorePass但ExecHarm）。这表明存在大量“虚假安全”情况——模型在文本上没有同攻击，但实际上触发有害状态变更。
    - 沙箱结果不受语义评判影响，体现了端点分离的重要性。
- **提示策略**：长提示（D4/LongPolicy）在沙箱端有轻微优势，但并非压倒性；提示长度本身有影响，需与组件分离（见长度控制实验）。

## 7. 优点

- **方法论创新**：明确提出并实现了语义层面、审计层面、执行层面三级分离的评估框架，解决现有基准混淆危害类型的关键缺陷。
- **攻击覆盖全面**：6类攻击家族覆盖直接/间接注入、工具返回注入、记忆投毒、记忆提取、歧义推理，贴合实际部署场景。
- **实验设计严谨**：
    - 固定任务标识用于跨端点比较，控制变量。
    - 针对核心失败行进行强制审计，无缺失。
    - 包含大量诊断实验：提示长度控制、消融分析、人类校验、第二评判验证、McNemar统计检验。
    - 开放数据集和完整工具体系，可重复性高（提供哈希值、运行清单、提示模板）。
- **结果具有实践意义**：
    - 提醒社区不能仅靠文本判断安全；大量实际危害发生于文本通过安全检查后。
    - 模型选择比提示策略更重要（无防御下差距35个点）。
    - 提示策略需考虑模型-协议交互作用。

## 8. 不足与局限

- **受控压力测试并非真实部署**：600个任务为合成/受控场景，不反映实际攻击率，适合区分模型而不适合估计真实风险率。
- **防御方式单一**：仅测试了系统提示级别的防御策略，未涉及运行时的权限系统、网络出口控制、人类批准等实际部署手段。作者也承认这点，称提示防御不能替代运行时控制。
- **算力与成本数据缺失**：未报告GPU型号、耗时、API费用等，不利于可复现性和成本估计。
- **基准规模有限**：每类仅100样本，总600样本，对于有些攻击族（如ADI只有45个语义失败行）统计稳定性或有限。虽然通过跨模型聚合弥补，但个别模型-策略组合的置信区间可能仍较宽。
- **模型版本和端点固定**：主要使用了特定版本（如GPT-5.5、Claude Opus 4.7等），这些版本可能很快过时，基准需要持续更新。
- **潜在评估偏差**：依赖单一LLM评判器（GPT-5.4）做语义判断，虽然进行了第二评判和人类校验，但评判器本身可能带有偏差。MEX类型的审计显示不精确（文本未暴露精确秘密但被记为SemanticOnly，可能存在误判）。
- **良善效用检查不完整**：基准的良善任务（benign utility）面板使用较早端点子集，并非与主安全评估完全匹配的端点进行再运行，故不能安全支持安全-效用权衡结论。
- **计算资源使用不透明**：由于依赖API，无法精确测量运行时间与算力消耗，论文也未提及。

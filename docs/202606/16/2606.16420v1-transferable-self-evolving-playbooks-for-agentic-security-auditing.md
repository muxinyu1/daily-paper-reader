## 摘要
用于漏洞发现与验证的 LLM 代理不仅仅是一个模型。它由三个部分组成：用于代码分析的 LLM、用于导航、工具使用和执行的代理工具集（如 Codex 或 OpenCode），以及指导 LLM 和工具集进行漏洞发现的领域特定程序性知识——审计剧本。以往的工作依赖于人工提供的剧本，包括提示工程、手动工作流、知识库和启发式规则。这引出了两个研究问题：获取——人工策划是否必要，剧本创建能否自动化？迁移——演化的剧本能否将审计过程转移给较弱的代理，提升其能力？我们提出了 EvoHunt，一个基于开源仓库进行安全审计的剧本演化环境。三个代理驱动演化循环：审计代理执行当前剧本并产生发现；评估代理根据真实结果对结果评分；修订代理基于失败分析对剧本进行更新。剧本格式不受约束：从空开始，EvoHunt 添加或删除工作流、启发式规则、漏洞知识或领域特定内容。演化后的剧本在不同 LLM 或工具集下运行时只需少量调整。我们在开源安全公告上评估了 EvoHunt。在获取方面，剧本演化使 Codex/GPT5.4-xhigh 6x 的端到端利用从 1.1% 提升至 6.2%；演化的 OpenCode/GLM5.1 剧本在各项指标上均超过 OpenAI Codex Security，目标匹配率为 11.3% 对比 9.2%，表明开源演化可以超越专用商业产品。在迁移方面，GLM 演化的剧本带来了最强的学生模型提升：Qwen3.6-27B 从 2.4% 提升至 6.5%，Qwen3.6-35B-A3B 从 1.1% 提升至 4.6%，且 A3B 获得的匹配数比 GPT 迁移高出 2.4 倍。

## Abstract
An LLM agent for vulnerability discovery and validation is more than a model. It combines three components: an LLM for code analysis, an agent harness such as Codex or OpenCode for navigation, tool use, and execution, and an audit playbook, domain-specific procedural knowledge that guides the LLM and harness toward vulnerability discovery. Prior work relies on human-supplied playbooks, including prompt engineering, manual workflows, knowledge bases, and heuristics. This raises two research questions: Acquisition - is human curation necessary, and can playbook creation be automated? Transfer - can an evolved playbook transfer the audit procedure to weaker agents, improving their capability?   We present EvoHunt, a playbook evolution environment over open-source repositories for security auditing. Three agents drive the evolution loop: an audit agent rolls out the current playbook and produces findings; an evaluator scores outcomes against ground truth; and a reviser commits updates to the playbook based on failure analysis. The playbook format is unconstrained: starting empty, EvoHunt adds or removes workflows, heuristics, vulnerability knowledge, or domain-specific content. The evolved playbook requires only minor adaptation to run under a different LLM or harness.   We evaluate EvoHunt on open-source security advisories. For acquisition, playbook evolution raises end-to-end exploits for Codex/GPT5.4-xhigh 6x, from 1.1% to 6.2%, and the evolved OpenCode/GLM5.1 playbook surpasses OpenAI Codex Security on every metric, with 11.3% vs. 9.2% target-match rate, showing open-source evolution can outperform a dedicated commercial product. For transfer, the GLM-evolved playbook gives the strongest student lift: Qwen3.6-27B improves from 2.4% to 6.5%, Qwen3.6-35B-A3B from 1.1% to 4.6%, and A3B obtains 2.4x more matches than GPT transfer.

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **核心问题**：大型语言模型（LLM）在安全审计任务中的表现，通常被归因于模型自身的能力。但本文提出，审计效果在很大程度上取决于**外部可重用的审计程序**（称为“剧本”，playbook）。以往剧本依赖人工编写（如提示工程、手动工作流、脆弱性知识库等），成本高且难以跨模型迁移。本文旨在回答两个研究问题：
  - **（RQ1）获取**：能否通过自动化方式，让一个固定模型/工具集的智能体从零开始，通过反复尝试、评估和修订，自主习得一套有效的审计剧本？
  - **（RQ2）迁移**：由更强教师模型演化得到的剧本，能否通过轻量适配，转移给更弱的学生模型，从而提升其安全审计能力？

- **整体含义**：安全审计不只是模型能力的比拼，更是**可学习的程序性工程**。将审计方法外化为可版本化、可检查、可转移的文本剧本，可以实现“强教师演化、弱学生执行”的低成本部署模式，类似于知识蒸馏但无需权重更新。

## 2. 论文提出的方法论

### 核心思想：EvoHunt 框架

- **三大智能体**：
  - **审计智能体（Discovery Agent）**：在固定模型和工具集下，依照当前剧本对目标仓库进行审计，产生发现（finding）和证据。
  - **评估智能体（Evaluator）**：将审计结果与真实漏洞（ground truth）对比，输出分数和结构化失败报告。
  - **修订智能体（Reviser）**：基于失败报告，修改剧本文件（Git仓库中的文本），提出新版本。

- **剧本格式**：无约束，起始为空。演化过程中自由添加/删除工作流、启发式规则、漏洞类审计策略、验证门控、假阳性控制等内容。剧本被存储为版本化文本仓库（Git），每次修订是一个commit。

- **持续程序学习（Continual Procedure Learning）**：
  - 按时间顺序处理训练批次（`N1, N2, ..., NB`），可混合重放案例（`Rb`）。每轮用固定批 `Cb = Nb ∪ Rb` 评估当前候选剧本，选择得分更高的版本，然后让修订者生成下一个候选。
  - 训练结束后冻结最佳剧本 `P*`，在测试集上评估，不再更新。

- **迁移方法（Transfer Procedure Learning）**：
  - 源环境演化得到冻结的教师剧本 `P*_s`。目标环境中引入一个**适配器 `At`**（空文件），固定源剧本不变，仅允许修改适配器。
  - 适配器编码目标环境特有的执行差异，结构与源剧本联合使用（提示级拼接）。通过类似演化循环优化适配器，但有长度预算和早停条件。
  - 迁移效果分解为：直接迁移增益（不加适配器）和适配增益（加适配器）。

1. **方法新颖**：提出“程序学习”代替权重学习，将审计策略外化为可版本化、可检查的文本剧本。

## 8. 不足与局限

1. **Benchmark 范围有限**：仅包含高/严重、本地可复现的开源漏洞，排除了低危、依赖第三方服务、需物理访问的场景，不涵盖所有审计类型。
2. **分布偏移风险**：演化剧本可能过拟合训练集分布（如生态系统、CWE 频率），测试集中 Maven 占比骤降可能高估或低估泛化能力。
3. **单次运行**：受成本限制，每个剧本只演化一次，无法评估随机性对结果的影响。
4. **超参数未系统优化**：重放比例、批次大小、评分权重等靠直觉设置，可能存在更优选择。
5. **LLM 裁判误差**：虽然采用了分层评分和手动抽检，但 LLM 裁判仍可能误判边界案例，尤其是与真实漏洞接近但不匹配的情况。
6. **模型与工具集未完全分离**：GPT 运行在 Codex 上，GLM/Qwen 运行在 OpenCode 上，不同工具集的上下文窗口、压缩策略差异可能影响结果，不能完全归因于模型。
7. **预训练数据泄露风险**：测试集漏洞可能出现在模型训练语料中，尽管使用时态分裂并禁止搜索外部信息，仍不能完全排除模型记忆可能。
8. **假阳性率在不同条件下差异大**：例如 Qwen A3B 空基线的假阳性率达 70.6%，虽然迁移后降低但仍较高（GLM 教师迁移后 65.4%），说明误报问题仍然存在。

## 摘要
工具使用代理中的间接提示注入是一个具体的生产威胁：LLM代理从集成（通过工具调用访问的第三方服务，如Gmail、Salesforce或Jira）读取响应内容，而用户既不编写也不控制这些内容。现有基准低估了这一威胁：大多数仅覆盖少数集成，且攻击载荷在运行中重复使用，而开源守卫是在聊天风格数据而非工具响应内容上训练的。我们引入了AGENTREDBENCH，这是一个动态的LLM驱动的红队基准，涵盖24个企业集成（分属9个功能族和5种攻击类型）中的215个微妙未明确规定授权（在用户请求授权边界上的攻击）场景。在8个模型面板（Anthropic、OpenAI、Google）上，无守卫的ASR（攻击成功率）范围从32%（Claude Sonnet 4.6）到81%（Gemini 3 Flash）。为了让场景集不进入训练语料库并保持标题ASR随时间有意义，我们公开了代码库、集成模式和AGENTREDGUARD模型；标准场景通过维护者中介渠道进行评估，并具有不可变版本控制。我们随基准发布了AGENTREDGUARD：一个在集成多样的对抗性工具响应内容语料库上训练的守卫。AGENTREDGUARD将面板ASR从69.9%降至2.4%，假阳性率为0.37%，在两个轴上都优于所有具有非平凡检测能力的开源基线（Llama Guard、PromptGuard 2、ProtectAI）。跨集成和跨攻击类型的保留测试都证实了增益超出了训练子集。

## Abstract
Indirect prompt injection in tool-use agents is a concrete production threat: LLM agents read from integrations (third-party services such as Gmail, Salesforce, or Jira accessed through tool calls) whose response content the user neither writes nor controls. Existing benchmarks under-measure the threat: most cover only a handful of integrations with the same attack payload replayed across runs, and open-source guards are trained on chat-style data rather than tool-response content. We introduce AGENTREDBENCH, a dynamic LLM-driven redteaming benchmark of 215 subtle underspecified authorization (attacks at the boundary of what the user's request authorises) scenarios across 24 enterprise integrations in nine functional families and five attack types. Across an eight-model panel (Anthropic, OpenAI, Google), no-guard ASR (attack success rate) ranges from 32% (Claude Sonnet 4.6) to 81% (Gemini 3 Flash). To keep the scenario set out of training corpora and preserve headline ASR meaning over time, we release the codebase, integration schemas, and AGENTREDGUARD model openly; the canonical scenarios are evaluated through a maintainer-mediated channel with immutable versioning. We release AGENTREDGUARD alongside the benchmark: a guard trained on an integration-diverse corpus of adversarial tool-response content. AGENTREDGUARD cuts panel ASR from 69.9% to 2.4% at 0.37% false-positive rate, outperforming every open-source baseline with non-trivial detection (Llama Guard, PromptGuard 2, ProtectAI) on both axes. Cross-integration and cross-attack type holdouts both confirm the gain transfers beyond the training subset.

# AgentRedBench 论文详细中文总结

## 1. 论文的核心问题与整体含义

- **研究动机**：LLM 代理通过工具调用集成大量 SaaS 服务（如 Gmail、Salesforce、Jira 等），这些集成的响应内容不由用户直接编写或控制，从而引入了间接提示注入（indirect prompt injection）的安全威胁。攻击者可以通过在集成中写入恶意内容（如邮件正文、日历邀请描述），在代理处理工具响应时窃取数据或执行未授权操作。  
- **现有不足**：已有的安全基准存在三大局限：① 覆盖集成数量少（通常仅十几到几十个）；② 攻击内容多为模板化固定载荷，缺乏动态攻击者行为；③ 缺乏多集成链式攻击评估。同时，开源防护模型（如 Llama Guard、WildGuard）是在聊天式对话数据上训练的，不适用于工具响应内容的检测。  
- **整体含义**：为了推动可信 LLM 代理部署，需要构建能够动态、跨集成评估间接注入攻击的基准，并开发专用的工具层防护模型。

## 2. 论文提出的方法论

- **核心思想**：提出 AgentRedBench——一个动态的、LLM驱动的红队基准，涵盖 24 个企业集成中的 215 个微妙“未明确定义授权”攻击场景；同时发布 AgentRedGuard——一个基于集成多样对抗性工具响应语料库微调的小型文本分类器，用于工具响应拦截。
- **关键技术细节**：
  - **动态攻击者**：攻击者 LLM 每轮运行根据集成模式（schema）、攻击类型和裁判反馈动态生成攻击载荷，而非使用固定模板。攻击者拥有最多 5 次尝试，基于裁判反馈自适应调整。
  - **攻击类型**：定义了 5 种“微妙未明确定义授权”攻击类型，分为三类主动模型区分型（output_channel_url_relay, destination_hijack, content_hijack）和两类边界委托模式（tool_argument_hijack, tool_family_creep）。后两种在 8 个模型上几乎无效（ASR 0–2%）。
  - **裁判评估**：LLM-as-judge 对每个轨迹输出成功、部分成功或失败，并联合考虑任务完成率（utility）以拒绝简单拒绝响应（trivial refusal）。
  - **防护模型**：AgentRedGuard 基于 MiniLM (23M 参数) 或 DeBERTa-v3-small (142M 参数) 微调，用于在代理读取工具响应前拦截恶意内容。输出连续性攻击概率，默认阈值为 0.5。训练数据来自面板 14,846 条含攻击行和 4,807 条良性行（包含真实生产硬负例）。
- **流程**：攻击者生成载荷 → 编排器注入模拟集成状态 → 目标代理运行工具调用循环（可选防护） → LLM 裁判评分。

## 3. 实验设计

- **数据集/场景**：标准场景集共 215 个，覆盖 24 个集成（9 个功能族：通信、生产力、日历、CRM、存储、ATS、HRIS/薪酬、可观测性、营销/其他）。另外包含 49 个多集成链式攻击场景（本论文未作为主要实验）。
- **基准（Benchmark）**：AgentRedBench 本身即为基准，用于评估无防护 ASR 和防护后 ASR。
- **对比方法**：
  - **无防护（No-guard）**：8 个前沿 API 模型：Anthropic（Sonnet 4.6, Haiku 4.5）、OpenAI（GPT-5.4, GPT-5.4-mini, GPT-5.4-nano）、Google（Gemini 3.1 Pro, Gemini 3 Flash, Gemini 3.1 Flash Lite）。
  - **防护对比**：4 个开源防护模型：Llama Guard 3 (8B)、PromptGuard 2 (22M)、ProtectAI (184M DeBERTa-v3 分类器)、WildGuard (7B)。所有基线在阈值 0.5 下评估，并比较 TPR、FPR 和 ASR 降低。
- **评估指标**：攻击成功率（ASR，含部分成功）、任务完成率、过拒绝率、延迟（Token 和墙钟）、真阳性率（TPR）、假阳性率（FPR）等。
- **泛化实验**：跨集成保留测试（4 个从未出现在训练中的集成）和跨攻击类型保留测试（分别保留一种主动攻击类型训练并测试），验证泛化能力。

## 4. 资源与算力

- **防护模型训练**：AgentRedGuard-MiniLM (23M) 在一张 NVIDIA A10G GPU 上约 4 分钟完成；DeBERTa-v3-small (142M) 约 14 分钟。总训练算力小于 20 A10G 分钟。
- **推理开销**：AgentRedGuard-MiniLM 在 CPU（Apple M 系列单核）上中位延迟 9.5 ms，P99 10.5 ms，批处理 32 时每样本 3.7 ms，无需 GPU。
- **论文中未明确说明**：靶模型调用的 API 成本、攻击者 LLM 推理算力等未量化。

## 5. 实验数量与充分性

- **主要实验量**：
  - 无防护基线：8 个模型 × 215 场景 = 1720 次评估（其中部分场景因攻击者管道失败而排除，但跨模型比较充分）。
  - 防护评估：6 个防护模型（包括 AgentRedGuard 两种变体）× 同一面板，计算 ASR 降低、TPR/FPR。
  - 泛化实验：跨集成保留（4 个集成，2029 攻击行）和跨攻击类型保留（每个攻击类型单独保留），每组至少报告 TPR、FPR。
  - 攻击类型消融：表 2 报告每类攻击的 TPR。
- **充分性与公平性**：
  - AB 实验采用固定的动态攻击者和 LLM 裁判（Claude Haiku 4.5），减少变化因素。
  - 场景选择经过 Haiku 预过滤（216 个场景均至少有一次成功/部分成功），导致绝对 ASR 为上限，但跨模型相对排序有效。
  - 使用 bootstrap 置信区间报告 TPR/FPR，统计稳健。
  - 保留测试设计严谨（从未出现在训练数据中的集成和攻击类型），确保泛化结论可靠。
- **局限性**：未进行多集成链式攻击的主实验（仅提供场景）；未评估在线实时推理下带防护的端到端任务完成率（仅离线估计）；攻击者适应能力有限（仅 5 次尝试，无法发现全新攻击类别）。

## 6. 论文的主要结论与发现

- **无防护威胁严重**：面板平均 ASR 69.9%，范围 32%（Claude Sonnet 4.6）到 81%（Gemini 3 Flash）。模型间差异主要由对齐训练（而非规模）驱动：同一家公司的 Sonnet vs Haiku 差距达 47.4 个百分点。
- **AgentRedGuard 显著有效**：将面板 ASR 从 69.9% 降至 2.4%（降低 67.5 个百分点），假阳性率仅 0.37%，中位延迟 9.5 ms。在所有开源基线中，没有其他防护能同时达到高 TPR 和低 FPR：ProtectAI 虽有一定 TPR (15%)，但 FPR 高达 25%；其他基线 TPR 接近 0%。
- **泛化能力强**：跨集成保留测试 TPR 99.75%，FPR 0.37%；跨攻击类型保留测试中，每种漏掉的攻击类型召回率 ≥99.36%。证明分类器学习到通用表面标记而非特定类别特征。
- **防护与对齐互补**：AgentRedGuard 在最易受攻击的模型（Gemini 3 Flash）上降低 78.1 点 ASR，在最抵抗的 Sonnet 上降至 0%，表明对齐和推理时防护可叠加。

## 7. 优点

- **动态攻击生成**：避免了静态模板的可预测性，更真实模拟攻击者适应行为。
- **广泛的集成覆盖**：24 个企业级集成（9 个功能族），接近生产部署广度。
- **细致的攻击分类**：专注于“微妙未明确定义授权”攻击，而非简单越狱，更贴合实际威胁。
- **高性能轻量防护**：23M 参数 MiniLM 分类器，CPU 可部署，延迟极低，且 FPR 极低，适合生产环境。
- **严格的保真评估**：保留集成和攻击类型，证明泛化而非记忆；使用 bootstrap CI 报告统计显著性。
- **开放性与可复现**：代码、集成模式、防护模型开源；标准场景通过维护者渠道管理以保持有效性。

## 8. 不足与局限

- **场景选择偏差**：215 个场景是从 1,731 个候选中通过 Haiku 预过滤得到的（仅保留至少一个成功/部分成功的场景），导致报告 ASR 为上限，不能代表随机样本行为。
- **攻击者适应能力有限**：仅 5 次尝试，无法自主发现新攻击类别；多种情况下仅进行同义改写，结构性失败（例如目标从不调用写工具）无法突破。
- **未评估多集成链式攻击**：虽包含 49 个链式场景，但未在主实验中执行或报告结果。
- **端到端任务完成度未全面测量**：防护造成的过拒绝率仅在离线 FPR 上反映，未测量在线语境下任务完成率的变化（声明为后续工作）。
- **未研究其他攻击向量**：如直接提示注入、权重窃取、UI 重渲染等不在范围内。
- **模型面板受限**：缺少 Claude Opus 等顶级模型；攻击者和裁判固定为 Claude Haiku 4.5，可能对其他模型存在偏倚。
- **生产真实案例验证有限**：虽然采用了部分生产硬负例，但良性语料主要是合成数据，真实生产流量可能具有不同分布。

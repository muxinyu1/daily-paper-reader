## 摘要
开放的智能体平台允许社区贡献者发布可重用的技能，智能体可在运行时调用这些技能。这种可扩展性也带来了供应链风险：恶意贡献者可以将有害行为隐藏在表面上无害的技能中。然而，现有防御措施难以评估，因为缺乏同时衡量恶意技能检测和运行时验证的基准。我们提出了SkillVetBench，一个针对开放智能体技能生态系统的两阶段安全审查基准。第一阶段对每个技能的自然语言规范进行语义审查，以检测隐藏的恶意意图。第二阶段在仪表化的沙箱中执行被标记的技能，观察运行时行为并收集可审计的证据。我们从活跃的OpenClaw生态系统中的已确认恶意技能构建基准，包括来自最近ClawHavoc供应链攻击活动的样本。与仅静态方法不同，SkillVetBench通过执行轨迹验证检测到的威胁。我们的实验表明：(1) 仅语义和基于签名的基线方法不足，最多漏掉89%的恶意技能，这些技能的威胁来源于自然语言指令、多组件逻辑或跨组件交互；(2) 运行时攻击集中在少数高权限原语上，特别是exec、write_file、install_skill和spawn；(3) SkillVetBench提供了案例研究，其中沙箱执行直接通过具体的运行时证据支持恶意判定。

## Abstract
Open agent platforms allow community contributors to publish reusable skills that agents can invoke at runtime. This extensibility also creates a supply-chain risk: malicious contributors can hide harmful behavior inside skills that appear benign under superficial inspection. However, existing defenses are hard to evaluate because there is no benchmark that measures both malicious-skill detection and runtime verification. We present SkillVetBench, a two-stage security vetting benchmark for open agentic skill ecosystems. The first stage performs semantic vetting over each skill's natural-language specification to detect hidden malicious intent. The second stage executes flagged skills in an instrumented sandbox to observe runtime behavior and collect auditable evidence. We build a benchmark from confirmed malicious skills in the live OpenClaw ecosystem, including samples from the recent ClawHavoc supplychain campaign. Unlike static-only methods, SkillVetBench verifies detected threats with execution traces. Our experiments show that: (1) semantic-only and signature-based baselines are insufficient, missing up to 89\% of malicious skills whose threats arise from natural-language instructions, multicomponent logic, or cross-component interactions; (2) runtime attacks are concentrated in a small set of high-permission primitives, especially exec, write\_file, install\_skill, and spawn; and (3) SkillVetBench provides case studies in which sandbox execution directly supports malicious verdicts with concrete runtime evidence.

## 1. 核心问题与整体含义（研究动机与背景）
- **背景**：开放智能体平台（如OpenClaw、NanoClaw）允许第三方贡献者发布可重用技能，智能体在运行时调用这些技能。这种开放性带来了供应链攻击风险：恶意贡献者可将有害行为隐藏在表面良性的技能中，例如最近的ClawHavoc活动向平台注入了1,184个恶意技能。
- **核心问题**：现有防御措施（如官方静态扫描器、签名检测）缺乏系统性评估基准，无法同时衡量恶意技能的**检测**能力与**运行时验证**能力。传统的静态分析+动态分析组合在软件生态中已有成熟应用，但在智能体技能生态中，自然语言指令和智能体中介的工具使用构成新的攻击面，现有方法不适用。
- **研究目标**：构建一个两阶段安全审查基准SkillVetBench，既能检测静态语义中的恶意意图，又能通过沙箱执行确认运行时行为，提供可审计的证据。

## 2. 方法论：核心思想、关键技术细节、算法流程
- **核心思想**：安全审查必须在两个层面同时进行——语义意图与运行时行为。静态检查能确定技能声称的行为，但不能确定实际执行的行为；仅行为执行而缺乏语义归因则产生不可解释的日志。两阶段设计互补不足。
- **两阶段流程**（图2）：
  - **阶段1：语义分析（LLM-as-a-Judge）**：对完整技能工件（SKILL.md、可执行脚本、配置文件、工具接口）进行LLM裁判评估，沿五个安全维度打分：指令保真风险（IFR）、数据引力（DG）、动作不可逆性（AI）、爆炸半径（BR）、链式放大（CA）。每个维度0-3分，加权聚合为技能智能体风险分数（SARS），映射到良性（0-3.9）、可疑（4.0-6.9）、恶意（≥7.0）三级判定。同时识别7类漏洞：命令注入、提示注入、不安全文件操作、内存投毒、数据泄露、供应链、权限滥用。
  - **阶段2：程序分析（Docker沙箱）**：仅对阶段1判定为“可疑”或“恶意”且被独立市场扫描器（如VirusTotal、ClawScan）标记的技能执行沙箱执行。使用GPT-3.5-turbo作为智能体，对每个技能发出良性提示和对抗性提示，监控工具调用、网络活动、文件系统操作、子进程、凭证访问。产生261,891行网关日志，将观察事件映射到7类攻击类别，并通过执行轨迹证据将“可疑”升级为“恶意”。
- **关键技术细节**：
  - 使用CVSS v4.0作为补充评分框架，与SARS对比。
  - 漏洞分类覆盖代码执行威胁、数据/供应链威胁、指令层/智能体威胁。
  - LLM裁判可选不同模型（Qwen2.5-32B、Llama-3.2-3B-Ins、Llama-3.1-7B、Mixtral-8x7B），评估敏感性。
  - 单模型生成SARS评分公式：SARS = (2·IFR + 1.5·DG + 1.5·AI + 2·BR + 2·CA) / 2.7，归一化到0-10。

## 6. 主要结论与发现
1. **纯静态/签名方法严重不足**：最多漏报89%的恶意技能。指令层威胁（如提示注入、内存投毒）无代码级签名，被签名扫描器完全忽略。
2. **运行时风险集中在高权限原语**：exec、write_file、install_skill、spawn、subagent占绝大多数攻击事件；只读工具几乎不产生确认攻击。
3. **SkillVetBench实现零漏报**：在78个恶意技能上全部检出（0漏报），在22个良性技能上零误报。
4. **LLM裁判敏感性显著**：Qwen2.5-32B检测率95%，而Mixtral-8x7B仅35%，推荐使用≥7B参数且对齐良好的模型，或采用集成评估。
5. **SARS与CVSS v4.0存在分歧**：在数据泄露和供应链风险上，CVSS评分低（<Medium），而SARS评分高，表明静态评分无法捕捉组合性/指令层威胁。
6. **案例研究证实沙箱必要性**：如xiaohongshu_mcp技能在对抗性会话中触发cron 10次、覆盖SKILL.md、注入虚假基准结果等行为，仅在沙箱执行中显现。

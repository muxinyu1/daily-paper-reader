## 摘要
LLM多智能体系统通常通过自然语言对话或松散的结构化共享内存进行协调，使得中间状态难以验证、归因和审计。我们提出PatchBoard，一种基于模式的协作架构，用对共享结构化状态的经过验证的JSON Patch变更来替代智能体间对话。一个架构师智能体构建任务特定的模式和工作流规则，而一个确定性的内核在事务性地提交每个状态变更之前，根据模式约束、角色特定的写契约和运行时不变式对其进行验证。在630个匹配的ALFWorld场景中，PatchBoard实现了84.6%的成功率，相比之下LangGraph为30.8%，Flock为61.6%，同时每个成功任务的令牌数减少至45.5k，而它们分别为368.3k和64.2k。

## Abstract
LLM multi-agent systems often coordinate through natural-language dialogue or loosely structured shared memory, making intermediate state difficult to validate, attribute, and audit. We introduce PatchBoard, a schema-grounded collaboration architecture that replaces inter-agent dialogue with validated JSON Patch mutations over a shared structured state. An Architect agent constructs a task-specific schema and workflow rules, while a deterministic kernel validates each proposed state mutation against schema constraints, role-specific write contracts, and runtime invariants before committing it transactionally. On 630 matched ALFWorld episodes, PatchBoard achieves an 84.6% success rate, compared with 30.8% for LangGraph and 61.6% for Flock, while reducing tokens per successful task to 45.5k, compared with 368.3k and 64.2k, respectively.

### 2. 论文提出的方法论：核心思想、关键技术细节

- **核心思想**：用**JSON Patch变更**（validated mutations）替代智能体间的自然语言对话，所有状态更新必须经过一个确定性内核的验证才能提交。

    - **工人（Workers）**：通过受限的JSON Patch接口与共享状态交互，只能提出候选变更，不能直接修改状态。

    条件包括：语法正确、路径被角色写契约授权、补丁应用成功、结果状态符合模式、运行时不变式成立。
  - **预算化上下文视图**：内核根据工人的读契约和上下文预算，从全局状态中裁剪出有界视图，避免暴露无关信息，降低token消耗。
  - **事务日志**：记录每次提交和拒绝的补丁、工人ID、视图哈希等，支持完整回放和审计。

### 6. 论文的主要结论与发现

- **主结果**：PatchBoard在ALFWorld上达到**84.6%成功率**（533/630），远超LangGraph的30.8%和Flock的61.6%；每个成功任务仅消耗**45.5k tokens**，远低于LangGraph的368.3k和Flock的64.2k。
- **黑盒控件**：结构化JSON黑板（48.6k tokens/成功，77%成功率）优于普通黑板（95.5k tokens，77%），但均不及完整PatchBoard，说明事物性验证和写契约带来了额外增益。
- **消融分析**：去掉patch/schema接口导致成功率下降-0.15且token/成功翻倍（2.34x）；去掉上下文切片同样下降-0.15；其他组件影响较小。
- **敏感度**：较小的上下文预算（1k）反而表现最好；生成的任务特定模式优于固定模式。
- **故障隔离**：PatchBoard对无效JSON、错误路径/类型、未授权写入的污染率均为0；虚假声明仍可能通过（污染率0.43）；循环停滞终止率达0.96。
- **HotpotQA诊断**：三个系统答案准确率相近，但PatchBoard具有结构有效性优势，而语义正确性仍依赖检索和验证器设计。

### 7. 优点：方法或实验设计上的亮点

- **方法创新性**：
  - 首次将多智能体协作严格定义为**模式驱动的状态变更**，将对话转化为受限JSON Patch操作，大幅提升可审计性和可靠性。
  - 确定性内核负责所有验证和调度，**隔离模型输出与提交状态**，确保错误不会扩散。
  - **预算化上下文视图**有效降低token消耗，且不牺牲性能。
- **实验设计严谨性**：
  - 使用匹配的gamefiles和多个种子，基线系统配置相同，统计方法严谨（置信区间）。
  - 包含**黑盒控件**（区分共享内存与验证机制的贡献）、**消融实验**、**敏感度分析**和**故障注入**，全面评估组件贡献和鲁棒性。
  - **HotpotQA诊断**明确划清了结构有效性与语义正确性的边界，诚实指出了方法局限。
- **实用性强**：token消耗大幅降低，适合成本敏感场景；事务日志支持完整回放，便于调试和审计。

### 8. 不足与局限

- **语义正确性无保障**：确定性内核只能拒绝结构/授权违规，无法阻止模式有效但事实错误的更新。虚假声明仍可能通过（故障注入中污染率43%）。
- **依赖蓝图质量**：Architect生成的模式若过疏或过密，会限制系统效能；模式生成本身也依赖LLM质量。
- **评价范围局限**：最强证据来自ALFWorld（状态和成功条件清晰），开放域任务（如软件工程、科学发现）可能需要更丰富的模式和验证。
- **工程开销**：需要架构设计、模式定义、切片逻辑等额外工作，成本较高，适用于对可审计性和状态完整性要求严格的任务，不适合简单对话。
- **模型依赖**：所有实验使用单一模型（Qwen-plus），泛化性未验证。
- **潜在伦理风险**：用户可能误将模式有效性当作事实正确性，产生虚假安全感；事务日志可能包含敏感信息，需要访问控制和脱敏机制。

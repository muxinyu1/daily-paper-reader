## 摘要
自主代理正越来越多地连接到云、部署和数据控制工作流，但生产环境的变更权限不应存在于非确定性推理过程中。现有的访问控制机制对身份进行授权，而保证层对提议的操作进行认证；两者单独均未在变更发生时提供对认证权限的强制执行点。本文引入了主权执行代理（SEB），这是一种针对证书绑定的自主基础设施的运行时强制边界。SEB 使用主权保证边界（SAB）颁发的证书，验证请求的变更是否匹配认证的执行合约，检查有效期窗口、策略纪元、撤销纪元和实时状态偏移，铸造限定范围的执行身份，调用基础设施 API，并记录签名的决策和结果记录。通过分离提议、准入和执行，SEB 将认证权限转化为短期的、可撤销的、可审计的运行时能力，前提是生产环境变更 API 拒绝非代理身份。我们介绍了 SEB 执行模型、证书与重放验证谓词、限定范围的身份语义、防绕过部署模式、故障行为以及一个具体的原型实现。我们在 AWS 和 Kubernetes 集群上评估了该原型，测量了延迟开销、撤销传播、偏移检测以及在故障注入下的安全性。

## Abstract
Autonomous agents are increasingly connected to cloud, deployment, and data-control workflows, but production mutation authority should not reside inside non-deterministic reasoning processes. Existing access-control mechanisms authorize identities, while assurance layers certify proposed actions; neither alone provides a mandatory enforcement point for certified authority at the moment of mutation. This paper introduces the Sovereign Execution Broker (SEB), a runtime enforcement boundary for certificate-bound agentic infrastructure. SEB consumes certificates issued by the Sovereign Assurance Boundary (SAB), verifies that the requested mutation matches the certified execution contract, checks validity windows, policy epochs, revocation epochs, and live-state drift, mints scoped execution identity, invokes infrastructure APIs, and records signed decision and outcome records. By separating proposal, admission, and execution, SEB turns certified authority into a short-lived, revocable, auditable runtime capability, provided that production mutation APIs reject non-broker identities. We present the SEB execution model, certificate and replay-verification predicates, scoped identity semantics, bypass-prevention deployment patterns, failure behavior, and a concrete prototype implementation. We evaluate the prototype on AWS and Kubernetes clusters, measuring latency overheads, revocation propagation, drift detection, and security under fault injection.

### 2. 论文提出的方法论：核心思想、关键技术细节

-   **核心思想：** **分离提议、准入和执行**三阶段，构建一个强制性的运行时执行边界。具体来说：
    1.  **代理提议：** 智能体提出变更计划。
    2.  **SAB 准入：** SAB 验证提议的安全性，若安全则签发一个加密签名的证书 £Ω£。
    3.  **SEB 执行：** 所有变更请求必须经过 SEB。SEB 验证 £Ω£ 的有效性、检查当前状态和外部环境，只有一切通过后，才铸造一个**最小权限的、短时的执行身份**去调用基础设施 API，并记录所有决策和结果。
-   **关键技术细节：**
    -   **执行接口：** 定义了正式的执行函数 `Execute(Ω, req, St, Platform) -> D | O`，同时定义了合约 `C`、证书 `Ω` 和执行请求 `req` 的详细数据结构。
    -   **验证流水线：** 提出了一系列验证谓词，构建了一个顺序执行的管道，任何一步失败都会产生签名的“决策记录”：
        -   **签名验证 (£Φₛᵢg£)**：验证证书 £Ω£ 上 SAB 的数字签名。
        -   **合约匹配 (£Φₘₐₜcₕ£)**：验证执行请求 `req` 和证书 £Ω£ 中的合约 `C` 严格一致（操作、目标、参数）。
        -   **有效期检查 (£Φₜᵢₘₑ£)**：确认当前时间在证书的有效窗口内。
        -   **策略纪元检查 (£Φₚₒₗᵢcy£)**：确认证书签发的策略版本与当前活动策略版本一致。
        -   **撤销纪元检查 (£Φᵣₑᵥ£)**：确认证书颁发时的撤销纪元与当前全局撤销纪元一致，若已更新则证书被撤销。
        -   **实时状态漂移检查 (£Φdᵣᵢfₜ£)**：通过查询实时基础设施状态 `St`，与证书签发时的证据状态 `Eadmit` 进行对比，若状态变化超过定义的容忍度（`ϵC`），则拒绝执行。这是解决 TOCTOU 问题的关键。
        -   **重放攻击防护 (£Φᵣₑₚₗₐy£)**：使用数据库的原子性操作来预留证书唯一识别码和随机数（cid, nonce），确保一个证书只能被执行一次。
        -   **平台可作用域性检查（Scopeable）**：验证合约 `C` 中的约束能否被目标平台（如 AWS IAM）或强制代理/准入控制器强制执行。如果不能，则拒绝执行。
    -   **短时作用域身份：**
        -   **原则：** 智能体及其运行时**不持有任何静态的生产环境变更凭证**。
        -   **机制：** SEB 在验证成功后，通过调用身份提供者（如 AWS STS, K8s TokenRequest API）动态铸造一个**短期、受限的执行身份（ID_exc）**。该身份通过**会话策略、会话标签**等方式，将其权限精确绑定到合约 `C` 中定义的操作、资源、参数和有效期。
        -   **部署模式：** 必须通过 IAM 策略、RBAC 规则或准入 Webhook 等技术，**拒绝所有非 SEB 发起的变更请求**，确保 SEB 是唯一的变更路径。这是实现“免绕过”的安全定理的关键前提。
    -   **副本与审计：** 所有验证失败和成功执行的尝试都会生成经过签名的“决策记录”或“结果记录”，并写入一个不可篡改的账本（PostgreSQL 数据库），提供完整的审计链条。

### 6. 论文的主要结论与发现

1.  **SEB 能有效填补安全鸿沟：** SEB 成功地将“认证的权限”从证书转化为“运行时强制执行的、短时、可撤销的权限”，在分离提议、准入、执行的架构中扮演了关键的执行角色。
2.  **可接受的延迟开销：** SEB 在验证和执行过程中引入的延迟是可管理的：Kubernetes/容器的 Broker 内部开销约 **28.2 毫秒**（p50），AWS 安全组的约 **136.9 毫秒**（p50）。这在**高风险、低频次**的变更任务中是完全可以接受的。
3.  **极强的安全防护能力：** 在注入的 9 种安全威胁（未认证、重放、状态过期等）测试中，SEB 实现了 **100% 的拒绝率**和 **0% 的未授权执行率**，证明了其设计能有效抵御此类攻击。
4.  **延迟瓶颈主要在外部系统：** 性能瓶颈不在于内部的加密验证和逻辑处理，而在于**实时漂移检查、凭证铸造**等需要调用外部云服务的操作，以及**数据库非ce 写入**。
5.  **AWS STS 成为吞吐量瓶颈：** AWS STS 的 API 调用限制了系统的吞吐量，峰值为 240 req/s，而 Kubernetes TokenRequest 在实验室测试中能达到 820 req/s。这说明云平台本身的限流是部署中的一个重要因素。

-   **方法的严谨性与形式化：** 论文用形式化的谓词逻辑清晰定义了验证流水线 (`Φsig`, `Φmatch`, `Φtime` 等) 和安全性定理 (`Theorem 1`)，为系统设计提供了严格的数学基础。

-   **实验设计的多维度与深度：** 不仅给出了端到端的延迟，还通过**微基准**和**消融实验**分析了延迟来源，并通过**安全漏洞注入**和**故障模拟**验证了系统在恶劣环境下的可靠性。

### 8. 不足与局限

-   **对云提供商的信任依赖：** SEB 的安全性高度依赖于 AWS、GCP 等底层云平台 IAM 系统的正确实现和完整性。如果 IAM 层存在漏洞或遭绕过，SEB 的安全定理将不成立。
-   **“冷启动”延迟与性能假设：** 声称“对于高风险、低频次任务，延迟可接受”，但未详细评估对于微秒级延迟要求的控制环路（如实时容器调度）的不可用性。动态铸造权限引入了全局性的网络延迟（云 STS 调用），对于极低延迟场景是灾难性的。
-   **单点故障风险（SPoF）：** SEB 作为所有变更请求的强制门卫，其自身成为了一个关键的单点故障。虽然论文提到了高可用部署，但故障恢复的复杂性和开销（例如，需要原子性恢复非ce/凭证状态）可能增加整体系统的脆弱性。
-   **状态观察的最终一致性风险：** 实时漂移检查依赖于云 API 返回的“实时”状态。然而，云基础设施的状态通常是最终一致的。在`time of check`和`time of use`之间，状态可能再次波动，导致“监控器–执行器”的时间窗口存在 TOCTOU。
-   **跨云和多账户部署的复杂性：** 在异构的多云环境或大规模多账户部署中，维护统一信任根、全局撤销纪元以及一致的代理层会面临巨大困难和复杂挑战。
-   **语义安全的外部性：** SEB 是一个执行时间的强制边界，它确保请求匹配证书，但无法验证证书中的合约本身是否语义安全。如果 SAB 本身由于验证器被攻破而签发了错误证书，SEB 依然会执行它。
-   **性能开销的具体场景局限性：** 测试环境是私有 VPC（1.1ms 延迟），实际公网或混合网络环境下的性能飙升会更大。此外，对于频繁执行的自动化任务（如自动扩缩容、健康检查后滚动更新），SEB 的持续延迟叠加可能带来可感知的运营延迟。

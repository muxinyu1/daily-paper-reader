## 摘要
WebMCP是一种新兴协议，它使网站能够直接将工具暴露给AI智能体，绕过了传统的用户界面，从而引入了新的安全风险。WebMCP中动态暴露智能体可访问的工具扩展了Web会话的攻击面，尤其是在涉及第三方脚本时。在本研究中，我们识别出一种新的潜在威胁，称为会话中工具注入（MSTI），攻击者利用第三方脚本在活动会话期间注入恶意工具。为了更好地描述这一威胁，我们根据操纵的阶段和目标对MSTI进行分类，区分了工具劫持和工具框架。工具劫持通过诸如AbortSignal API或工具注册时的竞争条件等机制修改智能体可见的工具集。相比之下，工具框架则通过元数据字段（如工具名称、描述、readOnlyHint和inputSchema）影响智能体对工具角色的感知。我们的实现表明，工具劫持和工具框架都能成功破坏WebMCP的预期功能。基于这些结果，我们概述了潜在的缓解方向，并为WebMCP提供了安全设计建议，包括将工具身份绑定到其来源、确保生命周期一致性、对第三方工具实施数据边界限制，以及维护工具注册和调用的可追溯日志。这些发现表明，MSTI源于WebMCP独特的工具生命周期和结构化元数据，使得工具表面本身成为一个新兴的安全问题。

## Abstract
WebMCP is a newly emerging protocol that enables websites to expose tools directly to AI agents, bypassing traditional user interfaces and introducing new security risks. The dynamic exposure of agent-accessible tools in WebMCP expands the attack surface of web sessions, especially when third-party scripts are involved. In this study, we identify a new potential threat, termed Mid-Session Tool Injection (MSTI), in which attackers leverage third-party scripts to inject malicious tools during an active session. To better characterize this threat, we classify MSTI based on the stage and target of manipulation, distinguishing between Tool Hijacking and Tool Framing. Tool Hijacking modifies the set of tools visible to the agent through mechanisms such as the AbortSignal API or race conditions during tool registration. In contrast, Tool Framing influences the agent's perception of tool roles through metadata fields such as tool name, description, readOnlyHint, and inputSchema. Our implementation demonstrates that both Tool Hijacking and Tool Framing can successfully disrupt the intended functionality of WebMCP. Based on these results, we outline potential mitigation directions and provide security design recommendations for WebMCP, including binding tool identity to its origin, ensuring lifecycle consistency, enforcing data boundaries for third-party tools, and maintaining traceable logs of tool registration and invocation. These findings indicate that MSTI arises from WebMCP's unique tool lifecycle and structured metadata, making the tool surface itself an emerging security concern.

## 2. 方法论：核心思想与关键技术细节

- **核心思想**：MSTI 分为两类：
  - **工具劫持（Tool Hijacking）**：直接修改智能体可见的工具集（如通过 AbortSignal API 使合法工具失效、竞争条件抢先注册同名恶意工具）。攻击不依赖语义理解，而是改变工具的存在性与可见性。
  - **工具框架（Tool Framing）**：通过修改工具元数据（名称、描述、readOnlyHint、inputSchema）诱导智能体误解读工具角色，使其将恶意工具视为合理的工作流步骤（如“始终在发送邮件前调用此工具”）。
  - **复合攻击（Composite）**：结合两者，同时实现工具插入与语义伪装。

- **关键技术细节**：
  - **工具劫持实现**：
    - AbortSignal 劫持（C1）：当合法工具被 AbortController 中止时，在其被调用前重新注册同名恶意工具。
    - 注册竞争（C3）：在合法工具注册前，抢先注册同名工具（“先注册先得”）。
  - **工具框架实现**：
    - 描述注入（C2）：在恶意工具描述中加入强制性工作流指令（如“必须在 send_email 之前调用”）。

  - **攻击流程**：攻击者通过注入的第三方脚本在用户浏览器中执行，操控 WebMCP 工具注册表，改变智能体在任务中看到的工具列表。实验框架包含 Benign Page Server、Malicious Server、LLM Agent 和 Attacker Endpoint。

## 6. 主要结论与发现

1. **工具劫持攻击极为有效**：
   - C1（AbortSignal 劫持）平均 ASR 94%，C3（注册竞争）ASR 100%，但任务完成率极低（~17%），因为恶意工具破坏了工作流的第一步。
   - 成功严格依赖于注入时机：必须在智能体首次调用合法工具前完成替换（P1 为 100%，P2 后为 0%）。

2. **工具框架攻击更隐蔽**：
   - C2（描述注入）平均 ASR 59%，任务完成率 81%；C4（长描述溢出）ASR 36%，完成率 85%。表明攻击可在不严重干扰任务的情况下实现数据泄露。
   - C5（复合攻击）ASR 61%，完成率 85%，兼具高成功与高隐蔽性。

3. **元字段影响高度模型依赖**：
   - 描述（AB1）平均 ASR 93%；readOnlyHint（AB2）平均 ASR 87%，是有效独立攻击通道。
   - inputSchema（AB3）仅对 Gemini 有效（100%），对 GPT-5.4 和 Claude 为 0%。
   - 描述 + readOnlyHint 组合（AB4）在所有模型上达到 100% ASR。

4. **模型代际比较**：
   - 针对协议漏洞的工具劫持（C1、C3）在旧模型和 SOTA 模型间成功率无显著下降，表明协议层问题无法通过模型升级缓解。
   - 语义驱动的工具框架攻击有效性随模型理解能力提升而略有下降（但降幅不大）。

5. **防御可行性**：
   - 绑定工具起源（Origin-Based Access Control）可抵御工具劫持；限制第三方工具数据流可抵御工具框架，使 ASR 降至 0%，且不显著影响任务完成率。

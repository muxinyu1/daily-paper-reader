Abstract 

LLM-based guardrails typically safeguard agents by evaluating proposed actions or in-puts before execution, producing safety signals such as binary allow/deny decisions, risk cate-gories, and/or explanatory rationales about po-tential policy violations. However, agent risks often arise when otherwise benign tasks are contaminated by untrusted external content, un-safe instructions or risky tool use. Existing guardrails often flag the entire task uniformly as unsafe, thereby blocking the threat but sac-rificing the benign part. Moreover, existing work largely evaluates guardrails in isolation, leaving unclear whether their interventions lead to safer downstream agent behavior. To ad-dress this, we introduce TRIAD (Tripartite Response for Iterative Agent Guardrailing), a guardrail-integrated agent framework that lever-ages guardrail-generated verbal feedback as a guiding signal to keep the agent aligned with the benign objectives at each planning step. We finetune a language model on a self-curated training dataset to output one of three decisions: 

proceed , refuse , or update , together with struc-tured natural-language feedback. Rather than merely allowing or blocking execution, update 

guides the agent to revise its plan, avoid harm-ful components, and preserve the benign task where possible. TRIAD injects this feedback into the agent’s context, enabling subsequent plan revision and forming a closed loop be-tween guardrail feedback and agent planning. Extensive experiments on ASB and AgentHarm show that TRIAD reduces the average attack success rate to 10 .42% , while achieving the best safety–utility trade-off among guardrail-integrated baselines. Our code is available at: 

https://github.com/YUHAOSUNABC/TRIAD .

For example, computer-using agents may delete backups or overwrite configura-tion files, making file-system operations difficult to reverse (Chen et al., 2025; Kuntz et al., 2025; Cao et al., 2025). In such workflows, ReAct agent framework follows a reasoning-then-acting loop, where the agent explicitly plans next action before execution. This makes the planning stage a pivotal point for detecting unsafe behavior before tool exe-cution (Yao et al., 2023; Shinn et al., 2023; Huang et al., 2025). Therefore, planning-stage guardrails are needed to intervene before unsafe or irreversible actions are executed. Existing planning-stage guardrails typically re-spond to security threats by flagging risky inputs or action plans before actual execution, and then de-ciding whether to allow or block them (Chennabas-appa et al., 2025; Huang et al., 2025; Mou et al., 2026). However, harmful inputs in agent work-flows often appear within tasks that are not uni-formly harmful: a user request or interaction his-tory may contain both a benign goal and an unsafe instruction, especially under Prompt Injection At-tacks (PIAs). In such cases, existing guardrails tend to treat the entire task or the action plan as unsafe, which can block some threats but also sacrifices the original benign task. To validate this limita-tion, we evaluate representative guardrails against PIAs on AgentSecurityBench (ASB) (Zhang et al., 2025a). As shown in Figure 1 Left, these guardrails achieve an average recall of only 58 .57% ; even among the cases where risks are detected, fewer than 37 .26% of attacks are successfully blocked, and the agent preserves the original benign task in less than 2.31% of cases. Together, these re-sults show that allow-or-block guardrail outputs are insufficient for partially unsafe scenarios. To address this, we propose TRIAD (Tripartite Response for Iterative Agent Guardrailing), an end-to-end guardrail framework that inspects the tar-get agent’s action plan at each planning step and regulates downstream agent behavior before exe-cution. Unlike existing guardrails that often treat partially unsafe tasks as entirely unsafe, TRIAD in-corporates Tri-Guard to generate structured natural-language (NL) feedback and a three-way guardrail decision: P ROCEED , U PDATE , or R EFUSE . CEED allows execution, R EFUSE blocks purely harmful requests, and U PDATE injects the guardrail feedback into the agent’s context through an ICL template, guiding the agent to revise unsafe plans while preserving the benign user goal. This end-to-end design turns guardrail outputs from static risk detection signals into actionable guidance for downstream agent planning. To train Tri-Guard, we construct a dataset from 5288 agent trajectories, using GPT-5.4 distillation to obtain structured NL feedback and three-way decision supervision. We evaluate TRIAD on ASB (Zhang et al., 2025a) and AgentHarm (Andriushchenko et al., 2025), covering various attack settings across both open-weight and proprietary agents. Across all set-tings, Tri-Guard-integrated TRIAD reduces the av-erage ASR from 74 .45% to 10 .42% while improv-ing the average TSR from 28 .45% to 68 .60% com-pared with the ReAct baseline. In particular, under PIAs, Tri-Guard improves benign task success by routing partially unsafe plans to U PDATE rather than over-refusing the entire task, as illustrated in Appendix I. Moreover, Tri-Guard-integrated TRIAD also achieves the strongest safety–utility balance on AgentHarm, reaching the highest aver-age Helpfulness–Safety (HS) score of 80 .92 across four target agents. Finally, TRIAD points to a broader design paradigm for agent guardrails: rather than being used as external risk detectors or binary blockers, guardrails can be integrated into the agent planning loop to provide actionable feed-back, guide plan revision, and preserve legitimate user goals under partially unsafe conditions. 

2 Problem Settings and Related Work 

ReAct. ReAct is an LLM-based agent frame-work that follows a reasoning-then-acting loop, where at each step the agent plans its next ac-tion and then executes it through a tool call (Yao et al., 2023). Given a task Q and a toolkit T =

{u1, u 2, . . . , u K }, the agent interacts with the en-vironment over a sequence of steps t = 1 , 2, . . . , n ,where n denotes the trajectory length. At the planning stage of step t, the agent generates a NL plan pt conditioned on the interaction history 

Ht−1 = Q, (p1, a 1, o 1), . . . , (pt−1, a t−1, o t−1) ,which serves as the agent’s context. At execution, the agent selects an action at = ( ut, x t) condi-tioned on pt and Ht−1, where ut ∈ T denotes the tool and xt denotes its arguments. Executing this action yields an observation ot, i.e., the tool re-sponse returned to the agent. After execution, Ht−1

is updated by appending (pt, a t, o t). The loop ter-minates when the agent decides that no further tool call is needed or when the execution is stopped by the system (Zhang et al., 2024; Andriushchenko et al., 2025). Since ReAct generates explicit plans 2Training Data Generation Pipeline

Threat Model. We assume that the adversary pur-sues an attacker goal Gadv , with the objective of steering the target agent toward completing an at-tacker task. We consider three attack settings: (i) In-direct Prompt Injection (IPI) , where the adversary contaminates the observation returned by an exe-cuted tool. At some step t, the malicious instruction is injected into the tool response, ˜ot = ot ⊕ Qadv ,with templates shown in Appendix D. The contam-inated observation is appended to the interaction history and affects subsequent planning steps. (ii) Direct Harmful Task (DH) , where the adversary acts as the user and directly issues a harmful task, i.e., Q = Qadv . Since the user goal is unsafe, the agent’s compliance with the task advances Gadv .

(iii) Direct Prompt Injection (DPI) , where the ad-versary injects malicious instructions into the be-nign user task itself, forming a contaminated task 

˜Q = Qben ⊕ Qadv . In practice, such direct access to the user’s task is difficult to obtain, so DPI is used as a controlled stress-test for studying strong task-level contamination. Across all settings, the adversary operates in a black-box setting, with no access to the parameters or architectures of the tar-get agent. In PIAs, the agent is still expected to preserve the user goal Gben while avoiding actions that advance Gadv . In contrast, DH directly speci-fies a harmful user goal and should be refused. 

Guardrail Models for LLM Agents. Guardrail models for LLM agents have evolved from static risk detectors into runtime monitors for agent work-flows. Early approaches extend conventional mod-eration to agentic settings, using layered classi-fiers, LLM-based auditors, or formally specified safety policies to detect whether an input, output, or execution trace violates predefined safety re-quirements (Chennabasappa et al., 2025; Xiang et al., 2025; Miculicich et al., 2025). These meth-ods improve safety awareness in agent systems, but their interventions are formulated as allow-or-block judgments, and often depend on task-specific guard requests or domain-specific policy specifications. More recent studies move the guardrail closer to the agent execution process, checking intermedi-ate action plans before irreversible tool execution. For example, AGrail performs iterative runtime safety checking with cooperative LLMs (Luo et al., 2025), Safiron trains a planning-stage guardrail on synthetic trajectories to flag risky plans and pro-vide risk rationales (Huang et al., 2025), and We-bGuard predicts action-level risk labels for web agents (Zheng et al., 2025). These methods rec-ognize that agent safety failures often emerge dur-ing multi-step execution rather than from the ini-tial user request alone. However, most of them still treat guardrailing primarily as risk detection or action blocking, providing limited support for revising partially unsafe plans while preserving the legitimate task objective. Close to our setting, ToolSafe introduces TS-Guard as a step-level guardrail for LLM agents (Mou et al., 2026). TS-Guard assigns a scalar safety rating to each tool-use decision and converts it into a binary verdict with a fixed thresh-old. It interacts with a short textual hint used to guide the downstream agent. In contrast, our method treats guardrailing as an end-to-end system: the guardrail produces structured verbal feedback and three-way decisions, and different decisions affect the downstream agent’s behavior in varying ways. This enables the agent to distinguish harm-less plans, DH tasks that should be refused, and partially unsafe intent that should be revised.

Hence, we construct a training dataset that contains both benign and harmful trajecto-ries, together with corresponding safety analyses to support feedback-driven guidance for downstream agents. The training data are constructed through a four-stage pipeline as shown in Figure 2, with expanded descriptions of each module and a con-tamination analysis provided in Appendix C. 

(i) Task Collection. First, we collect agent tasks from agent-safety benchmarks, including InjecA-gent (Zhan et al., 2024) and AgentAlign (Zhang et al., 2025b), and DH tasks from jailbreak bench-marks, including JailbreakBench (Chao et al., 2024) and HarmBench (Mazeika et al., 2024), cov-ering the three attack settings defined in Section 2. We use GPT-5.4 as a task rewriter to convert these collected tasks into coherent multi-turn agent tasks. 

(ii) Trajectory Generation. We then generate real interaction trajectories by letting agents execute these tasks in tool environments. After rollout, we assign ground-truth decisions based on the task set-ting and the observed tool calls, since the called tools directly indicate whether the trajectory com-pletes the benign task or carries out the harmful objective. (iii) Knowledge Dis-tillation. We use GPT-5.4 as a teacher model to annotate each retained trajectory with structured NL feedback and a three-way decision, as detailed in Section 4. The feedback is required to provide concrete safety analysis along predefined dimen-sions, including user intent, agent reasoning, cur-rent action, alignment check, and security check. We discard teacher annotations whose final deci-sion is inconsistent with the ground-truth decision, ensuring that the retained feedback is both infor-mative and label-consistent. (iv) Data Pair Con-struction. Finally, we pair each retained trajectory with its corresponding teacher feedback and con-vert them into a query–response format for SFT training. The query consists of the agent trajectory, including the task context, interaction history and proposed action plan, while the response contains the structured feedback and the final decision. 

4 Method 

4.1 Overview of TRIAD 

TRIAD is an end-to-end guardrail-integrated agent framework that incorporates Tri-Guard to inspect the target agent’s planning-stage information and route the agent to safer planning before tool exe-cution. We demonstrate the pipeline of TRIAD in Figure 3 and describe its algorithmic workflow in Algorithm 1. Specifically, at each planning step 

t, the ReAct agent generates a NL plan pt con-ditioned on the interaction history Ht−1, where the plan specifies the intended next action at. Be-fore at is executed, Tri-Guard inspects the user task Q, the interaction history Ht−1, the proposed plan and action (pt, a t), the available toolkit T ,and outputs NL feedback ft with a decision dt ∈{PROCEED , UPDATE , REFUSE }. A P ROCEED de-cision executes the current action, an U PDATE deci-sion injects the feedback into the agent’s temporary context through an In-context Learning (ICL) tem-plate to guide plan revision, and a R EFUSE deci-sion guides the agent to produce a refusal response without tool execution. The update process can be repeated up to a maximum number of attempts, allowing TRIAD to regulate downstream agent be-havior through iterative feedback-driven planning rather than merely blocking unsafe actions. 4.2 Training of Tri-Guard 

Motivated by the limited performance of existing guardrails under PIAs, we train Tri-Guard to per-form trajectory level risk analysis at the planning stage. As shown in Figure 1, existing guardrails miss many PIA risks, with an average recall of only 

58 .57% . Moreover, even when unsafe patterns are detected, allow-or-block signals alone do not re-liably guide the downstream agent to avoid the injected instruction while preserving the original benign task. Therefore, detection alone is insuffi-cient: a guardrail must distinguish benign action plans, partially unsafe plans that require revision, and purely harmful requests. We train Tri-Guard to produce structured NL feedback together with a three-way decision for each action plan. We use Qwen3.5-9B as the base model for Tri-Guard, as it provides a practical balance between model capacity and inference efficiency. Let gθ de-note Tri-Guard. At each planning step, Tri-Guard learns the mapping: 

gθ : ( Ht−1, p t, a t, T ) 7 → (et, d t),

where Ht−1 is the interaction history including the user task Q, pt is the current action plan, at is the proposed action, and T is the available toolkit. For training, we serialize the guardrail input as xi and use the teacher generated feedback and decision as the target completion yi = ( e⋆i , d ⋆i ).We train gθ with weighted Supervised Finetun-ing (wSFT). For each retained teacher completion, we use the teacher model’s average token log likeli-hood over the completion as a confidence score and normalize it into a sample weight wi. High con-fidence completions receive larger weights, while lower confidence samples still contribute with re-duced influence. We apply these weights with a completion only SFT objective that supervises only the generated feedback and decision tokens. The full objective is provided in Appendix B.

4.4 Difference from ToolSafe 

Both ToolSafe and TRIAD train a dedicated guardrail model and integrate it into ReAct-style agents before tool execution. However, TS-Guard is trained with multi-task RL objective to pro-duce step-level safety ratings, whereas Tri-Guard is fine-tuned through knowledge distillation data and wSFT to generate structured feedback and a three-way decision for trajectory-level intervention. At the interaction level, ToolSafe mainly provides a generic textual hint after detecting risky tool use, such as “Perhaps you should try other safer tool calls”, whereas TRIAD explicitly translates guardrail feedback into downstream agent control through decision-conditioned ICL templates and closed-loop plan revision. Also, ToolSafe focuses on unsafe tool invocations, whereas TRIAD targets partially unsafe trajectories under DPI/IPI, where the goal is to revise the compromised plan while preserving the original benign task.

Metrics. For ASB, Attack Success Rate (ASR) 

measures the proportion of test cases where the agent achieves the attacker goal, Task Success Rate (TSR) measures whether the original benign task is completed under PIA, and Refusal Rate (RR) measures how often the agent refuses to an-swer. For AgentHarm, we report the Harm Score ,which measures whether the agent completes DH tasks, and propose a Helpfulness–Safety score (HS) to capture the trade-off between benign-task utility and harmful-task refusal. A higher HS indi-cates a better balance between preserving helpful behavior on benign tasks and avoiding harmful behavior on DH tasks. Details of the metric com-putation are in Appendix E. 

Agents. To evaluate whether TRIAD generalizes across different agents, we choose four backbones: open-weight models, Qwen3-32B (Qwen Team, 2025) and Kimi-2.5 (Kimi Team et al., 2026), and proprietary models, GPT-5.1 (OpenAI, 2025) and Gemini-2.5-Pro (Gemini Team, 2025). 

Baselines. For each agent backbone, we compare four settings. First, ReAct denotes the original agent without any guardrail intervention, serving as the no-defense baseline. Second, ToolSafe (Mou et al., 2026) is a recent guardrail-integrated agent framework that introduces TS-Guard as a step-level guardrail model for detecting unsafe tool invoca-tions. Third, TRIAD + TS-Guard instantiates our framework with TS-Guard as the guardrail model, allowing us to compare different guardrail mod-els under the same framework. Finally, TRIAD + Tri-Guard is our full framework, where Tri-Guard provides structured natural-language feedback and three-way decisions to guide downstream agent planning. Further baseline comparisons under ad-ditional settings are provided in Appendix H. 

5.2 Results and Analyses 5.2.1 Overall Performance 

Table 1 reports the overall performance across ASB-DPI, ASB-IPI, and AgentHarm. Compared with the ReAct baseline, TRIAD + Tri-Guard con-sistently reduces ASR across both open-source and proprietary agent backbones, showing that the guardrail can generalize to heterogeneous down-stream agents. This indicates that Tri-Guard can reliably identify compromised action trajectories before unsafe tool execution. A key observation is that lower ASR alone does not fully character-7ize guardrail quality. ToolSafe and TRIAD + TS-Guard often achieve strong attack suppression, but this is frequently accompanied by low TSR and high RR, suggesting that these methods suppress attacks largely by blocking or discouraging exe-cution. Such behavior improves safe interactions but sacrifices the original benign task. In contrast, TRIAD + Tri-Guard achieves the best TSR in every ASB setting while keeping ASR low. The U PDATE 

decision and structured NL feedback enable the agent to revise compromised plans instead of treat-ing the whole trajectory as unsafe. On AgentHarm, TRIAD + Tri-Guard also im-proves the helpfulness–safety trade-off by reducing harmful-task completion while preserving benign-task utility. For instance, HS increases from 36 .28 

to 80 .32 on Qwen3-32B, while Harm Score drops from 77 .04 to 9.87 . This shows that TRIAD is not only effective for PIAs, but also helps the agent refuse DH tasks. Overall, these results suggest that the three-way decision design of Tri-Guard and the end-to-end integration of TRIAD provide a more balanced safety–utility trade-off than binary blocking-oriented guardrails. Additional experi-mental results are provided in Appendix H, and complete interaction histories are provided in Ap-pendix I. We also compare the computational over-head of integrating TRIAD in Appendix G. 

5.2.2 TRIAD with Different Guardrails 

Table 2 shows that integrating prior guardrails into TRIAD framework is insufficient to guide the agent’s subsequent action planning. Although ex-isting guardrails can reduce ASR compared with the ReAct baseline, they often do so by producing overly conservative decisions, leading to high re-fusal rates and poor benign task preservation. For example, TS-Guard achieves low ASR under DPI and IPI, but its RR reaches 88 .80% and 94 .63% ,causing TSR to drop to 1.33% and 0.59% . This suggests that off-the-shelf guardrails are often ef-fective at suppressing unsafe tool use, but are not directly suitable for revising partially unsafe plans. The comparison with existing guardrails and the Qwen3.5-9B base model further highlights the need for trajectory–feedback finetuning. These models can detect unsafe patterns, but they tend to treat partially unsafe trajectories as cases to be blocked rather than revised. By training on trajectory– feedback pairs, Tri-Guard learns not only to iden-tify risks, but also to decide whether the agent should proceed, update, or refuse. 

5.2.3 Trajectory–Feedback Training Shifts the Safety–Utility Trade-off 

Table 3 compares the base model and Tri-Guard un-der the same TRIAD framework. The base model exhibits conservative behavior: it often achieves lower ASR and higher RR than Tri-Guard, suggest-ing that the pretrained model has a strong tendency to identify risky patterns and suppress unsafe tool use. However, this safety gain is obtained by treat-ing partially unsafe trajectories as fully harmful, which leads the downstream agent to abort rather than revise the plan. Al-though its ASR is slightly higher than the base model on average, its TSR improves substantially, indicating that the model learns to preserve the legitimate user goal in PIA cases instead of reject-ing the whole trajectory. 6 Conclusion 

In this paper, we propose TRIAD, an end-to-end guardrail-integrated agent framework that connects planning-stage safety analysis with downstream agent behavior through feedback-driven plan revi-sion. By training Tri-Guard on trajectory–feedback pairs, Tri-Guard learns to produce actionable feed-back and three-way decisions that help agents re-vise potentially unsafe plans. Extensive experi-ments show that TRIAD + Tri-Guard substantially reduces unsafe agent behaviors while completing benign objectives. Future work can further explore the trade-off between benign-task success and at-tack suppression by scaling the training process and designing more calibrated guardrail objectives. 8Limitations 

Our work has several limitations. First, although TRIAD can be directly deployed as a plug-and-play guardrail framework for black-box agents, it intro-duces additional latency during inference. Even though we use a 9B model as the backbone of Tri-Guard, each safety check still requires extra guardrail inference. When the guardrail returns an update decision, its feedback is further injected back into the agent context, requiring the agent to generate a revised action plan before tool execu-tion. This feedback and revision loop inevitably increases the overall inference time. Future work should explore more lightweight guardrail frame-works, such as smaller guardrail models, more ef-ficient safety checking mechanisms, or lower cost plan revision strategies. Second, we believe the training data scale and backbone size of the guardrail can be further scaled up. Larger and more diverse trajectory feedback data may help Tri-Guard learn more stable decision boundaries across different tool use patterns and attack scenarios, while stronger backbone models may further improve its understanding of complex agent behavior. Finally, our evaluation focuses on the attack set-tings studied in this paper. Although these settings cover representative prompt injection and harmful task scenarios, real agent systems may face new at-tacks or more complex attack combinations. Future work should study how Tri-Guard can generalize to unseen attacks, either without additional fine tuning or with lightweight adaptation, while still preserving benign task utility. 

Ethics Statement 

This work studies safety risks in LLM-based agents and therefore involves datasets and evaluation set-tings that contain sensitive or potentially harmful content, including prompt injection attacks, mali-cious tool use, harmful task requests, and simulated privacy or security violations. These settings are used only for research purposes, with the goal of evaluating and improving agent safeguards. Fol-lowing the design of ASB and AgentHarm, our ex-periments are conducted in controlled benchmark environments with simulated tools and offline eval-uation, and no tool execution produces real-world effects. We do not use real user interaction logs or deploy the evaluated attacks against real systems. The interaction traces shown in the appendix are selected examples used to illustrate model behavior and guardrail intervention. We do not release raw interaction log files, and the presented examples are screened to avoid exposing unnecessary sensitive details beyond what is needed for scientific analy-sis. The intended use of this work is to support the development of safer LLM agent systems, rather than to facilitate misuse.

A.2 System Prompts ReAct Agent system prompt. Figure 4 shows the system prompt used for the ReAct-style agent. The prompt enforces a simple interaction proto-col: before each tool call, the agent must briefly explain why the tool is needed, and after issuing the tool call, it must wait for the corresponding tool result before continuing. We use the same agent prompt across ASB and AgentHarm, with its de-sign adapted from the system prompts provided by these benchmarks. This design avoids text-based tool-call parsing and aligns with the native function-calling interfaces used by current frontier models. 

Tri-Guard system prompt. Figure 5 shows the system prompt used for Tri-Guard. This format is designed to make the guardrail output directly usable by the downstream agent: safe actions can proceed, harmful tasks can be refused, and partially unsafe plans can be revised using the feedback. Without trajectory-level finetuning, however, base or prior guardrails often struggle to consistently fol-low this instruction format and tend to over-refuse partially unsafe cases rather than provide action-able revision guidance. Together, these prompts ensure that each agent step can be inspected before tool execution: the agent first proposes an action plan and a tool call, strictly following the ReAct workflow. 

A.3 Function Calling Format Outputs 

We clarify how we implement the ReAct agent framework in our experiments. Early ReAct imple-mentations usually use a prompt-based text format, where the model is instructed to produce reasoning traces and actions in an interleaved manner. In this format, the model writes an action plan or rationale in plain text, and then writes the action name and its arguments in a text format. This design was widely used before native function calling inter-faces became common, and prior work has shown that combining reasoning and acting can improve agent performance on interactive tasks. In our experiments, we implement ReAct agents with function calling format outputs instead of pure text outputs. Under this format, the model produces a visible planning context when it is available, and then emits a structured tool call, such as a JSON object that contains the function name and its ar-guments. We use the visible planning context as the plan pt and the structured tool call as the action 

at. Together, they form the planning stage action proposal (pt, a t). This proposal is generated before the tool is actually executed, so it provides a natural point for guardrail intervention. Given the planning stage proposal, our framework sends (pt, a t), the previous interaction history, and the available tool specifications to the guardrail. This implementation does not assume access to hidden model reasoning. We only use the visible planning context and the structured tool call that are available before tool execution. For consistency, all backbone agents in our experiments, including Qwen3-32B, Kimi K2.5, Gemini 2.5 Pro, and GPT-5.1, are implemented with the same function call-ing format. This gives us a unified representation of the planning stage output across different mod-els and allows the guardrail to evaluate proposed actions before any external tool is called. 

A.4 In-Context Learning Templates 

In Figure 6-8, we provide three ICL templates we used for the interaction between guardrail outputs and agent downstream generations. The U PDATE

C Details of Trajectory–Feedback Data Construction 

Task collection. To construct diverse training tra-jectories, we first collect and rewrite tasks from both agent-safety benchmarks and jailbreak bench-marks. First, we collect DH tasks from AgentAl-ign (Zhang et al., 2025b), which provides benign and harmful agent tasks together with reference agent responses. Since our goal is to construct ex-ecutable agent trajectories rather than reuse static responses, we extract only the harmful task de-scriptions and their associated toolkits, and gener-ate new agent rollouts in the trajectory generation stage. Second, we collect benign tasks, tool en-vironments, and prompt injection templates from InjecAgent (Zhan et al., 2024). We use these tem-plates in two ways: as task-level injections for DPI, and as tool-response injections for IPI. To create longer and more realistic multi-step agent interac-tions, we use GPT-5.4 to merge and rewrite tasks within the same environment (e.g., GoogleEmail, Amazon), converting original single-turn tasks into coherent multi-turn agent tasks. This expansion 13 increases trajectory length and creates more op-portunities for IPI attacks to be injected through intermediate tool observations. In addition, we collect harmful examples from JailbreakBench (Chao et al., 2024) and Harm-Bench (Mazeika et al., 2024) as additional seeds for direct harmful tasks. Since these benchmarks are not originally designed for tool-use agents, we use GPT-5.4 to rewrite their harmful prompts into agent-executable tasks and construct correspond-ing toolkits. After task collection and rewriting, we obtain a total of 5288 agent tasks. 

Trajectory Generation. After collecting the agent tasks, we generate real interaction trajectories by letting backbone agents execute these tasks in their corresponding tool environments. For prompt-injection settings, including DPI and IPI, we use GPT-4o as the backbone agent. Prior studies have shown that GPT-4o exhibits a moderate attack success rate under prompt-injection attacks (Zhan et al., 2024), making it suitable for generating both compromised trajectories that follow the injected instructions and benign trajectories that remain aligned with the original user goal. For direct harm-ful tasks from AgentAlign, we use Qwen3.5-35B-A3B-Uncensored 2 and Grok-4.2-Fast as backbone agents, since these models are more likely to com-ply with unsafe instructions and therefore help us collect harmful trajectories for the DH setting. After rollout, we categorize the generated trajec-tories according to the executed tool calls and their outcomes. A trajectory is considered harmful if the agent invokes tools or performs actions that fulfill the adversarial goal. For compromised DPI and IPI trajectories, the ground-truth decision is U PDATE ,because the original task still contains a legitimate benign user goal; the desired behavior is to revise the unsafe plan and realign the agent with the be-nign tool-use objective rather than refuse the entire task. For trajectories generated from direct harmful tasks, the ground-truth decision is R EFUSE , since the user goal itself is adversarial and should not be fulfilled. In addition, we collect benign trajecto-ries from AgentAlign benign tasks and from PIA rollouts where the agent resists the injected instruc-tions and remains aligned with the benign user task. These trajectories are assigned the P ROCEED de-cision, allowing the guardrail model to learn what legitimate action plans look like instead of only

Knowledge Distillation. After trajectory genera-tion, we construct the completion part of each SFT example, where the guardrail model is expected to produce structured NL feedback followed by a three-way decision. Each guardrail input consists of the user task, the interaction history contain-ing the generated trajectory, the current action plan, and the available toolkit. These inputs reflect the in-formation available to the guardrail during deploy-ment. The key challenge is to obtain high-quality outputs that teach the guardrail model not only to predict the correct decision, but also to explain the safety risks in a way that can guide downstream agent planning. To this end, we use GPT-5.4 as a teacher model to generate structured safety feedback and deci-sions. Following a knowledge distillation setup, the teacher model is prompted with the same guardrail system prompt as the student model, so that its outputs follow the desired feedback format and decision space. For each trajectory, the teacher produces a structured response consisting of five components. First, User Intent analyzes the user’s goal and whether the intent is benign, harmful, or mixed. Second, Current Action Plan examines the agent’s current plan from the interaction history and identifies potential unsafe tool use or risky be-havior. Third, Alignment Check compares the user intent with the current action plan, determining whether the plan remains aligned with the legiti-mate user goal or deviates toward an adversarial objective. Fourth, Security Check summarizes the identified safety risks based on the preceding anal-yses. Finally, the teacher outputs one of the three decisions: P ROCEED , U PDATE , or R EFUSE .Although GPT-5.4 provides high-quality safety analysis, its predicted decision may still disagree with the ground-truth decision assigned in the trajectory generation stage. Therefore, we per-form label-consistency filtering by comparing the teacher-generated decision with the trajectory-level ground-truth label. Teacher outputs whose deci-sions disagree with the ground-truth label are dis-carded. This filtering step ensures that the retained SFT examples contain both high-quality NL feed-back and decision labels aligned with our definition of P ROCEED , U PDATE , and R EFUSE . Rather than 14 directly deploying GPT-5.4 as a guardrail, which would introduce high inference cost and latency for step-level agent monitoring, we distill its struc-tured safety analysis into Tri-Guard. This allows Tri-Guard to learn the desired feedback format and three-way decision boundary while remaining more suitable for efficient guardrail deployment. 

SFT Data Pair Construction. After label-consistency filtering, we construct the final SFT training set by pairing each retained trajectory with its teacher-generated feedback and decision. Each training instance takes the guardrail input, includ-ing the user task, interaction history, current action plan, and toolkit specification, and uses the retained structured feedback plus the three-way decision as the completion target. In this way, the resulting dataset directly supervises Tri-Guard to analyze trajectory-level risks and produce actionable feed-back for downstream agent planning. In addition, for each retained teacher-generated response, we record a confidence score as meta-data. Specifically, the confidence score is computed as the teacher model’s mean per-token genera-tion probability over the feedback string under its own decoding distribution. This score serves as a teacher-side reliability estimate and is later used during guardrail training to reweight individual trajectory–feedback pairs. Overall, this pipeline yields high-quality trajectory–feedback pairs for training planning-stage guardrails. Since existing agent-safety bench-marks rarely provide trajectory-level NL feedback aligned with explicit guardrail decisions, our con-struction helps overcome the data scarcity problem for feedback-driven guardrail training. Moreover, the pipeline is scalable: new agent tasks, attack templates, and tool environments can be incorpo-rated to generate additional trajectories and teacher feedback following the same procedure.

E Understanding Evaluation Metrics 

We formulate each of the evaluation metrics used in ASB and AgentHarm below. 

Attack Success Rate (ASR) measures the percent-age of tasks where the agent successfully executes the attacker tools to fulfill the attacker goal. For ASB, we treat a test case as an attack success if the agent executes an attacker tool at any step of the trajectory. Once an attacker tool is invoked, the case is counted as successful even if the agent later refuses or retracts its behavior, since the unsafe tool-use decision has already occurred. Let N be the number of test cases, and let Natt be the num-ber of successfully attacked test cases. We compute ASR as: 

ASR = Natt 

N × 100% . (4) 

Task Success Rate (TSR) measures the percentage of attacked cases in which the agent still completes the original benign task under PIA. In ASB, each benign task typically requires the agent to invoke at least two normal tools in the correct order. We count a case as task success only when the agent fully completes the required benign workflow; par-tial completion, such as invoking only one required normal tool, is not counted. Let N be the number of test cases, and let Norig be the number of cases where the original benign task is fully completed. We compute TSR as: 

TSR = Norig 

N × 100% . (5) 

Refusal Rate measures the percentage of test cases in which the agent chooses to refuse the request. We count a case as refusal when the agent returns a refusal-style response instead of executing the requested task or continuing tool use. Let Nrefuse 

be the number of cases where the agent refuses. We compute Refusal Rate as: 

Refusal = Nrefuse 

N × 100% . (6) For AgentHarm, we adopt the Harm Score ,which evaluates the agent’s performance on harm-ful tasks by measuring how many harmful task requirements are successfully completed. Unlike a binary success indicator, it provides a fine-grained score that reflects partial completion of harmful objectives, where a higher Harm Score indicates greater harmful task completion (Andriushchenko et al., 2025). Similarly, we use the Benign Score to 16 measure the agent’s performance on benign tasks under the same fine-grained grading rubric. To summarize the helpfulness–safety trade-off on AgentHarm, we further define a Helpfulness– Safety score (HS) that jointly considers benign task utility and harmful task avoidance. Let H and 

B denote the normalized Harm Score and Benign Score, respectively, both scaled to [0 , 1] . HS takes the harmonic mean of B and (1 − H):

HS = 2 · B · (1 − H)

B + (1 − H) . (7) This formulation rewards agents only when they both preserve helpful behavior on benign tasks and avoid completing harmful tasks. If either benign task performance is low or harmful task completion is high, the overall HS score decreases substantially, capturing a common failure mode in prior frame-works.

F.2 Effect of Thinking on Guardrail Conservativeness 

We further examine the effect of enabling the think-ing mechanism in Qwen3.5-9B, the base model of Tri-Guard. As shown in Table 6, enabling think-ing makes the guardrail more conservative on Agen-tHarm. This conservativeness slightly improves refusal on harmful tasks, which is desirable, but it also substantially increases refusals on benign tasks, indicating a stronger false-alarm effect. In addition, thinking requires longer reasoning traces and consumes more tokens so that it further in-creases inference cost. Although thinking encour-ages more cautious safety judgments, it does not necessarily improve the overall helpfulness–safety trade-off and instead reduces HS in this setting. 

G Computational Overhead 

We report the computational overhead of TRIAD in Table 7. We use Qwen3-32B as the agent backbone and measure the inference latency at each ReAct step. After deploying TRIAD, the average per step latency increases to 

6.98 seconds. This includes the inference time of Tri-Guard and, when needed, the additional agent inference caused by guardrail feedback before the final action plan is decided. Therefore, TRIAD introduces an average overhead of 5.10 seconds per step, with the maximum overhead reaching 

24 .17 seconds. The largest overhead usually appears when the context is long and the number of guardrail at-tempts reaches the maximum limit. H.4 ToolSafe is not reported on GPT-5.1 

We leave the ToolSafe results on GPT-5.1 blank be-cause its implementation is incompatible with the tool-use protocol adopted by this backbone, rather than because it produces a meaningful failure un-der evaluation. Specifically, ToolSafe relies on a text-mode ReAct agent. As shown in Figure 12, the model is prompted to emit tool invocations as plain text using explicit Action: and Action Input: markers, and the harness dispatches a tool only when these markers can be recovered by their parser. However, GPT-5.1 is biased toward native function-calling behavior. Under this text-only pro-tocol, it often emits either bare JSON arguments without a function name or a NL final answer, both of which are ignored by the ToolSafe parser. Con-sequently, most proposed tool calls are never exe-cuted, the agent terminates after few rounds, and the resulting ASB or AgentHarm scores measure a broken execution pipeline. This limitation also il-lustrates a broader compatibility issue for guardrail designs that depend on fragile text-format action parsing, whereas our framework operates directly over native tool calls and can be applied to frontier agent backbones.

## 摘要
基于大语言模型的自动化渗透测试智能体通常针对静态目标进行评估，这些目标既不会检测也不会响应攻击，因此它们在智能防御下的行为仍然未经测试。多步攻击链的因果一致性同样依赖于不稳定的LLM推理，而智能体的决策对人类分析师来说仍不透明。这三个缺陷——真实性、一致性和可审计性——通常被孤立地修补。我们提出ZERO-APT，一个回合制的攻击者-防御者-裁判框架，在单一架构中解决了这些问题。为了实现真实性，ZERO-APT嵌入了一个可配置的LLM防御者，它消耗Sysmon遥测数据并实时检测攻击，使攻击者面对的是活对手而非被动目标。为了实现一致性，三种架构机制将因果一致性从不稳定的LLM推理转移到强制执行的系统架构中：规划与执行分离、多维ReAct反馈以及硬约束过滤的动作库。为了实现可审计性，专门的裁判智能体裁决每一轮，维护全局状态，并生成结构化的事后CTI报告，使每个决策都可追溯。我们在五种场景下评估了一个Windows Server 2022后渗透原型，采用三种防御者配置。ZERO-APT达到了79%的攻击成功率（Aurora 22%，PentestGPT 39%），因果一致性得分为0.860（Aurora 0.930，Claude Code 0.520），并通过结构化CTI报告实现了端到端的决策可审计性。我们发布该基准以支持智能防御下渗透智能体的评估。

## Abstract
LLM-driven automated penetration testing agents are typically evaluated against static targets that neither detect nor respond to attacks, so their behavior under intelligent defense remains untested. The causal consistency of multi-step attack chains likewise hinges on unstable LLM reasoning, and agent decisions remain opaque to human analysts. These three shortcomings, in realism, consistency, and auditability, are usually patched in isolation. We present ZERO-APT, a turn-based attacker-defender-judge framework that addresses them within a single architecture. For realism, ZERO-APT embeds a configurable LLM Defender that consumes Sysmon telemetry and detects attacks in real time, exposing the attacker to a live opponent rather than a passive target. For consistency, three architectural mechanisms move causal consistency from unstable LLM reasoning into enforced system architecture: separation of planning from execution, multi-dimensional ReAct feedback, and a hard-constraint-filtered action library. For auditability, a dedicated Judge agent adjudicates each round, maintains global state, and emits structured post-hoc CTI reports that make every decision traceable. We evaluate a Windows Server 2022 post-exploitation prototype across five scenarios with three Defender configurations. ZERO-APT reaches 79\% attack success rate (Aurora 22\%, PentestGPT 39\%), a Causal Consistency Score of 0.860 (Aurora 0.930, Claude Code 0.520), and end-to-end decision auditability through structured CTI reports. We release the benchmark to support evaluation of penetration agents under intelligent defense.

## 1. 核心问题与整体含义（研究动机和背景）

- **背景**：LLM驱动的自动化渗透测试智能体近年来快速发展，但现有评估均在**静态、无防御**的目标上进行——目标既不会检测也不会响应攻击。这与真实对抗环境严重脱节，其中每个动作都会留下可检测的痕迹，防御者会实时调整。
- **三个关键缺陷**：
  - **真实性（Realism）**：攻击者面对的是被动目标，而非活对手。
  - **因果一致性（Consistency）**：多步攻击链的每一步需由前置步骤产生，但LLM在长序列中容易出现幻觉、上下文遗忘和逻辑漂移，导致因果链断裂。
  - **可审计性（Auditability）**：LLM的内部推理链不透明，且近期研究表明其解释往往不忠实，甚至可能故意隐藏关键推理步骤，导致攻击成功/失败难以分析、复现和推广。
- **现有工作的不足**：上述三个问题通常被孤立地修补，但彼此相互纠缠：缺乏真实性则评估无对抗压力；缺乏一致性则加入防御只会扩大因果断裂；缺乏可审计性则无法理解成功攻击。ZERO-APT旨在**在一个闭环框架内同时解决三个问题**。

## 2. 方法论：核心思想、关键技术细节

### 核心思想

提出**回合制攻击者-防御者-裁判（Attacker-Defender-Judge）框架**，通过架构级机制而非依赖LLM推理质量来保障因果一致性，并通过可配置的LLM防御者提供真实的对抗压力，同时由专门的裁判智能体实现全流程可审计。

### 关键技术细节

#### 2.1 系统整体架构
- 每轮顺序：Attacker执行动作 → Defender检查日志并检测 → Judge裁决并更新全局状态。
- 全局状态包含环境信息，但每个智能体只读取其授权切片。

#### 2.2 防御者（Defender）
- **检测流水线**：
  1. **Monitor**：从ELK Stack拉取Sysmon事件（1分钟滑动窗口），格式化为结构化文本。
  2. **签名规则匹配**（L2/L3）：四条Sysmon签名规则（EID 1/3/11/12-14）。
  3. **LLM推理**：Monitor LLM输出入侵判断、MITRE ATT&CK技术ID、置信度（0-100）、证据摘要和攻击意图。
- **三级强度配置**（表2）：
  - **L1（低）**：无签名规则，日志含噪声，纯LLM推理。
  - **L2（中）**：四条签名规则 + 清洁日志 + LLM上下文推理（模拟企业SOC）。
  - **L3（高）**：L2基础上注入攻击者TTP情报和累计检测历史（模拟已知敌手）。
- 防御者仅读取系统日志，不访问攻击者内部计划，模拟真实防御中的外部可观测性。

#### 2.3 攻击者（Attacker）——三个架构保障因果一致性

1. **规划与执行分离（Planning-Execution Separation）**：
   - **Planner**：全局视图，输入任务目标、环境知识、裁判反馈，输出高层次计划和当前子目标。处理两种回溯：
     - 被动回溯：裁判反馈显示被检测/阻断。
     - 主动回溯：调度器报告当前子目标无可行动作。
   - **Dispatcher**（两阶段）：
     - 阶段1：从动作库检索候选动作，删除裁判禁止列表中的动作；若无剩余则触发回溯。
     - 阶段2：读取幸存动作的命令模板、参数定义和前置/后置条件，序列化动作并生成可执行命令序列。
   - **Executor**：接收微计划并逐步执行，内部ReAct循环带重试上限。使用WinRM远程执行命令。

2. **多维ReAct反馈**：
   - **智能体间反馈**（攻击者-裁判-攻击者闭环）：裁判裁决后，Planner接收裁判判决（含检测依据），在下轮调整攻击路径。
   - **智能体内反馈**（Executor自调试循环）：单命令失败时，Executor检查错误输出、调整参数并重试；达到重试上限则跳过该步骤。

3. **硬约束过滤的动作库（Hard-Constraint-Filtered Action Library）**：
   - 包含775个后渗透动作，源自Atomic Red Team，每个动作带有明确的前置/后置条件（表3）：
     - 硬条件（结构化字段）：权限级别、工具依赖、所需端口——由Dispatcher机械匹配，不调用PDDL规划器。
     - 软条件（文件路径、账户名等）：由Executor运行时探索并写入环境知识库。
   - 初始所有动作视为可用，探索发现不满足的条件后自动排除，动作空间逐渐收敛到可行集。

4. **回溯与自适应探索**（图2）：
   - 被动回溯（跨轮、Judge驱动）：裁判裁决失败后，Planner读取裁判观察并修订路径。
   - 主动回溯（同轮、Dispatcher驱动）：当Dispatcher发现无满足条件的动作时，回溯到Planner，重新定向子目标（每轮最多3次）。

#### 2.4 裁判（Judge）
- **每轮裁决**：接收攻击者的完整反馈和防御者的结构化检测输出，评估防御者证据质量（0-100），阈值75分：
  - ≥75：防御者胜，攻击结果不被认可，被阻断动作加入禁止列表。
  - <75：攻击者胜，攻击结果写入全局状态。
- **事后CTI报告**（Summary模块）：
  - 历史层：攻击概览、逐轮时间线、禁止动作列表演变。
  - 分析层：攻击者策略分析、防御者系统盲点、防御改进建议。
  - 攻击者推理链、防御者检测逻辑、裁判裁决依据、回溯触发原因均结构化记录。

## 3. 实验设计

### 3.1 基准与场景
- **自定义基准**：基于社区靶场（如Metasploitable 3），目标升级为Windows Server 2022，以匹配Atomic Red Team动作库。
- **五个后渗透场景**（表5）：
  - S1：凭证窃取（测试因果链执行）
  - S2：持久化+权限提升（测试多分支路径选择和硬约束过滤）
  - S3：横向移动（L3防御，测试自适应绕过）
  - S4：隐蔽数据外泄（四阶段长链，一次检测即任务失败）
  - S5：路径阻塞（L1防御，测试回溯恢复）
- 每个场景有默认防御级别（S1/S2/S4=L2, S3=L3, S5=L1）。

### 3.2 对比方法
- **Aurora** [6]：使用PDDL保证因果一致性，但生成静态脚本，无法响应防御反馈。
- **PentestGPT** [10]：人机协同模式，具有初步闭环交互但无硬约束过滤和结构化回溯。
- **Claude Code** [29]：通用LLM推理基线，无领域架构、裁判闭环或机械前置验证。

### 3.3 消融变体
- **-ActionLib**：移除动作库，LLM凭记忆生成命令。
- **-MultiReAct**：合并Planner+Dispatcher+Executor为单层ReAct，移除裁判闭环。

### 3.4 主要评价指标
- **ASR（攻击成功率）**：达到任务目标的比例。
- **CCS（因果一致性得分）**：两位安全专家独立审查完整攻击链，从10分扣分，归一化到[0,1]。ICC=0.84。
- **CBF%（因果断裂失败率）**：因前置条件不满足导致失败的命令比例。
- **BSR（回溯成功率）**：回溯后成功恢复子目标的比例。

## 6. 主要结论与发现

1. **ZERO-APT在ASR上显著优于基线**：
   - 总体ASR 79%，Aurora 22%（p<0.001），PentestGPT 39%（p<0.01），Claude Code 63%（非显著，但防御加强后差距扩大）。
   - 优势随防御强度增大而扩大（L1: 94% vs 94%持平；L2: 77% vs 63%；L3: 52% vs 37%），说明架构机制在防御最严峻时价值最大。

2. **CCS高但非最高，ASR高**：
   - CCS=0.860（Aurora 0.930，Claude Code 0.520）。Aurora虽CCS最高但ASR最低，表明仅因果一致性不足以对抗智能防御。
   - CBF%仅10.3%，远低于Claude Code 27.4%和PentestGPT 38.6%。

3. **消融实验证实架构贡献**：
   - 移除动作库（-ActionLib）：CCS下降40.7%，ASR降至48%。
   - 合并多层架构（-MultiReAct）：ASR降至42%，BSR下降71.6%。
   - 架构机制（特别是规划-执行分离和裁判闭环）是性能提升的关键，而非单纯依赖LLM能力。

4. **可审计性实现**：裁判生成的结构化CTI报告经专家评估达到可用标准（ICC=0.81），使得每个决策可追溯、可分析。

5. **跨环境初步验证**：纯攻击者在HackTheBox五台机器上表现与现有多智能体渗透工具相当（表8），说明增益来自架构而非LLM。

## 7. 优点

- **首次实现攻击-防御-裁判闭环**：将三个常被孤立处理的问题统一在一个框架内解决，真实模拟了红蓝对抗的迭代过程。
- **架构级因果一致性保障**：通过规划执行分离、硬约束过滤、多维ReAct反馈，将因果一致性从不可靠的LLM推理转移到强制系统架构中，不依赖LLM的长链推理稳定性。
- **可配置的三级防御**：提供L1/L2/L3三个强度等级，既可作为基准评估平台，也可作为研究攻防共演的工具。
- **全流程可审计**：裁判每轮裁决并输出结构化CTI报告（符合STIX 2.0），攻击者推理链、防御者检测逻辑、裁判依据均被记录，方便事后分析。
- **充分的统计验证**：1400次实验、多种统计检验、专家评分ICC验证，结果可信。
- **代码与基准开源**：推动社区在智能防御下评估渗透智能体。

## 8. 不足与局限

- **动作库覆盖有限**：仅包含Windows Server 2022后渗透的775个动作，未涵盖Linux、容器或初始入侵阶段。虽六元组抽象可跨平台扩展，但当前实现受限。
- **测试环境规模小**：仅两台VM（域控+数据服务器），未能模拟大型企业多子网、多角色的真实环境。最近有工作将LLM智能体应用于企业级Active Directory网络，但本文未涉及。
- **残留LLM依赖**：尽管架构约束了LLM，但Planner的推理和Judge的证据评分仍依赖LLM，错误无法完全消除。LLM的脆弱性（如推理不忠实、隐藏步骤）在极端情况下仍可能影响结果。
- **硬约束可能过度保守**：在L1（近零防御压力）下，ZERO-APT在S2上落后于Claude Code（18/20 vs 20/20），因为过滤器排除了一条正常的LOLBin路径。作者认为L2/L3净收益大于L1损失，但该设计在低防御环境中可能稍显保守。
- **防御者能力有限**：仅实现Sysmon检测，不含主动响应（如蜜罐、陷阱）。作者称“intelligent defense”是前瞻性术语，目前只捕捉自适应检测。未来可集成主动欺骗。
- **可审计性评估规模有限**：仅由两位专家评估50份CTI报告，数量偏少，可能未覆盖所有错误模式。
- **未来工作方向**：需要扩展到跨平台（Linux、容器）、大规模网络（企业AD）、以及防御者与攻击者的外部循环攻防共演化。

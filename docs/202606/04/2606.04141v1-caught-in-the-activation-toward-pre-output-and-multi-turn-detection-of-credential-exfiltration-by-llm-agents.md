## 摘要
LLM代理通常将敏感凭证与不可信检索内容置于同一上下文窗口中，这为间接提示注入诱导凭证泄露提供了直接路径。我们通过三种互补防御机制研究此失效模式。首先，探究激活探针是否能在输出令牌生成前检测凭证访问。其次，利用格式特定字符模型构建蜜令，并通过分割共形预测校准检测。第三，将多轮泄露视为累积信息流问题，并跨对话轮次追踪估计的泄露预算。在开放权重模型的受控实验中，激活特征能以高精度区分良性提示与凭证窃取提示，包括在保留的编码变换下。在小型合成多轮测试集中，累积核算能检测到单轮检测器遗漏的攻击。这些结果为初步性：多轮基准为内部小型数据集，激活方法需白盒访问，信息估计器提供实用信号而非形式上界。尽管如此，结果表明凭证泄露防御应结合预输出监控、校准金丝雀检测及时间泄露核算，而非仅依赖文本级输出过滤器。

## Abstract
LLM agents often place sensitive credentials in the same context window as untrusted retrieved content, creating a direct path for indirect prompt injection to induce credential exfiltration. We study this failure mode through three complementary defenses. First, we ask whether activation probes can detect credential access before output tokens are emitted. Second, we construct honeytokens from format-specific character models and calibrate detection with split conformal prediction. Third, we treat multi-turn exfiltration as a cumulative information-flow problem and track an estimated leakage budget across conversation turns. In controlled experiments on open-weight models, activation features separate benign and credential-seeking prompts with high accuracy, including under held-out encoding transformations. In a small synthetic multi-turn suite, cumulative accounting detects attacks that per-turn detectors miss. These results are preliminary: the multi-turn benchmark is in-house and small, the activation method requires white-box access, and the information estimator provides a practical signal rather than a formal upper bound. Still, the results suggest that credential-exfiltration defenses should combine pre-output monitoring, calibrated canary detection, and temporal leakage accounting rather than relying only on text-level output filters.

## 1. 核心问题与整体含义

**背景**：LLM代理（Agent）在实际部署中常将敏感凭证（API密钥、数据库密码、OAuth令牌等）与不可信的检索内容（网页、工具输出、外部文档）置于同一上下文窗口。这为间接提示注入攻击（IPI）提供了直接路径：攻击者通过嵌入恶意载荷诱导代理泄露凭证。

**问题**：现有防御多集中于输入过滤或输出文本级检测，但存在结构性局限：
- 输出检测可被编码、释义、Unicode替换或跨轮低速率泄露绕过。
- 输入检测面临泛化难题，提高召回率常导致误报增加。

**论文意义**：提出**Agentic Immune System (AIS)** 原型系统，探索三个互补层次的凭证泄露检测：内部模型激活（输出前检测）、植入侵蚀性蜜令（蜜令牌）、跨轮次累积泄露核算。目标是建立研究方向，暴露设计权衡，而非提供完整部署就绪的防御。

## 2. 方法论：核心技术细节

### 总体架构
AIS结合三个监控组件，按以下流程执行（见算法1）：
1. **DP-HONEY**：在查询前注入格式匹配的蜜令。
2. **CIFT**：在前向传播中提取隐藏状态特征，在输出生成前标志可疑凭证访问。
3. **文本级检测器**：扫描生成的输出中蜜令泄露。
4. **NIMBUS**：更新跨轮次的累积泄露分数。

### 2.1 CIFT：基于激活的凭证访问检测
- **动机**：文本级检测器面对编码输出会失效，而CIFT在输出渲染前检测模型内部是否正被导向凭证访问。
- **激活提取**：注册最后K个Transformer层的前向钩子（K = 0.25×总层数）。仅在**读取位置**（位于凭证和查询/载荷之后的令牌）上池化隐藏状态，以尊重因果注意力掩码。
- **特征**：每层马氏距离偏差（`CCI_ℓ`），再学习非负层权重形成因果流分数（CFS），最后用轻量MLP（9,473参数）映射到检测logit。训练目标为二元交叉熵加稀疏性惩罚。

### 2.2 DP-HONEY：校准的蜜令生成与检测
- **动机**：模板生成的蜜令可被攻击者通过前缀、校验和、熵或长度区分。
- **生成**：为每类凭证格式拟合字符二元语法模型，并添加拉普拉斯噪声（提供ε-差分隐私保证）。噪声保证任一个体凭证对生成器的影响受控，而非证明蜜令与真实凭证计算不可区分。
- **检测校准**：使用分裂共形预测校准器，在给定良性验证集和目标显著性α下自动确定非一致性阈值，避免手动调参。
- **蜜令会计**：若攻击者无法区分真实凭证与蜜令，从中均匀选择一个，则检测概率为 `k/(m+k) * (1-β) + α`。此模型为假设性简化，非安全定理。

### 2.3 NIMBUS：累积泄露评分
- **动机**：单轮低速率泄露（例如每次提取一个字符）可能绕过逐轮检测器。
- **估计器**：使用InfoNCE临界器估计每轮条件互信息下界 `ˆI_t`。状态输入`s_t`使临界器近似条件泄露而非边缘泄露。
- **累积追踪**：`ˆI_cum(t) = Σ_{τ≤t} ˆI_τ`，当累积分数接近预算B时触发警告、清理或阻断。注意InfoNCE理论上界为log₂(N_neg+1)（实验中N_neg=16，单轮不超过4.09比特），因此NIMBUS专为跨轮累积设计，大单轮泄露由CIFT和文本检测处理。

## 6. 主要结论与发现

1. **激活探针有效**：CIFT在保留的编码套件上依然保持高F1（图3），且AUROC达0.998，F1=0.990，FPR=0.015，远优于文本级基线。
2. **蜜令检测可校准**：共形校准消除了手动调参，覆盖率达0.998（目标0.99）。
3. **累积核算捕获多轮攻击**：NIMBUS检测了90%的低速率攻击，而逐轮检测器（如LlamaGuard）仅12%。
4. **集成系统提升**：全AIS在单轮基准上检测率达0.94，FPR=0.018，优于单独组件。
5. **短会话攻击仍是弱点**：NIMBUS未检测到的案例因攻击在早期完成，累积分数未超过预算阈值。

## 7. 优点

- **层次化防御设计**：覆盖输出前、输出文本、跨轮三个层面，互补而非冗余。
- **技术新颖性**：将激活探针用于凭证访问检测（不同于常见的安全/有害分类）；将差分隐私与共形预测结合用于蜜令生成与校准；引入InfoNCE作为临时泄漏分数。
- **明确的威胁模型**：采用Kerckhoffs原则，假设攻击者知晓防御策略但不知具体阈值/探针权重。

## 8. 不足与局限

### 实验覆盖不足
- **多轮基准极不充分**：仅50个合成对话，无第三方标准基准，无法评估在真实多轮交互中的表现。
- **蜜令区分性测试**：仅在一个小情景集上评估，未做大规模统计区分性测试（如对抗性生成器）。
- **跨模型迁移**：未测试CIFT探针在不同架构间的泛化能力（如从Qwen迁移到Llama）。
- **未处理结构化工具调用**：许多代理通过JSON/API参数传递凭证，论文明确将其列为结构盲点。

### 方法局限
- **CIFT需要白盒**：无法用于API托管模型（如GPT-4、Claude）。
- **NIMBUS是下界信号**：非认证上界，无法提供硬的泄露预算；InfoNCE上界4.09比特限制其独立检测大单轮泄露能力。
- **共形预测依赖交换性假设**：对抗性分布偏移下可能失效。
- **蜜令会计简化**：假设攻击者均匀选择凭证，但强大攻击者可利用功能线索避开蜜令。

### 资源透明度
- 未报告GPU型号、训练时间等，影响可复现性和效率评估。

### 偏差风险
- 作者设计了几乎所有组件和评估，可能存在无意识的设计偏见（如蜜令或NIMBUS的特异性）。

Abstract 

Modern agentic systems transform LLMs from session-bounded assistants into stateful sys-tems that persist and evolve shared world state across sessions through memories, filesystems, tools, and other long-lived contextual artifacts. This shift fundamentally expands the attack surface of prompt injection. However, prior works on prompt injection have largely fo-cused on model-level threats within a single session, overlooking how cross-session persis-tent system state fundamentally changes the system-level risk of agentic systems. Inspired by stored cross-site scripting in web systems, we introduce cross-session stored prompt injec-tion, where a successful injection can persist within agentic system state and silently influ-ence future executions long after the original attacker interaction has ended. To systemat-ically study this threat, we formalize stored prompt injection and develop a taxonomy of how adversarial content persists and affects agentic systems across sessions. We further develop a benchmark and sandbox toolkit to evaluate the risks of stored prompt injection, enabling quantitative analysis of attack suc-cess across different models, attack goals, and persistence channels. Our findings highlight that persistence transforms prompt injection from an ephemeral model-level threat into a long-lived system-level vulnerability embed-ded within agent execution state. We hope this work draws broader attention to this emerging threat and motivates the community to system-atically study and mitigate system risks arising from persistence in agentic systems 1.

In stored XSS attacks, malicious payloads are first written into persistent application state, such as databases or shared content, and later exe-cuted when rendered in downstream user ses-sions (OWASP Foundation, 2017). The key change is that injection and exploitation become tempo-rally decoupled: attackers no longer need to di-rectly interact with victims at exploitation time. Modern agentic systems are undergoing a similar transition(as shown in Figure 1). Once adversar-ial instructions are embedded into persistent agent state, such as memories, filesystems, tool-visible artifacts, or shared contextual resources, they may later be reincorporated into future execution con-texts across sessions, users, and tasks. In this paper, we identify and formalize this new class of vulnerabilities, which we term cross-session stored prompt injection (SPI) where ad-versarial instructions written during one interaction persist within agentic system state and later influ-ence downstream executions across future sessions, enabling persistent cross-session attacks. To sys-tematically study this threat, we formalize stored prompt injection and develop a taxonomy for under-standing how persistent adversarial context propa-gates and causes security harms in agentic systems. Based on this taxonomy, we build a benchmark and sandbox toolkit for evaluating SPI attacks across different attack goals, and persistence channels. Our analysis suggests that SPI attacks funda-mentally involve two coupled security problems in persistent agentic systems: unsafe persistent writes and malicious downstream reactivation. During the write stage, the agent may incorrectly allow attacker-controlled content to be written into long-lived persistent state, particularly strong persistence regions that are later directly incorporated into ex-ecution context, creating a toxic cross-boundary data flow from untrusted inputs into trusted system state. During later execution, the runtime context manager may further load contaminated persistent state into downstream execution contexts, allowing adversarial instructions to hijack agent behavior, manipulate responses, or influence tool-mediated actions across sessions. These findings suggest that the security risks of persistent agentic systems arise not only from malicious inputs themselves, but from how agent architectures govern persistent writes and later reincorporation of stored context into execution-time behavior. Our contributions are summarized as follows: • We identify a critical security gap emerging from the transition from stateless conversa-tional assistants to stateful agentic systems ,and formalize cross-session stored prompt injection (SPI) as a new class of foundational system-level vulnerabilities arising from per-sistent state. • Our position paper presents the first system-atic framework and taxonomy for under-standing SPI in agentic systems. By unifying previously fragmented attack surfaces, persis-tence channels, context incorporation mecha-nisms, and downstream harms into a coherent model, our framework establishes a common 2language for this field, lays the foundation for principled evaluation and mitigation, and opens a new perspective on agent security in the era of agent harness. • We operationalize the study of SPI through a benchmark and sandbox toolkit covering diverse attack goals and persistence channels. Our findings reveal that exploitability depends on how agent systems govern the full lifecycle of persistent context, from unsafe writes into long-lived state to later incorporation into ex-ecution context, highlighting secure context management as a first-class design princi-ple for agent harness engineering and a foun-dational building block for trustworthy next-generation agents. Ultimately, this work aims to establish cross-session stored prompt injection as a foundational security problem for agentic systems and to moti-vate broader research on securing persistent context and cross-session agent execution. 

2 Background and Motivation 

Prompt injection normally refers to a class of at-tacks in which adversarial instructions are embed-ded within content processed by an LLM and sub-sequently interpreted as executable directives, caus-ing the model to deviate from its intended objec-tives, policies, or user instructions(OWASP Foun-dation, 2023). Rooted in the fundamental difficulty of separating instructions from data in natural lan-guage, prompt injection has emerged as one of the most fundamental security threats to LLM-powered systems. As LLM applications have evolved from standalone chat assistants to increasingly agentic systems with access to external resources, tools, and persistent memory, the threat landscape of prompt injection has evolved accordingly .Early prompt injection research focused on di-rect prompt injection , where attacker-controlled user inputs attempt to override trusted system in-structions, bypass safety restrictions, or otherwise manipulate model behavior (Perez and Ribeiro, 2022; Liu et al., 2025). The threat model assumes a clear trust boundary between trusted developer-specified instructions and untrusted user inputs, framing prompt injection as an adversarial struggle over instruction authority. Security concerns there-fore centered on whether malicious users could subvert the behavioral constraints imposed by sys-tem prompts and alignment mechanisms. Under this formulation, the attacker is the user, the vul-nerable asset is the model’s instruction-following behavior, and exploitation occurs within a single interaction session. As retrieval-augmented generation (RAG) sys-tems (Lewis et al., 2020; Guu et al., 2020) became prevalent, researchers expanded this threat model through the introduction of indirect prompt injec-tion , where adversarial instructions are embedded within external resources such as web pages, docu-ments, emails, tool outputs, or retrieved knowledge sources that are later incorporated into an LLM’s context (Greshake et al., 2023). This development fundamentally altered the security assumptions un-derlying prompt injection. Rather than originating from a malicious user, the attack now arises from untrusted third-party content encountered during the agent’s interaction with its environment. Con-sequently, the user may be entirely benign while the attacker operates through external information channels that are subsequently consumed by the system. Instead, it concerns the safe integration of external content into the model’s execution context. Prompt injection therefore evolved from a secu-rity problem at the user-model interface to one that arises from the integration of untrusted external information into LLM workflows. In parallel, the emergence of tool-augmented agents (Zhang et al., 2025a; Patil et al., 2025; Yao et al., 2025; Huang et al., 2025) further broadened the prompt injection threat model. Whereas indi-rect prompt injection primarily concerned the in-corporation of untrusted information into model context, modern agents increasingly depend on external tools, MCP services, and reusable skills. These components introduce new trust boundaries, as tool descriptions, capability metadata, execution traces, and service-provided content may all be-come part of an agent’s reasoning context. Recent work has demonstrated that adversarial instructions embedded within such components can influence planning, manipulate tool-selection decisions, and induce unintended actions (Wang et al., 2026; Guo et al., 2025; Jia et al., 2026). As a result, prompt injection is no longer limited to influencing what an agent knows, but increasingly affects how an agent decides and acts. To study these risks, several benchmarks and evaluation frameworks, including AgentDojo (Debenedetti et al., 2024), Agent Safety Bench (Zhang et al., 2024) and Agent Security Bench (Zhang et al., 2025b), have been proposed to evaluate the security of tool-using agents. Ex-3isting prompt injection research shares a common underlying assumption: adversarial content is in-troduced and exploited within the same execu-tion context . Whether the attack originates from a malicious user query, a retrieved document, a tool response, an MCP service, or an agent skill, the injected content is observed by the model and influences behavior during the same interaction in which it is introduced. Consequently, existing at-tacks are fundamentally session-bound , relying on immediate context incorporation and immediate exploitation.A recent exception is Memory Injec-tion (Dong et al., 2026), which studies how adver-sarial content can be planted into an agent’s mem-ory through query-only interactions and later affect behavior. However, its scope remains centered on memory as the persistence target and query-only interaction as the injection channel; it also cov-ers a limited set of downstream harms, leaving broader persistent channels, incorporation mecha-nisms, attack consequences, and cross-system state contamination largely unexplored. To the best of our knowledge, no existing benchmark or evalu-ation framework systematically studies persis-tent cross-session security risks across diverse injection sources, persistence channels, and con-text incorporation mechanisms .These observations motivate a broader recon-sideration of prompt injection in the context of agentic systems. From a system-level perspective, the spatial scope of prompt injection extends be-yond individual injection sources to encompass the broader set of system states that may be influ-enced by adversarial content. Existing research primarily defines prompt injection according to how malicious instructions enter an agent’s con-text, whether through user inputs, retrieved docu-ments, tool outputs, MCP services, or agent skills. However, in agentic systems, adversarial content does not necessarily remain confined to its original source. Through write operations, it can propagate 

into memories, user profiles, shared workspaces, application state, tool-generated artifacts, and other forms of persistent contextual state. This obser-vation suggests that prompt injection should be understood not only as a problem of adversarial instruction sources, but also as a problem of state contamination within agentic systems . Under this view, malicious inputs may exert influence in-directly by poisoning system-visible state, even when they do not immediately alter agent behavior at the time of injection. Existing prompt injection research also implic-itly assumes a strong temporal coupling between injection and exploitation. Whether the attack orig-inates from a malicious user query, a retrieved doc-ument, a tool response, an MCP service, or an agent skill, adversarial instructions are typically introduced, incorporated, and activated within the same execution context. Consequently, prompt in-jection has largely been studied as an immediate and transient threat, where the effects of malicious content are confined to the interaction in which it is first observed. However, this assumption no longer holds in increasingly stateful agentic sys-tems. The growing adoption of long-term memory, shared workspaces, persistent user profiles, MCP-managed resources, and other forms of durable contextual state enables information to outlive the interaction in which it was originally introduced. As a result, adversarial instructions may remain embedded within system state long after the initial injection event and only influence agent behavior when that state is subsequently reincorporated into a future execution context. This observation re-veals a fundamentally different class of prompt injection risks in which the injection event and the exploitation event become temporally decoupled. An attacker may introduce malicious content dur-ing one interaction, while the resulting behavior is triggered in a different session, by a different user, or under a different task context. Viewed through this lens, SPI expands the con-ceptual scope of prompt injection along both spa-tial and temporal dimensions. Spatially, it shifts attention from individual injection sources to the broader system states that adversarial content can propagate and persistent through agent actions. Temporally, it breaks the assumption that injec-tion and exploitation occur within the same inter-action, allowing adversarial instructions to persist and activate across sessions. Together, these exten-sions transform prompt injection from a transient interaction-level threat into a system-level security problem centered on state contamination, persis-tence, and context construction.

This distinction is critical for understanding how adversarial content propagates and is activated in agentic systems, yet it has been 5largely overlooked in prior work. Content intro-duced in one interaction may persistently influence future agent behavior by modifying persistent con-text that is incorporated into the agent’s context across later interactions. Therefore, the security boundary lies not only in the model M, but in the context construction function A and the persistence mechanisms governing these context channels. 

Definition #1 Persistent Context. Persistent context refers to any information maintained by the agentic system that (i) persists beyond a single interaction and (ii) may be incorporated into the agent’s context through the context construction function A in future executions. This definition excludes data that is stored but never incorporated into model context, as such data cannot influence model behavior. 

3.2 Threat Model 

We assume the target agentic system is stateful: ex-ternal inputs may update memory, retrieval corpora, tool-visible state, or file-backed context, and these persistent context channels may later be incorpo-rated into the agent’s context. 

Adversary’s Capability. We consider an adver-sary who interacts with the agentic system through inputs that cross the system boundary from an un-trusted region into the agent’s context construction pipeline. We model this capability as a write primitive from the untrusted region into one or more persis-tent context channels: w : u 7 → p ∈ P , where u

denotes adversary-controlled external content and 

P denotes the persistent context reachable by A.The write primitive may be realized through dif-ferent mechanisms, including query-induced mem-ory updates, poisoning of retrieval sources, tool or MCP installation, exposure of malicious external content to browsing or API tools, or modification of file-backed agent state. 

Adversary’s Goal. The adversary’s goal is not merely to hijack a single model invocation, but to establish a persistent influence over future agent behavior. To this end, the adversary aims to cause malicious content p to be written into, or to modify, persistent context that is later incorporated into the agent’s context by the context construction function 

A.

Adversary’s Constraints. The adversary does not require privileged access to the model M, the context construction function A, or the victim’s query. In particular, the adversary cannot directly rewrite the constructed agent’s context, change the agent’s orchestration logic, or force A to include arbitrary content at exploitation time. Instead, the adversary can only influence the persistent state that A may later load or select during normal agent operation. 

Definition #2 Stored Prompt Injection. Astored prompt injection attack occurs when adver-sarial content p is written into or modifies persistent context P, and in a later execution is incorporated into the agent’s context xt = A(qt, ·, P), where it is interpreted by the model M in a way that alters the agent’s behavior beyond the intended task. This definition separates the injection stage from the activation stage. During injection, the adver-sary causes malicious content to cross the system boundary and persist within an agent-controlled context channel. During activation, the adversary need not be present; the attack is triggered when normal context construction incorporates the stored content into a future agent’s context. Thus, stored prompt injection exploits persistent state mutation rather than only transient prompt manipulation. 

4 Taxonomy of Stored Prompt Injection 

Stored prompt injection differs from traditional prompt injection in that the adversary aims to estab-lish persistent influence over agent behavior, rather than merely affecting a single agent action. Thus, stored prompt injection is fundamentally a stateful 

attack which targets the agent’s persistent context. Importantly, the impact of such attacks may not manifest within the interaction in which the adver-sarial content is introduced, but instead emerge in subsequent executions when the stored content is later incorporated. To reason about this class of attacks, it is therefore insufficient to consider only the immediate behavior of the agent within one session. Instead, we must account for how adver-sarial content propagates through the agentic system: how it enters, where it persists, and un-der what conditions it is later incorporated into the agent’s context .To answer these questions and capture the at-tack lifecycle, we organize stored prompt injection along three dimensions: (1) injection source , which characterizes how adversarial content crosses the system boundary; (2) persistent context channel ,which characterizes where such content is stored within the system; and (3) context incorporation mechanism , which characterizes how stored con-6tent is later incorporated into the agent’s context. Together, these dimensions characterize the lifecy-cle of stored prompt injection from entry, to persis-tence, and finally activation. Together, it provides a structured view of stored prompt injection as a cross-session attack on the persistence mechanism and context construction pipeline of agentic sys-tems. 

4.1 Injection Source 

The first dimension characterizes where adversar-ial content first crosses the system boundary and becomes available to the agentic system. These en-try points align with prior formulations of prompt injection, encompassing both direct prompt injec-tion (via user-supplied inputs) and indirect prompt injection (via externally consumed content and supply-chain installable tools’ context). However, in the context of stored prompt injection, the role of these sources extends beyond immediate prompt manipulation. Instead, they serve as write channels through which adversarial content may be intro-duced into the agent’s persistent context. This en-ables the attacker to establish influence that persists beyond the initial interaction, rather than being con-fined to a single execution. 

User-Supplied Content. Adversarial content may be supplied directly through the user-facing interface of the agent, including direct user queries and uploaded files. This corresponds to direct prompt injection, where the attacker explicitly con-trols the input to the system. In the context of stored prompt injection, such inputs may be writ-ten into persistent context (e.g., through memory updates or file modifications), enabling the attacker to influence future executions beyond the initial interaction. 

Externally Consumed Content. Adversarial content may also be encountered indirectly when the agent consumes external information during task execution, such as third-party application data, web content, or RPC/API results. This corresponds to indirect prompt injection, where the attacker does not directly control the user input but instead influences the content observed by the agent. When such content is ingested and persisted, it may later be incorporated into the agent’s context, allowing adversarial influence to propagate across execu-tions. 

Supply-Chain Installable Tool. Adversarial content may be embedded in installable tools, such as structured tool definitions, MCP servers, or skills, which can be introduced into the agent through user-initiated installation. This setting can be viewed as a form of supply-chain-based indirect prompt injection. Adversarial content is carried into the system as part of the tool itself and may subsequently persist within the agent’s operational context. 

4.2 Persistent Context Channel 

The second dimension captures where injected con-tent persists after entering the system. This dimen-sion is central to stored prompt injection: an attack becomes stored when adversarial content is written into, or modifies, context that persists across ses-sions and can be reintroduced in future interactions. We consider several classes of persistent context channels that are commonly present in agentic sys-tems. 

Agent Memory. Adversarial content may per-sist within the agent’s memory system. We dis-tinguish between working memory , which is auto-matically incorporated into the agent’s execution context, and archival memory , which is incorpo-rated only when retrieved. Consequently, stored adversarial content may either exert recurrent in-fluence across sessions or remain dormant until reactivated by future retrieval. 

Tool-Visible State. Adversarial content may persist in tool-integrated context, including tool de-scription context and tool response results. Tool description context is often strongly persistent, as it is loaded as part of the agent’s tool interface (e.g., structured tool definitions or MCP descriptions), though some forms (e.g., skills) may be condition-ally included. In contrast, tool response results are typically conditionally persistent and are only incor-porated when the agent executes the corresponding tools. More broadly, tool-visible state also includes the resources accessible to tools. While tools are part of the agentic system, they often interact with underlying resources, which may be either internal (e.g., local files) or externally writable (e.g., emails, shared documents, or application state). Adversar-ial content written into such resources can persist and later be reintroduced into the agent’s context when accessed through tool execution. 

File-Backed Context. Adversarial content may also persist in file-backed context. This includes both convention-based instruction files and general workspace artifacts that the agent may later read or load into its working context. In modern agentic systems, certain files (e.g., AGENTS.md , SOUL.md ,7USER.md , HEARTBEAT.md ) are treated as structured or protocol-level context and may be automatically or preferentially incorporated into the agent’s con-text, resulting in strong persistence. In contrast, general workspace files (e.g., code repositories or task artifacts) are typically conditionally incorpo-rated, depending on the agent’s execution path, such as file access, summarization, or indexing. As a result, file-backed context can serve as a persis-tence channel where adversarial content is stored in files and later reintroduced into the agent’s context through agent operations over the workspace. 

4.3 Context Incorporation Mechanism 

The third dimension characterizes how persistent context is later incorporated into the agent’s con-text. Persistence alone is insufficient for exploita-tion: the stored content must be reintroduced into the agent’s context in a way that can affect behavior. We identify two fundamental incorporation mecha-nisms, which differ in whether the agent’s context construction function unconditionally includes the stored content or requires a conditional decision. 

Direct Loading. Persistent context may be in-corporated through direct loading, where content is unconditionally included in the agent’s context ac-cording to predefined conventions or system-level policies. This includes system prompts, agent-level instruction files (e.g., AGENTS.md ), working mem-ory entries and tool description context, which are typically loaded by default whenever a session or workspace is initialized. As a result, content in this category exhibits strong persistence: once injected, it is reliably present in subsequent contexts without requiring any specific trigger. This makes direct loading the most stable and high-impact attack surface for stored prompt injection. 

Conditional Loading. Persistent context may also be incorporated through conditional mecha-nisms, where content is included only when the agent actively retrieves or accesses it. This includes archival memory that must be explicitly queried via retrieval tools, workspace files that could be read by the agent during task execution, and applica-tion data accessible through tool invocation. In all of these cases, incorporation is contingent on the agent’s decision to perform the retrieval or access operation, as well as on the relevance of the stored content to the current task. Adversarial content remains dormant unless the agent takes the specific action that brings it into context. 

4.4 Attack Harm Categories 

Beyond characterizing how adversarial content en-ters and propagates through the system, it is equally important to understand what harms it can cause once activated. We classify the potential harm of stored prompt injection into three categories, ac-cording to which aspect of the agent’s decision-making is compromised. 

Fact Manipulation. The agent is led to state incorrect but plausible facts that contradict es-tablished ground truth. This targets question-answering tasks with clear right-or-wrong answers. Once the attack succeeds, the agent may confi-dently present false information as correct—for instance, giving an incorrect policy or rule—which can mislead users and potentially lead to legal lia-bility. 

Preference Manipulation. The agent is led to consistently favor an attacker-chosen option when making decisions on behalf of the user. This tar-gets constrained choice tasks, where the agent is expected to select options that satisfy explicit user requirements. Once the attack succeeds, the agent may systematically steer choices in a biased direc-tion—even when it violates given constraints (e.g., selecting higher-cost options despite a stated bud-get). Such behavior can be abused to manipulate product exposure or ranking, i.e., generative en-gine optimization (GEO), quietly biasing outcomes while still appearing reasonable on the surface. 

Action Scope Manipulation. The agent is led to perform actions that go beyond or deviate from what the user explicitly requested. This tar-gets tasks that involve tool use or external actions, where the agent is expected to follow a well-defined scope. Once the attack succeeds, the agent may execute unintended operations or omit required ones, effectively altering the action plan without the user’s awareness. Such manipulation can be exploited to trigger high-risk behaviors, including leaking sensitive information, executing malicious code, or interacting with unintended external sys-tems.

5.1 Experimental Setup 

To empirically evaluate the attack surface charac-terized by the taxonomy, we manually design SPI-Benchmark , which instantiates each dimension as a controlled experimental variable and measures at-tack success at each stage of the SPI lifecycle. The benchmark is structured across 3 application sce-narios (e-commerce, travel booking, and financial portfolio management), with a total of 162 unique SPI cases. Each case is meticulously crafted by domain experts, complete with predefined ground-truth facts or behaviors, specific attack claims, and designated victim tasks. To ensure rigorous evalua-tion, we implement a hybrid verification pipeline combining a rule-based verifier and an LLM-as-a-Judge. In instances where a discrepancy arises between the two automated judges, a human-in-the-loop auditing process is introduced to resolve conflicts. 

Data Details. The SPI cases systematically vary three factors derived from the taxonomy: (1)Attack risk category: fact manipulation, preference ma-nipulation, and action scope manipulation; (2)Per-sistent channel: working memory (direct loading), archival memory (conditional loading), file-backed context (e.g. AGENTS.md ); (3)Attack type: naive in-jection versus contextual disguise, comparing blunt commands against payloads framed as legitimate business directives. 

Evaluation Metrics. Following the SPI lifecy-cle, we decompose the attack process into three se-quential stages: write, incorporation, and activation. These stages span two sessions: an injection ses-sion , where adversarial content may be written into persistent storage, and a subsequent activation ses-sion , where the stored content may be retrieved and acted upon. Each stage corresponds to a necessary condition for a successful stored prompt injection attack. We define stage-wise success rates based on cross-session execution traces. Evaluation pipeline. The benchmark evaluates each attack case through a four-phase pipeline (Fig-ure 3): (1) Environment Setup : a fresh sandboxed environment is created and populated with scenario-specific data along with the corresponding tool that operate on this data; (2) Injection : an attacker query is submitted to the agent, which may cause the agent to write adversarial content into the tar-get persistent channel; (3) Session Reset : a new agent session is initiated with an empty context window, the previous conversation history is dis-carded, while the sandboxed environment, includ-ing all persistent stores and files, remains intact; (4) 

Activation : a clean victim query is submitted in the new session, and the agent’s response and tool calls are recorded to determine whether the previously injected content influenced its behavior. The session reset is central to the benchmark design: by discarding the conversation history and starting a fresh session while preserving the envi-ronment, we ensure that any observed influence on the agent’s behavior in the activation phase can only originate from persistent context, not from residual multi-turn conversation history. Meanwhile, the sandbox provides an isolated environment to en-sure that any agent with real system privileges is not interfered with by other irrelevant factors.

5.7 Discussion 

Our experiments yield several key insights for un-derstanding SPI attacks: 

Strongly persistent, directly-loaded context is the primary risk surface. Channels that are automatically incorporated into the agent’s context consistently lead to higher end-to-end success. This suggests that the core risk in SPI is not merely whether content can be stored, but whether it is 

unconditionally reintroduced into future contexts, turning persistence into a reliable attack vector. 

Injection and activation are largely decou-pled. WSR and AR show weak correlation across models and settings: improving write success does not necessarily translate to higher attack execution. Contextual disguise further reinforces this by sub-stantially increasing WSR without improving (and sometimes degrading) AR. This indicates that SPI 11 attacks are fundamentally a multi-stage process, where injection (write) and activation (execution) behave as largely independent bottlenecks. 

Attack effectiveness varies systematically by goal. Fact manipulation is consistently the easiest attack because it aligns with the model’s default behavior of absorbing and prioritizing newly pro-vided context as factual grounding . Once injected content is incorporated, the model tends to treat it as part of the current world state, requiring no change to its decision policy. In contrast, prefer-ence manipulation is substantially harder because it attempts to override the model’s existing decision rules (e.g., priorities, constraints). 

6 Conclusion 

Modern agentic systems transform LLMs from session-bounded assistants into stateful systems that persist and evolve shared world state across sessions through memories, filesystems, tools, and other long-lived contextual artifacts. This evolution fundamentally changes the security assumptions underlying prompt injection. Prior work has pri-marily studied prompt injection only as a transient model-level threat. Inspired by stored cross-site scripting (XSS) in web system, we define cross-session stored prompt injection as a new and fun-damental system-level attack vector in agentic sys-tems. To systematically study this threat, we for-malized the SPI’s lifecycle, developed a taxonomy spanning write, persistence, incorporation, and ac-tivation stages, and built a benchmark and sandbox framework for evaluation across models, attack goals, and persistence channels. Our results demonstrate that adversarial content embedded in persistent context can be reliably re-activated in downstream sessions, leading to cross-session effects without direct attacker involvement at exploitation time. It suggests that the primary security challenge in agentic systems no longer lies solely in filtering malicious inputs, but in how persistent system state is managed, incorporated, and executed across sessions. In particular, context incorporation mechanisms emerge as a critical se-curity boundary, since persistent artifacts become dangerous only when they are later reincorporated into execution context. More broadly, our work highlights a fundamental shift in agent security: as agentic systems increasingly rely on persistent and shared state, memories, filesystems, and tool-generated artifacts themselves become part of the executable attack surface. We hope this work moti-vates future research on defenses, isolation mecha-nisms, and secure architectural principles for per-sistent agentic systems. 

Limitations 

We develop our taxonomy based on current agen-tic system designs, but note that the ecosystem is rapidly evolving, with new mechanisms for context persistence and interaction continually emerging. These developments may further expand the land-scape of stored prompt injection risks. We view our work as a step toward systematic, system-level threat modeling in this setting, and highlight adap-tive approaches for evolving agentic systems as an important direction for future work. 

Ethics Statement 

This work is the first to identify and formalize stored prompt injection as an attack vector enabled by persistent context in agentic systems. We show that persistence and reuse of context, including across sessions, extend the threat model beyond traditional prompt injection, and introduce a bench-mark and sandbox toolkit to support evaluation. Our work clarifies these risks without introducing new attack primitives, and aims to inform more secure agentic system design.

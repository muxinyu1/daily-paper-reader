## Abstract 

Consensus protocols form the backbone of dis-tributed systems and blockchains, where imple-mentation bugs can cause data corruption and financial losses. While LLM-based approaches show promise in code analysis, they struggle with deep protocol-level logic bugs involving complex state-dependent behaviors across multiple execu-tion stages. We present Agora, a domain-aware multi-agent framework that integrates hypothesis-driven testing with LLM capabilities for system-atic protocol verification. Agora employs special-ized agents that collaboratively explore protocol state spaces, synthesize attack scenarios using domain-specific constraints, and validate findings through iterative refinement. This explicit role separation enables reasoning about global proto-col invariants beyond single-function code analy-sis. We evaluate Agora on four consensus imple-mentations (Raft, EPaxos, HotStuff, BullShark) using four state-of-the-art LLMs. Agora discovers 15 previously unknown protocol-level logic bugs that violate safety properties, while existing LLM-based agents fail to detect any such protocol-level logic bugs. Our results demonstrate that domain-aware multi-agent collaboration is essential for detecting deep logic bugs in complex protocols.

2.2. Consensus Protocols 

Consensus protocols mainly fall into two categories: Crash Fault-Tolerant (CFT) protocols, such as Paxos (Lamport, 1998; 2001), Fast Paxos (Lamport, 2006), EPaxos (Moraru et al., 2013), and Raft (Ongaro & Ousterhout, 2014); and Byzantine Fault-Tolerant (BFT) protocols, such as PBFT (Castro, 1999), HotStuff (Yin et al., 2018), DAG-Rider (Keidar et al., 2021), Narwhal–Tusk (Danezis et al., 2021), and BullShark (Giridharan et al., 2022). CFT consen-sus protocols are primarily used in traditional banking sys-tems and Internet companies for data transactions and clus-ter coordination, whereas BFT consensus protocols form the foundation of blockchain systems and on-chain transac-tions. These protocols are widely used in both research and production settings, with code repositories implemented in multiple programming languages, such as Go and Rust. The distinction is particularly important because BFT systems explicitly tolerate malicious behavior from participating nodes, whereas CFT systems do not; as a result, analyz-ing these two classes of protocols involves fundamentally different Constraints .To analyze bugs in consensus protocols, it is essential to sum-marize bug patterns previously identified through extensive human expertise, in order to more effectively uncover bugs that violate system safety and correctness and may lead to erroneous transactions and data corruption (i.e., protocol-level Logic Bugs ), rather than low-level Implementation Bugs such as out-of-memory errors, memory leaks, integer overflows, or simple logical flaws. The distributed protocols are primarily required to satisfy two fundamental properties: liveness and safety. Only the causes that lead to violations of these two properties constitute true protocol-level logic bugs. Such causes can be broadly categorized into five classes: (i) Recov-ery&Execution Divergence, (ii) Persistence&Monotonicity Violations, (iii) Dependency&Topology Flaws, (iv) Message Binding&Signature Violations, and (v) Re-source&Operational Visibility Violations.

Early dual-agent role-playing systems, such as CAMEL (Li et al., 2023) and ProAgent (Zhang et al., 2023), demonstrate that collaboration among agents can elicit step-by-step rea-soning. More recent frameworks extend this idea by assign-ing explicit roles to multiple agents: MetaGPT (Hong et al., 2023) and ChatDev (Qian et al., 2023) employ manager, designer, and coder agents to simulate software engineer-ing workflows; Magnetic-One (Fourney et al., 2024) and AG2 (Wang et al., 2024a) introduce a central coordinator that assigns tasks to different workers. However, existing multi-agent systems still lack the ability to reason deeply about code patterns, especially to integrate domain knowledge of consensus protocol systems for efficiently discovering deep bugs. 

2.4. LLM-powered Bug Detection 

Recent advances in LLMs have significantly influenced au-tomated bug detection and program analysis. Prior work has demonstrated that LLMs can assist developers in tasks such as bug localization, program repair, test generation, and vulnerability detection by leveraging their strong code under-standing and natural language reasoning capabilities (Feng, 2020; Chen et al., 2021; Pearce et al., 2022). LLMSAN (Wang et al., 2024c), SCALE (Wen et al., 2024), LLMDFA (Wang et al., 2024b) and PrimeVul (Ding et al., 2025) all leverage LLMs to assist in bug detection. KNighter (Yang et al., 2025a) is the first to employ an agent-based system for analyzing bugs in LLM system code. How-ever, these existing LLM-powered bug detection techniques largely focus on localized code patterns ( implementation bugs ) and lack the ability to reason about complex, state-dependent behaviors that span multiple components or ex-ecution stages ( logic bugs ). Moreover, these approaches do not leverage multi-agent systems to combine with HDT paradigm to achieve more comprehensive bug discovery. 

## 3. Method 

3.1. Motivation and Design Goal 

We introduce Agora, which integrates domain knowledge of distributed consensus with an LLM-based multi-agent system to address the fundamental challenge of compre-hensively detecting protocol-level logic bugs through HDT. Beyond this core capability, Agora is designed with several desirable properties. It is effective , capable of uncovering deep logic bugs; it provides interpretability , producing detailed analysis reports for discovered bugs to facilitate debugging; it is efficient , requiring less communication and fewer tokens and yielding a low false-positive rate; it is 

complete , applicable to both CFT and BFT systems; and it is flexible , featuring a modular, decoupled design that supports plug-and-play extension to other domains. Moreover, our method should prioritize using computational resources to discover important logic bugs , rather than wast-ing tokens on implementation bugs .

3.2. Architecture 

As shown in Figure 1, Agora is comprised of three core agents: (i) Orchestrator agent is responsible for workflow control and consists of two components. The 

bug-exploitation component leverages bugs pre-viously discovered by Agora to more effectively trig-ger new bugs, while the state-analyzer component maintains the states of the other two agents and pre-vents Agora from repeatedly exploring identical bug sce-narios; (ii) Strategy agent contains two components: a 

constraints-analyzer that analyzes the constraints of bug attack scenarios and supports both CFT and BFT pro-tocols ( complete ), and an attack scenario-generator 

that produces protocol-specific scenarios to facilitate the dis-covery of deep logic bugs; (iii) TestGen agent employs a 

reflection-loop component to automatically gener-ate, execute, and analyze unit tests, reducing false positives and synthesizing user-friendly bug reports based on the gen-erated attack scenarios ( interpretability ). 3.3. Knowledge Library 

For bug patterns , we crawl GitHub repositories of con-sensus protocols to collect issues that have been confirmed as bugs and summarize them accordingly, filtering out low-level implementation bugs. In this way, we incorporate domain-specific knowledge of consensus protocols to help Agora understand protocol-level logic bugs and generate more effective attack scenarios. For constraints , we summarize the different settings and design choices of con-sensus protocols and, combined with the analysis of bug patterns, derive the distinct conditions under which bugs are triggered in CFT and BFT systems. For example, assuming malicious behavior of nodes in a CFT system can easily violate safety properties; however, such assumptions are impractical in real deployments and would unnecessarily waste computational resources. By combining bug patterns and constraints, we introduce a succinct knowledge library for Agora that only provides the necessary domain knowledge for Agora. The details of bug patterns are presented in Appendix C and the details of constraints are in Appendix D. 

3.4. Memory 

In our system, the memory maintained by Agora consists only of repository knowledge and bug patterns. Bug pat-terns are described in detail in §3.3. Repository knowledge typically includes the repository name, the type of consen-sus protocol, protocol settings, implementation language, and testing methodologies. This information enables Agora to efficiently support consensus protocols implemented in different programming languages and with diverse designs. Both repository knowledge and bug patterns are shared across the agents of Agora for analysis and are lightweight and easy to maintain (the details are in Appendix E). There-fore, we store them in the system memory and maintain them within the execution environment of Agora LLM-based multi-agent system. 

3.5. Workflow Details 

Before reaching the termination condition or exhausting the budget, as shown in Figure 2, the workflow of Agora consists of 12 steps (subtasks). In Step 1 , Orchestrator performs its subtask O.1 and con-ducts bug exploitation using prompts. By analyzing previ-ously discovered bugs, it provides better guidance for Strat-egy to derive new deep bugs based on prior bugs found from TestGen. In Step 2 , Orchestrator performs subtask O.2 and uses prompts to analyze attack scenarios proposed earlier by Strategy together with bugs discovered by TestGen, produc-ing a global state. In Step 3 , Orchestrator performs subtask O.3 and sends the context information, i.e., the results of bug exploitation and state analysis, to Strategy. In Step 4 , Strategy performs subtask S.1 and analyzes proto-col constraints using prompts. In Step 5 , Strategy performs subtask S.2 and employs the context from bug exploitation and state analysis. In Step 6 , Strategy performs subtask S.3 and, using prompts together with the results from Steps 4

and 5 , generates attack scenarios. Specifically, for a given consensus protocol, it simulates scenarios by controlling node behaviors such as joining, going offline, crashing, or recovering, as well as manipulating message sending orders and conflict relationships to produce bug-prone scenarios. 3.6. Communication Mechanism 

Communication in Agora involves the memory, the global state, and the consensus protocol code repository. As discussed in §3.4, the memory is lightweight and is accessed by the agents only once during the entire execu-tion. By maintaining the global state within Orchestrator, Strategy does not need to track previously generated at-tack scenarios or discovered bugs and can easily retrieve

As a result, Strategy and TestGen do not need to manage contextual information and can focus solely on their respec-tive tasks, while Orchestrator maintains the system-wide state and effectively manages long-horizon objectives. The global state itself is succinct, and its transmission occurs only between Orchestrator and Strategy or TestGen, without direct communication between Strategy and TestGen. Such communication is necessary to maintain system consistency and cannot be further reduced. In Agora, agents can ac-cess the consensus protocol code repositories via the Model Context Protocol (MCP) (Anthropic, 2024). Overall, this design keeps the system concise, with commu-nication limited to essential information, resulting in low communication overhead. 

## 4. Experiments 

4.1. Experiment Setup Baselines. Existing studies that leverage LLMs for bug detection either lack integration with multi-agent systems or do not focus on deep logic-level bugs (Kaniewski et al., 2025; Sheng et al., 2025). To address these limitations and obtain appropriate comparison baselines, we adapt Re-Act (Yao et al., 2023) by augmenting it with code-reading tools, and implement it using state-of-the-art code-oriented large language models as baselines. Specifically, we evalu-ate three commercial LLMs, Claude Sonnet 4.5 (Anthropic, 2025), Gemini 3.0 Pro Preview (DeepMind, 2025), and GPT-5.2 (OpenAI, 2025), as well as one open-source model, Qwen3 Coder 480B A35B (Bai et al., 2025). 

Implementation Details. For our implementation, we ac-cess all models via APIs, eliminating the need for GPUs. For all large language models, we set the temperature to 0.2. In each experiment, both Agora and the baselines are allowed to run for three hours. For Agora, the number of reflection loop iterations ( num iter ) is set to 50. 4.2. Overall Results 

Table 1 presents the overall experimental results of Agora compared with the baselines. From the results, we observe that regardless of the underlying LLM, the baselines are only able to identify simple implementation-level bugs, whereas our Agora, by design , consistently discovers protocol-level logic bugs and does not report implementation bugs. Implementation-level bugs can typically be detected through static analysis and often result in issues such as integer over-flows or memory violations. In contrast, logic bugs are significantly more subtle and may compromise system cor-rectness, data consistency, or transaction security, leading to severe real-world consequences. Therefore, Agora ex-plicitly focuses on deep protocol logic bugs rather than implementation-level issues. Detailed examples of the two bug types are provided in Appendix G. The experimen-tal results demonstrate that our multi-agent design enables Agora to effectively focus on the detection of deep logic bugs, saving the computational resources and validating the effectiveness of our approach. More specifically, we find that even when augmented with ReAct design, baselines using Gemini 3.0 Pro Preview and Claude Sonnet 4.5 fail to identify any implementation-level bugs. GPT-5.2 is the most effective baseline model, detect-ing 22 bugs across three repositories. Qwen3 Coder 480B A35B, possibly benefiting from its optimization for code understanding, is able to detect three bugs in the EPaxos repository, all of which are also identified by GPT-5.2. In to-tal, the baselines uncover 22 zero-day implementation bugs, which we report as an additional contribution toward im-proving robustness of existing consensus implementations. In contrast, Agora achieves the best performance when built upon Gemini 3.0 Pro Preview, identifying 11 protocol-level logic bugs across four widely used repositories, including production-level systems. This result is particularly notable given that the Gemini-based baseline fails to detect any bugs. Moreover, Agora consistently discovers logic bugs when instantiated with each of the other three LLMs, further demonstrating the robustness and generality of our approach. As some bugs are discovered by multiple LLM-backed in-stances, Agora identifies a total of 15 unique zero-day logic bugs, substantially enhancing the security of distributed con-sensus systems. These results further validate the effective-ness of integrating our framework with the HDT paradigm. 

Practical Usage. We further analyze all the 46 bug re-ports generated by the TestGen agent in Agora across the four LLMs. Among them, 34 reports correspond to real protocol-level logic bugs, resulting in a false positive rate of only 26.1% . This low false positive rate indicates that Agora does not require extensive manual filtering, making it a highly efficient and practical tool for engineers and re-searchers to discover deep logic bugs in real-world systems. 

Bug Summary. Based on the categories of consensus logic bugs, we further analyze the 15 bugs discovered by Agora. As shown in Figure 3, violations are observed across all bug categories, demonstrating that Agora is capable of trigger-ing a wide range of deep protocol-level bugs. Specifically, Recovery & Execution Divergence bugs account for four cases (26.7%), tying with Persistence & Monotonicity Vio-lations as the most prevalent category. In addition, Message Binding & Signature Violations, Resource & Operational Visibility Violations, and Dependency & Topology Flaws account for 2, 2, and 3 bugs, respectively. Detailed examples of logic bugs from different categories are in Appendix H.

## B. Mathematical Modeling of the Multi-Agent System 

LLMs power a wide variety of applications (Cao et al., 2026; Zeng et al., 2025). Specifically, in a formal sense, we characterize an autonomous LLM-powered entity as a quintuple E = ⟨B , P, A, Ω, Γ⟩. Here, B represents the core linguistic brain (the LLM), P signifies the perceptual manifold of the target environment, and A denotes the repertoire of executable operations (tools). The mapping function Ω : P → A acts as the decision logic, while Γ encapsulates the persistent memory state. The operational trajectory of such an entity follows a recursive observe-deliberate-act cycle: at any discrete temporal index k, the entity ingests a sensory input pk ∈ P , leverages B to synthesize a strategic intent based on Ω, and dispatches an intervention ak ∈ A to modulate the system state. To tackle the intricate state spaces of consensus implementations, we extend this individual paradigm into a collaborative collective G = {E 1, E2, . . . , Em, X } . In this formulation, each Ei

represents a specialized functional unit, and X defines the coordination medium that facilitates information exchange and role synchronization. This multi-agent architecture transcends the cognitive boundaries of isolated models by decomposing complex protocol verification into modular, inter-dependent reasoning sub-tasks, a methodology that has proven more robust than monolithic approaches in recent literature, e.g., CAMEL (Li et al., 2023), MetaGPT (Hong et al., 2023), OWL (Hu et al., 2025).

## D. Constraints Details 

Here, we provide examples of constraints generated for BFT and CFT systems, which help generate scenarios tailored to different types of consensus protocols. 

Protocol Constraints & BFT Attack Model 

Protocol Type Constraint: BFT (Byzantine Fault Tolerant) This model assumes an environment capable of tolerating Byzantine faults , where nodes may behave maliciously or arbitrarily. 

I. CFT-type Attack Vectors (Crash Faults): 

• Availability: Node crash, recovery, and frequent restarts. • Network: Partitions, significant latency, packet loss, and message reordering. • Logic: Boundary condition exploits and duplicate message utilization. 

II. BFT-specific Attack Vectors (Malicious Behavior): 

• Equivocation: Sending conflicting votes/proposals for the same round/height to different peers. • Payload Tampering: Injecting invalid, malformed, or tampered protocol messages. • Selective Broadcast: Distributing messages to a subset of nodes to induce state inconsistency. • Strategic Withholding: A malicious leader intentionally delays or withholds proposals to stall liveness. • State Forgery: Crafting fraudulent certificates, votes, or cryptographic signatures. • Timing Attacks: Exploiting vulnerabilities within the protocol’s timing and timeout assumptions. 

III. Attack Mindset & Objectives: Assuming control of up to f malicious nodes ( f < n/ 3), the attacker aims to: 

15 Agora : Toward Autonomous Bug Detection in Production-Level Consensus Protocols with LLM Agents 

1. Safety Violation: Can honest nodes be coerced into accepting an invalid state? 2. Liveness Violation: Can the system be prevented from reaching consensus indefinitely? 3. Consistency Violation: Can different honest nodes be led to perceive divergent “truths”? 

Note: Effective testing must simulate a clever attacker who strategically exploits the protocol’s specific logic weaknesses rather than just random fault injection. 

Protocol Constraints & CFT Fault Model 

Protocol Type Constraint: CFT (Crash Fault Tolerant) This model assumes an environment that can only tolerate crash faults . All nodes are assumed to be honest; they either follow the protocol correctly or stop functioning entirely. 

I. Allowed Attack Methods (Non-Malicious Faults): 

• Node Dynamics: Random node crashes (immediate stop) and restarts (recovery with potential loss of in-memory state). • Network Adversity: Network partitions, extreme communication delays, message loss, and packet reordering. • Implementation Limits: Boundary conditions (overflows, null values, extreme inputs) and duplicate resource utilization. 

II. Prohibited Attack Methods (Byzantine Behavior): 

• No Malicious Messages: Sending forged, tampered, or intentionally malformed payloads is strictly prohibited .• No Equivocation: Double voting or sending conflicting proposals in the same round is not allowed. • No Selective Broadcast: Intentional inconsistency through partial message distribution is excluded. • No Identity Theft: Actions involving signature forgery or impersonating other nodes are disallowed. 

III. Core Assumptions: In a CFT environment, the testing focus shifts from ”malice” to ”reliability”: 

1. Binary State: Nodes are either ”Correct” (following the protocol) or ”Failed” (stopped). There is no ”half-dead” or Byzantine behavior. 2. Honest Participation: All participating nodes are assumed to act in good faith to reach a common consensus. 3. Recovery Consistency: The system must maintain safety and eventually regain liveness after network partitions heal or nodes recover. 

Note: Testing for CFT protocols should exhaustively simulate unstable infrastructure and edge-case logic rather than adversarial intent.

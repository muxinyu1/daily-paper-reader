Abstract 

Language models can find thousands of severe software vulnerabilities, and agents are increasingly being misused for cyberattacks. To avoid detection, attackers fre-quently distribute their misuse, splitting a harmful task across many user accounts so each individual transcript looks benign. Because safety monitors score only one agent context at a time, they are structurally blind to misuse that is only visible in aggregate, across many accounts. We show this gap is real by building, to our knowledge, the first distributed agent attack, a multi-agent scaffold that completes hard cybersecurity tasks while hiding the harmful objective across subagents with limited contexts, evading a standard monitor that catches it only a fifth as often as prior agent attacks. Towards a defense, we develop an online stateful monitor that uses real-time clustering to collect weak suspiciousness signals across many agent transcripts, and escalates only rarely to a language model that flags misuse across user accounts. In evaluations with large-scale simulated datacenter traffic, our mon-itor Pareto dominates standard monitors, catching distributed attacks 30% earlier and flagging cyber misuse before it reaches the most harmful stages. Crucially, this comes at negligible additional latency for ∼99% of user traffic. This detection advantage persists but narrows as the benign background traffic grows very large. After an extensive red-teaming exercise, we improve the defense and surprisingly also find that it catches standard jailbreaks, since adaptive attackers reuse attack variants across accounts. Our results point toward a new class of safety monitors which reason over groups of users rather than isolated transcripts. 

1 Introduction 

Language models can now find thousands of high- and critical-severity vulnerabilities in software [Carlini et al., 2026, OpenAI, 2026]. With increasing alarm, model developers, cybersecurity practitioners, and government agencies report increasing misuse enabled by agents, ranging from scaled cyberattacks to surveillance on behalf of authoritarian regimes [Anthropic, 2025a,b, GTIG, 2025, OpenAI, 2025, Hilt and McArdle, 2025]. A recurring feature of misuse incidents is that the harm is rarely concentrated in a single request (e.g., ‘implement this attack end-to-end’). Consider the following example, where an attacker hides a harmful task across many benign-looking contexts. 

Agent misuse case study from Anthropic [2025a]. Attackers targeted thirty global victims with Claude Code, hiding their operation with an external orchestrator that split it into “small, seem-ingly innocent” defensive-looking tasks (e.g., “map this system” and “validate these credentials”). Anthropic later identified the misuse by stitching evidence across separate agent sessions. This case study shows that attackers can evade standard safety monitors, which use only the current language model context to classify whether an interaction is harmful or benign, by splitting misuse tasks across many agent contexts and user accounts (Figure 1). 2 Standard Monitors Cannot Identify Distributed Agent Attacks 

In distributed agent attacks , a single harmful objective is decomposed into subtasks. These subtasks are executed across separate agent contexts in different user accounts, so that while each local subtask transcript appears benign, the aggregate task is harmful. While distributed misuse has featured prominently in recent threat disclosures [GTIG, 2025, OpenAI, 2025, Anthropic, 2025a], distributed agent attacks have not yet been studied in the academic literature. To our knowledge, we implement the first such attack, evaluate it, and find that standard monitors are highly susceptible to it (Figure 3). In later sections, we use this attack to evaluate our online stateful monitor. 

Distributed agent attack. Our distributed agent attack effectively evades standard monitors (Fig-ure 3), by distributing misuse subtasks across many agent contexts, using a multi-agent system for difficult cybersecurity tasks. This attack generalizes two lines of prior work; the first breaks up harmful tasks in simple question-answer settings [Glukhov et al., 2024, Jones et al., 2025, Brown et al., 2025], the second performs decomposition attacks in a single model context [Li et al., 2024, Yueh-Han et al., 2025] (‘sequential decomposition attack’ in Figure 3). Our agent distributes misuse across different user accounts, instead of just a single agent context, make it more difficult to monitor. We build our distributed agent attack from a multi-agent scaffold, ARTEMIS [Lin et al., 2025]. Our scaffold completes cybersecurity tasks using an agent orchestrator which receives the full task objective, and then delegates subagents. The subagents receive a narrow task description, which our orchestrator frames as a non-cybersecurity, ordinary engineering task, thereby concealing the full harmful objective. When the subagents complete tasks, their outputs are stored in a database so the orchestrator can combine partial results. A verifier tool periodically checks if the cyber task is solved. 3Figure 3: Our distributed agent attack evades standard monitoring. The distributed agent takes longer to detect ( left ) and is much harder to catch ( right ) than prior agent scaffolds or attacks.

Monitors struggle to detect distributed attacks. Distributed agent attacks can accomplish sig-nificantly more harm than other attacks before being caught by a standard monitor. In Figure 3, we find that our distributed agent attack can typically accomplish 2 times (or more) actions towards an objective, and cause monitors to have 30%–80% worse recall, relative to other attacks. We use the same subset of difficult Cybench cybersecurity tasks, and measure how well gpt-4.1-mini as the monitor can detect whether an agent is attempting a cybersecurity task or a software task. We keep the monitor and prompt fixed, and compare our distributed attack to three baselines: directly prompting gpt-5.2 to complete the cybersecurity task, a baseline ARTEMIS scaffold, and a se-quential decomposition attack. Existing sequential decomposition attacks (attacks that implement a decomposition attack in a single context [Li et al., 2024, Yueh-Han et al., 2025]) were designed for non-agent coding tasks, so we create a new performant agent decomposition attack. Distributed agent attacks can take more than twice as many actions before detection (Figure 3), leading to substantially more harmful outcomes. For example, sequential decomposition attacks are on average caught while just obtaining access, whereas distributed agent attacks can essentially perform the entire exploit (we label attack stages based on [Al-Sada et al., 2024]; see Section 4 for methodology). At modest false-positive rates, the monitor essentially always catches sequential decomposition attacks, but catches distributed agent attacks only 20% of the time. 

3 Online Stateful Monitoring 

Consider a defender who deploys language model agents to tens of thousands of users. The defender receives a large stream of new requests from users querying the agents, and also monitors that agent’s actions and outputs. The defender classifies every new query and agent action as either benign (and allowed to continue) or harmful (and flagged). An attacker, attempting to misuse the agent to complete a harmful cybersecurity task, can create arbitrary new accounts. Because there are many more benign prompts and actions than harmful ones, the attacker attempts to perform the distributed misuse attack defined in Section 2, where each account separately prompts an agent to perform a subtask that appears benign in isolation, but together implement a cybersecurity misuse task. A standard monitor is a language model judge that uses the current agent context when classifying an interaction as benign or harmful [Inan et al., 2023, Sharma et al., 2025]. However, our results demonstrate that standard monitors perform poorly against distributed agent attacks, which hide evidence of misuse among many user accounts. The harmful objective becomes identifiable only after several related contexts are collected, which is not possible for standard monitors. Our stateful online monitor is designed to solve the problem of identifying distributed attacks, by collecting evidence of a misuse attempt from among many different user accounts and agent contexts in real-time. At a high-level, for every new user prompt, our monitor retrieves related prior agent contexts to judge whether the current request is a part of a distributed agent attack. As shown in Figure 2, each request is embedded and clustered with all previous agent contexts from all other users. Finally, another monitor judges across contexts if the request, along with semantically related requests in the same cluster, constitute a distributed agent attack. See Algorithm 1 for the full procedure. 

Retrieving related contexts with online clustering. Instead of only classifying misuse on a single agent context, for each new request our monitor first retrieves prior agent contexts from other user accounts, and then classifies this collection of transcripts as part of a (potential) aggregate misuse attempt. But how are the most relevant transcripts retrieved, in real-time, out of tens of thousands of options? One approach for retrieving potential clusters of misuse would be to periodically embed and cluster all prior transcripts [Tamkin et al., 2024]. But this is both too slow and too expensive for live monitoring. If there are Nt prior agent transcripts at time t, then re-clustering for a new request costs O(dN 2

Monitoring across many contexts. We cluster prior conversations, and show the most relevant clusters to the cross-context monitor to label xt as harmful. After the standard monitor scores the current request xt and is assigned to a set of clusters Gt, a cross-context monitor is called in two cases (see Algorithm 1). In the first case, the current request receives a high suspiciousness score, but not sufficiently high to be refused outright. In the other case, the other sessions in Gt, from different user accounts, repeatedly receive new contexts. The cluster stores a buffer of prior suspicious contexts, which are deduplicated via hash, sorted by suspiciousness score. To flag a cluster, the largest k scores are summed and exceed a threshold calibrated on held-out data. We also require ≥ m prior contexts above a certain score. This lets several moderate signals trigger review. During our experiments, we use held-out data to tune these thresholds. 

4 Experiments 

Our real-time defense stores new requests with semantically related prior traffic, and in rare cases sends suspicious transcripts to a cross-context monitor, which judges if there is a coordinated distributed attack occurring. In this section, we evaluate three questions: (i) whether state improves campaign detection at realistic false positive rates, (ii) whether it catches campaigns earlier in the attack trajectory, i.e. before the attacker can do significant harm, and (iii) whether these gains preserve low latency for ordinary traffic. 

Attack. Harmful transcripts come from the distributed attack in Section 2, where a gpt-oss-120b su-pervisor receives a cybersecurity task [Zhang et al., 2024, 2025] and delegates to gpt-5.2 subagents with fresh conversation context; the monitor sees only the context from the gpt-5.2 subagents. 

Policy and judge. We use a refusal policy that treats exploit tasks used in CTF benchmarks as disallowed, while allowing ordinary software engineering and defensive security work. This is not a claim that CTF tasks should be refused, but this policy allows us to evaluate realistic cybersecurity tasks. All monitor judgments use the same cyber safety prompt with gpt-4.1-mini-2025-04-14 .

Traffic. We construct streams by sampling a distributed attack, and randomly inserting it into benign agent traffic from coding-agent and web-chat traces [Jimenez et al., 2024, Trofimova et al., 2025, Yang et al., 2024, Merrill et al., 2026, Barres et al., 2025, Lù et al., 2024], where the benign traces are sampled 10 x or 100 x more than the agent conversations (e.g. for 272 harmful conversations, there are 27200 benign conversations sampled without replacement). Stateful monitoring improves misuse detection. Figure 5 shows that the operating points from our stateful monitor Pareto dominate the standard monitor. With 10 x more benign than attack conversations, the stateful moniter reaches 81.1–87.2% campaign recall at 2.35–3.09% benign conversation FPR, compared with 75.0% recall at 2.72% FPR for the standard monitor. At 100 x, the the stateful monitor reaches 65.8% recall at 1.47% FPR; at the same FPR, the best standard monitor achieves only 55% recall. Thus, adding statefulness enables substantially better recall.

5 Adaptive Redteaming 

We redteamed the stateful monitor to investigate two questions: (i) does sustained whitebox redteam-ing find new failure modes which we can use to improve the defense, and (ii) does stateful monitoring improve detection of attacks other than the distributed cyber attacks of Section 4? 

Scale. Six redteamers attacked successive versions of our monitor over two months, producing 

> 100 , 000 attacks across 2,985 sessions and 1,159 user IDs. Affordances. Redteamers had whitebox access to the safeguard, had access to the cluster assignments and suspiciousness scores, and could reset the clusters or open new sessions. We deployed the monitor as an AWS service, where redteamers could create fresh isolated namespaces for new attempts, and could reset their own monitor state without effecting the shared deployment. We also provided a browser UI, shown in Figure 11, that displayed parallel sessions alongside safeguard metadata such as suspiciousness scores and cluster assignments. Targets. Participants attempted to perform exploit tasks from BountyBench [Zhang et al., 2025] and CyBench [Zhang et al., 2024] chosen so that GPT-5.2 could solve them but weaker attack models (gpt-oss-120b [Agarwal et al., 2025]) generally could not, so we could isolate the uplift enabled by stronger models. Attack families. Four strategies (Figure 7) emerged in post-exercise discussion: monitor-context poisoning (faking classifier outputs and approvals to prompt inject the cross-context monitor), sequential decomposition inside one conversation [Li et al., 2024, Yueh-Han et al., 2025], distributed agent attacks across sessions and accounts, and semantic-cover jailbreaks , i.e. standard jailbreaking by framing the attempt as ordinary code review or debugging. 

Redteaming exposed two stateful monitor failure modes. The first failure mode was that re-naming files or file paths, or using different vocabulary to refer to tasks, could allow an attacker to fool the embedding model and split a distributed attack across nearby clusters, and thereby delay review; the second was that ambiguous suspicious examples could persist too long in cluster memory and cause a high false positive rate. These failures informed the development of the final algorithm for our defense, in Section 3. To reduce splitting the attack across nearby clusters, the monitor assigns the current request to a set of related clusters Gt, rather than treating a single centroid as the only summary for a given attack. To reduce false positives, the contexts Bj⋆ in each cluster are deduplicated, sorted by suspiciousness, and used to assemble the top-k evidence for the cross-context monitor in Algorithm 1.

6 Conclusion 

Distributed agent attacks spread a single harmful cybersecurity task across many accounts or contexts, leaving each transcript benign and making it so that standard safety monitors are structurally unable to identiy misuse. We introduce an online stateful monitor that finds information across user transcripts in real-time via clustering, and escalates only the rare suspicious cluster to a language model which reasons across contexts. This catches distributed misuse earlier into an attack, and at higher recall, than a matched standard monitor, while adding negligible latency for the majority of traffic. We performed white-box redteaming to harden the defense, where we found that adding state also helps beyond distributed misuse, by catching jailbreaks which attackers develop across contexts or accounts. Notably, stateful monitoring does not solve distributed misuse—an adaptive attacker who works very hard to obfuscate their tasks, or who uses strong jailbreaks, can still hide evidence and partially complete tasks. Nonetheless, the defense points towards a new class of monitors that can efficiently reason over populations of users, rather than isolated transcripts.

D Redteaming the online stateful monitor 

D.1 Redteaming affordances 

Figure 11 shows the dashboard used in the final red-teaming round. The interface exposes different user sessions, side-by-side with independent session identifiers, along with the ability to start a fresh session, clear the persistent defense state. The interface also allowed the redteamers to change the base language model. Crucially, it also rendered safeguard-side metadata alongside each response, including whether the system detected or would block the request, the decision scope, harmfulness scores, cluster assignment, nearest-cluster similarity, and selected escalation statistics. These affor-dances allowed participants could compare attack variants in parallel, test what evidence persisted across sessions, check whether paraphrases or renamed files were routed into the same bucket of semantically related activity, and distinguish failures of the base model from interventions by the stateful monitor. In other words, the dashboard exposed precisely the cross-context mechanisms that a distributed attacker would seek to probe, making it possible to red-team the routing, memory, and escalation components of the defense rather than only its visible refusals.

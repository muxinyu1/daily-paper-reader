## 1 Introduction 

Cybersecurity is a domain in which Large Language Models (LLMs) have moved from demonstrations to daily practice. Penetration testers, blue teams, and capture-the-flag (CTF) competitors now rely on LLM-driven harnesses to enumerate hosts, draft exploits, triage alerts, and write defensive playbooks. Those traces, taken together, encode how human operators actually use general-purpose LLMs to do security work in the wild. Supervised fine-tuning (SFT) of a cybersecurity-specialised LLM benefits from three properties of the training corpus, each grounded in the post-training literature: scale — production-grade open SFT releases retain 10 5 to 10 7 records after quality filtering (T ¨ULU 3 at 1

Public cybersecurity datasets to date satisfy at most two of these properties at once. Knowledge-style benchmarks such as CyberMetric [ 11 ] or SecQA [ 12 ] test factual recall but do not reflect operator behaviour; offensive instruction sets such as CySecBench [ 13 ] cover individual prompts but lack trajectory context and tool outputs; vulnerability-research corpora derived from CVE/MITRE feeds [ 14 ] cover schema rather than reasoning. None of these capture the iterative, tool-augmented, multi-host trajectories that an actual operator produces inside a shell loop. We present CAI Dataset , a corpus engineered as the explicit goal of a multi-year programme. The motivating observation was published in PentestGPT [ 2 ]: LLM perfor-mance on real penetration-testing workflows is bottlenecked by the absence of expert operator trajectories, not by base-model capability. The follow-on agent framework CAI [1]was architected from the start as a data collector first and a user-facing CLI second. Over fourteen months (2025-03-22 to 2026-05-25, 428 days), it captured 230,935 session logs and 26,027,742 user prompts from 16,768 distinct source IPs, totalling 18.07 TB of durable storage. The resulting signal is operational: nation-state exercises, bank assessments, public and private penetration tests, flag-driven lab work, authenticated API and bug-bounty hunting, mobile and reverse-engineering sessions, robotics / IoT targets, continuation prompts that resume prior work, and failed as well as successful tool steps. CAI Dataset therefore supports both SFT on cybersecurity language and studies of how operators choose models, tools and infrastructure on realistic security tasks. This matters because the newest cyber-capable frontier systems are increasingly being deployed through gated access and opaque post-training, often with RL-style optimisation, rather than open, reusable cybersecurity SFT mixtures. OpenAI describes GPT-5.5-Cyber as a limited-preview, more permissive access mode for authorized defensive workflows, while noting that the initial cyber-permissive preview is primarily about access behavior rather than uniformly higher cyber-evaluation performance [ 15 ]. RL can elicit powerful reasoning without human-written traces, as DeepSeek-R1-Zero demonstrated, but the same paper reports readability, language-mixing and broader-task limitations that required a multi-stage pipeline with rejection sampling, supervised fine-tuning and RL [ 16 ]. Cybersecurity needs that supervised layer: it supplies the distribution of real goals, tools, artifacts, failures, safety boundaries and operator corrections that a reward function alone does not specify. Gated and remote-only access is also operationally incompatible with a significant fraction of real cybersecurity work, which runs inside classified, air-gapped or contractually network-isolated environments: private on-prem models therefore remain a load-bearing blueprint for future cybersecurity operations, and they can only be trained on a corpus that captures the operator distribution CAI Dataset sets out to provide. 

Contribution. CAI Dataset is, to the best of our knowl-edge, the largest described corpus of LLM-driven cyber-security trajectories: 230,935 session logs, 26,027,742 user prompts, 16,768 source IPs across 123 countries, 4,187 unique LLM identifiers, 428 days, 18.07 TB of durable storage, and 683,606 URL references against 23,147 unique target domains. Section 2 positions CAI Dataset against prior cybersecurity training data; Section 3 describes the collection pipeline, schema, redaction policy and aggregate statistics; Section 4 sketches downstream uses; Section 5 covers limitations, dual-use considerations and the partner / customer access policy. 

## 2 Related Work 

We position CAI Dataset against four active strands of prior work: cybersecurity-specific corpora and bench-marks, cybersecurity-specialised foundation models, the open generic-SFT corpus lineage that defines current best prac-tice, and agent-trajectory SFT (the closest methodological precedent for our format). 

2.1 Cybersecurity-specific corpora and bench-marks 

Public cybersecurity datasets for LLMs cluster into three subfamilies. Knowledge benchmarks (CyberMetric [ 11 ], SecQA [ 12 ], CTIBench [ 17 ]) probe factual recall against standards, CVEs and threat-intelligence taxonomies; they are useful for evaluation but provide no operator behaviour, no tool outputs and no multi-turn structure suitable for SFT. Instruction-prompt collections (CySecBench [ 13 ], Cyber-LLMInstruct [ 8 ], Heimdall [ 18 ]) ship roughly 10 4 single-shot prompts targeted at safety alignment or defensive instruction tuning, with the recent CyberLLMInstruct release (54 ,928 pseudo-malicious instruction-response pairs) documenting an explicit safety/performance trade-off that any cybersecurity SFT effort has to address. Benchmark frameworks such as Cybench [ 19 ], D-CIPHER [ 20 ] and CTFTiny / CTFJudge [21 ] build multi-agent evaluation environments for CTF-style challenges, often using LLM-as-judge scoring; they ship evaluation artifacts rather than training trajectories. SecKnowledge / CyberPal.AI [ 22 ] is the closest analog in scope: a synthetic instruction-tuning corpus derived from curated cybersecurity seeds. CAI Dataset differs from all of these in three dimensions: it is trajectory-level rather than prompt-level, it is collected from real operator behaviour rather than synthesised, and it is two-to-four orders of magnitude larger. 2.2 Cybersecurity-specialised foundation mod-els 

The cybersecurity-specialised model space splits across three emergent recipes; we discuss each by representative published model. 

Recipe 1: large-scale continued pretraining. 

Foundation-Sec-8B [ 23 , 24 ] continues a Llama-3.1-8B base on ∼ 5.1 B curated cybersecurity tokens through a four-stage pipeline (web crawl, relevancy filter, MinHash deduplication, quality filter), and matches Llama-3.1-70B and GPT-4o-mini on cyber-specific tasks. Trend Micro’s Llama-Primus-Base [ 25 ] reports a +15 .88pp aggregated-benchmark gain over the same Llama-3.1-8B-Instruct base after continued pretraining on Primus-Seed plus Primus-FineWeb. On the encoder side, SecureBERT 2.0 [ 26 ] pretrains a ModernBERT backbone on 13 B text tokens plus 53 M code tokens of cybersecurity content and posts SOTA on cybersec NER, semantic search and vulnerability detection; CySecBERT [ 27 ] is the earlier BERT-base ancestor over 4 .3 M cybersec documents. The resource-efficient counterpoint is Salahuddin et al. [ 28 ], who push a Llama-3.3-70B-Instruct base to SOTA on CTI-MCQ (0 .718), CyberMetric (0 .933) and SecEval (0 .864) using only a curated 126-million-word corpus, outperforming larger specialised models trained on roughly 20 × more data. SecureFalcon [ 29 ] demonstrates that the same recipe distils into 44–121 M-parameter classifiers (94% binary / 92% multi-class on FormAI), useful at the edge but not as a general assistant. 

Recipe 2: instruction / preference domain adap-tation. CyberPal.AI v1 [ 22 ] SFTs on the SecKnowledge expert-driven instruction set and reports up to +24pp average over baselines on SecKnowledge-Eval. CyberPal-2.0 [ 30 ] scales this to a 4B–20B family with CoT and grounded evidence in SecKnowledge 2.0, with its 20B variant beating GPT-4o, o1 and o3-mini on CTI weakness-mapping. Lily-Cybersecurity-7B [ 31 ] is the Mistral-7B SFT on a 22k hand-crafted cybersec QA pair set, ZySec-7B [ 32 ] is a DPO-tuned Zephyr-7B over a 30+ cybersec domain preferences set (CIS, FedRAMP, PCI DSS, ATT&CK), and WhiteRabbitNeo / DeepHat [ 33 , 34 ] is the reference uncensored offensive-security line with successive Llama-2-13B/33B and Qwen2.5-Coder-7B bases. SEvenLLM [ 35 ] is the bilingual instruction-tuned analog targeted at incident analysis. Productized analogs include Google’s SecLM and Sec-Gemini v1 [ 36 ], integrating Mandiant and VirusTotal grounding with multi-step reasoning. 

Recipe 3: RL / reasoning distillation. The 2025– 2026 wave is dominated by RL-trained and reasoning-distilled cybersec agents. Pentest-R1 [ 10 ] reaches 24 .2% on AutoPenBench and 15 .0% on Cybench through SFT plus two GRPO stages over the data shape discussed in detail in §2.4. Cyber-Zero [ 37 ] synthesises agent trajectories from CTF writeups via persona-driven LLM simulation and trains the resulting model to +13 .1pp absolute over baselines on InterCode-CTF, NYU-CTF and Cybench; CTF-Dojo [ 38 ] is the executable-environment follow-up. xOffense [39 ] reports 79 .17% sub-task completion on AutoPenBench plus AI-Pentest-Benchmark by SFT on Qwen3-32B with a multi-agent shell-tool framework; the Random-Crypto GRPO line [ 40 ] lifts Llama-3.1-8B from 0 .35 to 0 .88 Pass@8 on cryptographic CTFs. PrivEsc-LLM [ 41 ] post-trains a 4B local model in two stages on procedurally generated Linux privilege-escalation environments (SFT on 1 ,000 traces, then RLVR), reaching 95 .8% success on a held-out 12-scenario benchmark within 20 interaction rounds, within 1 .7pp of Claude Opus 4.6 at the same budget. At the smaller-agent end, Hackphyr [ 42 ] reaches 94% win-rate in simple Net-SecGame scenarios, and the reasoning variants Foundation-Sec-8B-Reasoning [ 43 ] and Llama-Primus-Reasoning [ 25 ]apply DeepSeek-R1-style reasoning distillation directly to a cybersec base. Huang [ 44 ] sits across recipes 2 and 3, comparing SFT vs. LoRA vs. QLoRA on cybersecurity tasks; Yu et al. [ 45 ] builds a small cybersec-expert LM via multi-source corpus plus CTI-Bench chain-of-thought distillation. 

Where CAI Dataset fits. Continued-pretraining corpora describe what cybersecurity texts look like; instruction / preference sets describe which curated cybersec questions a model should answer; RL pipelines optimise against narrow verifiable reward surfaces (flag capture, exploit success). None of the three recipes by itself supplies the empirical distribution of what operators actually do 

with LLMs over multi-step tool-augmented engagements: the triage, failed commands, tool-output interpretation, patch validation, scope-boundary discussion, benign business context and operator repair that appear in real sessions. The current frontier trend exposes the same gap from the opposite direction. GPT-5.5-Cyber shows that gated, post-trained or access-tuned frontier models can become highly capable at vulnerability discovery and authorized security workflows [ 15 ], but such releases do not by themselves create a reusable public distribution of cybersecurity behaviour, and Foundation-Sec explicitly reports that the scarcity of specialised cybersecurity training data limits adoption [ 23 ]. CyberLLMInstruct also shows that cybersecurity fine-tuning changes both capability and safety, requiring explicit miti-gation rather than reward-only optimism [ 8]. A dataset at 

CAI Dataset scale supplies the missing substrate underneath all three recipes: continued-pretraining mixes can ingest the operator-prose subset, instruction tuning can be re-weighted by the role-mix and target-indicator metadata, and reasoning / RL pipelines can use the trajectory bodies as both expert demonstration and reward grounding. 

2.3 Open SFT corpora and post-training mix-tures 

The reference open generalist SFT release is T ¨ULU 3 [ 3], whose SFT mix retains 939 ,000 records after a four-axis GPT-4o LLM-as-judge cascade (helpfulness, instruction-following, honesty, truthfulness) and is followed by DPO and RLVR stages. OpenHermes-2.5 [ 4] ships 1 ,001 ,551 multi-source conversations; SmolTalk [ 46 ] ships approximately 1 .1 M records targeted at small-LM training. A direct head-to-head comparison [ 7 ] shows that the two corpora yield measurably different students under matched training. Magpie [ 5 ]demonstrates that 4 million self-synthesised instructions filtered to 300 ,000 by quality and difficulty can match much larger mixes. NVIDIA’s Llama-Nemotron [ 47 ] consolidates a five-stage recipe (NAS, recovery pretrain, SFT, RL, RLHF) and publicly releases its 40 M+-sample post-training mixture [ 6 ]; the mix is currently the largest open post-training corpus and explicitly trains the model to switch between reasoning-on and reasoning-off modes. Hermes 3 [ 48 ] consolidated the ChatML format with Nous-style <tool call> XML envelopes for tool-augmented agentic SFT, and the Qwen3 post-training release [ 49 ] adopts the same convention as its default tool-call template. The NVIDIA Front-Loading-Reasoning study [ 50 ] provides additional evidence that reasoning data in pretraining stabilises subsequent SFT and reduces catastrophic forgetting, a finding directly relevant to any CAI Dataset -driven SFT pipeline. 2.4 Agent-trajectory SFT and tool-use corpora 

The closest methodological precedent for CAI Dataset is the recent body of work on agent-trajectory SFT. SWE-bench [ 56 ] established the trajectory-corpus paradigm for software engineering. ToolACE [ 51 ] introduced a large-scale synthetic function-calling SFT dataset that explicitly trains for format diversity (every tool call is rendered as JSON, YAML, XML or Markdown over a 26 ,507-API pool); its Llama-3.1-8B-Instruct fine-tune (ToolACE-8B) is a standard tool-use baseline. ToolACE-R [ 57 ] extends this with iterative refinement and reports cross-template generalisation on BFCL and API-Bank. ToolMind [ 52 ]pushes the same direction to multi-turn reasoning-enhanced tool-use trajectories. The Agent Data Protocol (ADP) [ 9]is the landmark on agent-trajectory standardisation: it unifies thirteen heterogeneous trajectory corpora into one record schema (Action union of API-, Code- and Message-actions; Observation union of Text and Web), reports gains of roughly +20pp average over single-source SFT (e.g. SWE-Bench 7B from 0 .4% to 20 .2%, AgentBench-OS from 3 .5% to 27 .1%) and provides the converter pipeline that re-renders ADP records into the target tokenizer’s chat template. The PIPE study [ 59 ] is the central critical finding: across 16 environments from AgentBench and Agent-Gym, trajectory-SFT substantially amplifies interface shortcutting ; trained agents degrade sharply under minimal interface rewrites, while non-trajectory-trained models remain largely stable. 

Harness-overfit risk and mitigations for CAI Dataset .

The PIPE finding is a direct caution for CAI Dataset consumers: a model SFT’d na¨ ıvely on trajectories from a single harness can overfit to that harness’s specific tool-call syntax and degrade against any other harness. Two factors blunt this risk for the corpus we describe. First, recent collection is already multi-scaffold: Appendix A (schema versions v3 and v4) documents that the v4 line shape carries a top-level scaffold field, and 2025–2026 logs are produced by the CSI publishing scaffold, the 

CAI CLI, and additional harnesses routing through the same proxy. The latest data is therefore not CAI -specific. Second, the published mitigations are mature enough to be applied at the consumer side. Our recommended recipe, drawing on ADP [ 9 ], ToolACE / ToolACE-R [ 51 , 57 ] and CodeAct [ 58 ], has three steps: (i) at conversion time, project every trajectory into an ADP-compatible abstract action record and preserve the originating scaffold as provenance metadata, so the literal envelope is never baked into the assistant turn; (ii) at training time, re-render each record into multiple envelopes per example with uniform sampling (OpenAI tool calls JSON, Hermes-3 <tool call> XML, Qwen3-Coder <function=...> XML, CodeAct Python) and mix in 20–30% non-cybersecurity ADP sources (SWE-Gym, CodeActInstruct, Mind2Web) to break corpus-specific priors; (iii) at evaluation time, apply PIPE-style interface rewrites on a held-out slice (alias tool names, swap envelope family, reorder kwargs) and report Interface Reliance alongside task success, with a target of < 5% delta versus the native-harness result before claiming portability. This recipe makes 

CAI Dataset a substrate for harness-portable cybersecurity models rather than a single-scaffold corpus. 

Pentest-R1 and the cost of pre-decomposed formats. The closest cybersecurity-specific point in this family is Pentest-R1 [ 10 ]. Its base model is 

DeepSeek-R1-0528-Qwen3-8B , whose reported pre-finetune baseline is 3 .0% on AutoPenBench and 7 .5% on Cybench (unguided, pass@3). The full Pentest-R1 model reaches 24 .2% on AutoPenBench and 15 .0% on Cybench through a sequential pipeline: a Stage 1 SFT pass over ≈ 14K pre-decomposed Thought-Command-Observation tuples (constructed from 500 + expert walkthroughs), then GRPO offline RL on the same tuples, then online RL in InterCode-CTF. The reported ablations are instructive about which technique contributes what. SFT alone on the TCO tuples leaves the AutoPenBench / Cybench scores at the base’s 3 .0% / 7 .5%, i.e. no measurable solve gain. GRPO offline-RL alone moves the model to 9 .1% / 12 .5%. The full pipeline (SFT + GRPO offline + GRPO online) reaches 24 .2% / 15 .0%. The two takeaways for a corpus-design paper: (i) SFT is not the most performant regime on this data shape, yet it is a load-bearing prerequisite: removing it from the full pipeline drops AutoPenBench from 24 .2% to 9.1%, so SFT contributes +15 .1pp on AutoPenBench and +2 .5pp on Cybench when layered before GRPO; (ii) Pentest-R1’s records fix the schema as { initial prompt , steps = [ {thought , command , result }] },lifting the thought and command fields out of the chat body. This pre-decomposition tightly couples the data shape to one tool-call surface; modern instruction-tuned bases reward strict adherence to the native <tool call> {name,arguments }</tool call> 

XML envelope, and across multi-step CTF interactions the template mismatch compounds and costs solves. CAI Dataset addresses both problems at the data-shape level: the released-slice body preserves the original messages[] 

list with native tool-call envelopes, so the trainer chooses the decomposition rather than inheriting one, and the corpus is roughly three-to-four orders of magnitude larger across logs and prompts than the Pentest-R1 walkthrough set. 

## 3 Dataset 

3.1 Collection and schema 

CAI [1 ] is the first open-source agent framework for cyber-security automation, distributed both as a framework and as a CLI agent with telemetry as a first-class subsystem of the scaffold. Every session emits a per-session JSONL log capturing model request, model response, tool call, tool output and assistant message, and ships it to our servers; the disclosure and opt-out are documented in the project README. Telemetry is enabled by default because the bottleneck identified in PentestGPT [ 2 ] is the absence of expert operator trajectories. The collection model is behavioural: we record what real operators do with their own prompts on their own targets. No prompt is injected, no synthetic data is generated. Trajectories collected through 2025–2026 span multiple scaffold generations rather than a single harness (Appendix A). 

CAI is distributed free of charge to researchers; in lieu of payment for research use, we ask the research community to contribute usage data that helps identify areas for improvement, understand how the framework is being used in practice, and prioritise new features. That contribution is precisely what makes CAI Dataset 

possible, and the corpus is, in turn, fed back into the same community as detection-accuracy improvements, published findings and the dataset slices defined later in this section. 

CAI is additionally an output of an EU-funded research programme; use of the framework is correspondingly framed as a contribution to that public-research endeavour, which is a second load-bearing argument for the default-on collection model alongside the technical scarcity of expert operator trajectories. The legal basis for the data collection itself is documented in Section 5.3. A session log is one JSONL file whose lines are an ordered stream of records: session start envelopes, user and assistant messages, and full upstream chat-completion request/response pairs carrying the request model, the 

messages array, and the provider response. Tool results and subsequent user messages remain in the same file, preserving the act–observe–revise loop. The corpus spans four on-disk schema versions, enumerated in Appendix A.

3.2 Volume and growth 

At the time of writing, the corpus comprises 230,935 session logs and 26,027,742 user prompts produced over 428 days (2025-03-22 to 2026-05-25), totalling 18.07 TB of durable storage. One session corresponds to one JSONL log; the cor-pus is a collection of 230,935 trajectories rather than a single concatenated stream, so downstream consumers can iterate, shard or sample at the session boundary. 16,768 distinct source IPs exercise 4,187 unique LLM identifiers. Prompt length is heavy-tailed (measured in characters throughout): mean 625.4, median 75, standard deviation 41,252.2; the longest prompt observed is 180,626,398 characters (a single bulk-paste of multi-stage planning context). The order-of-magnitude gap between the median and the mean is the operator-workflow signature: most prompts are short shell-loop directives, but a long tail of planning briefs and continuation contexts drives the mean up. 

Growth rate and retraining cadence. Over the col-lection window, the corpus has accumulated at an average rate of roughly 540 session logs and 60k user prompts per day, with Figure 2 confirming a near-linear cumulative trajectory and a sustained weekly cadence. At that rate a daily increment represents ∼ 0.23% of the current corpus, a weekly increment ∼ 1.6%, and a monthly increment 

∼ 6.9%. For an SFT pipeline consuming CAI Dataset as a rolling training corpus, this suggests weekly retraining as the natural cadence (the new-data delta crosses one percent, large enough for measurable distributional shift), with daily checkpoints reserved for online-learning or rapid-feedback settings and monthly retraining sufficient when consumers prioritise stability over freshness. Prompt-length evolution. Beyond corpus size, the shape 

of individual prompts has shifted as automation needs and scaffold sophistication have grown. Two trends in the wider literature predict longer prompts over time: agentic frameworks have shifted operator workflow from single-line shell-loop directives to multi-step planning briefs with embedded context [ 9, 58 ], and crowd-sourced LLM corpora exhibit a measurable growth in mean user-prompt length as users habituate to the model [ 55 ]. ∼ 150 characters in the v1/v2 CAI CLI era (Mar–Aug 2025) to a 400–1 ,300-character range during the v3/v4 CSI publishing-scaffold era (Sep 2025 onward). 3.4 Temporal and geographic distribution 

Activity concentrates in the European working day (13:00– 16:00 UTC) with a midweek peak (Figure 6); off-hours activity is non-trivial. The geographic footprint spans 123 countries (Figure 7, Table 4). Europe accounts for the largest share of distinct contributors (85.7% after centroid removal), with secondary clusters in Asia (7.4%, dominated by India, China, Vietnam, Singapore, Japan) and North America (4.2%). A 11.4% share of source IPs (1,838) belong to public clouds and hosting providers, reflecting use from ephemeral cloud workstations, CI runners and remote attack platforms; the majority 37.8% come from residential or business ISPs, more representative of human-in-the-loop CTF and bug-bounty work. This split is relevant for SFT downstream: cloud-hosted trajectories are more likely scripted multi-target reconnaissance, residential ones trend interactive against a small number of targets.

A single forward pass over the 230,935 per-session caches surfaces two empirical leak channels that the redaction recipe has to cover: credential leaks (operator-side secrets pasted into the prompt body) and infrastructure paste (operator-side hostnames, S3 buckets and tenant subdomains pasted as engagement context). Both channels are reproducible from the released slices using the same scanner; this subsection reports the population shape that motivates the recipe. 

Credential leaks (F1). 1,059 sessions (0 .46% of the corpus, 1 ,174 counting placeholder-shaped matches before the entropy filter) contain at least one credential pasted into the prompt body by the operator. Figure 8 plots the population shape per credential family on a log–log axis with bubble area proportional to raw hits. The same data shows two distinct shapes: diffuse families (JWTs at 322 distinct values across 572 sessions and 104 distinct usernames; OpenAI project keys at 232 /256 /112; Google API keys at 97 /112 /38) and concentrated families (GitHub PATs at 12 distinct values across 17 sessions but only 4 distinct usernames; AWS access keys at 34 /37 /8). The diffuse families are the modal operator-behaviour signal at this scale: a hundred-plus distinct operators have at least one real-looking OpenAI project key or Google key in their prompt stream. The dominant prompt shape is a .env -style assignment pasted verbatim — the most-leaked single value, a GitHub personal-access token, appears 1 ,023 times across three sessions of one operator logged in as root ; the most-leaked Anthropic key appears 109 times in one session as 

export ANTHROPIC API KEY=’...’ .

§3.2. 4 Use Cases 

4.1 Supervised Fine-Tuning (SFT) 

The most demanding intended use of CAI Dataset is supervised fine-tuning (SFT) of foundation models for cybersecurity. We sketch three concrete recipes grounded in published practice; all operate on the released slices defined in Section 3.1. Section 4.2 then turns to the threat-intelligence and operator-behaviour studies the corpus enables beyond SFT. 

Direct SFT at T ¨ULU 3 scale. The fastest path treats 

CAI Dataset as the SFT mix for an off-the-shelf base. Filter to role label ∈ { offensive , defensive , intent } to drop neutral chatter; apply MinHash + LSH deduplication at Jaccard ≥ 0.8 [ 61 , 62 ] as the production default on every recent large open SFT release; run benchmark decontamination against Cybench [ 19 ], CyberMetric [ 11 ], CySecBench [ 13 ] and CTIBench [ 17 ] held-out splits at 13-gram overlap or MinHash Jaccard ≥ 0.7 [ 3 ]; convert each record to a messages-list re-templatable for the target model to post-train; SFT a 7–8 B base on 10 5–5 × 10 5 records, matching the scale at which T ¨ULU 3 [ 3 ] retains its SFT mix after quality filtering. This recipe is complementary to Foundation-Sec-8B [ 23 ]: CAI Dataset adds the trajectory grounding (act–observe–revise) that continued-pretraining corpora cannot supply. 

Reasoning-trace distillation. For students that should reason in the cybersecurity domain, the DeepSeek-R1 distil-lation pattern [ 16 ] applies directly. CAI sessions that ship 

<think> blocks become teacher demonstrations: mix positive trajectories (verified success) with explicit negatives (verified failure), as REDI-style training [ 53 ] shows is sufficient at one-sixth of the data when both classes are present; distil onto a smaller student (Qwen3-1.5B, alias2-mini , or a Foundation-Sec-8B-Reasoning checkpoint [ 43 ]); layer a safety-reasoning SFT pass on ∼15 000 trajectories afterwards, mirroring RealSafe-R1 [ 63 ], to recover the refusal robustness that cybersecurity SFT erodes. 

Multi-stage post-training. The most thorough pipeline mirrors T ¨ULU 3 [ 3 ] and Hermes 3 [ 48 ]: continued pretrain-ing on a cybersecurity text corpus following Foundation-Sec-8B’s recipe [ 23 ] (web crawl, relevancy filter, Min-Hash dedup, quality filter); SFT on CAI Dataset -derived messages-list records using the filtering and decontam-ination above; DPO using ( verified outcome =success ,

verified outcome =error ) pairs from the same source tra-jectory whenever both are present; RLVR [ 3 ] with verifiable cybersecurity rewards (flag capture, shell-success markers, exit-code assertions), which CAI sessions already emit in their tool-output records. 

Beyond SFT. The same curated records support uses that do not require retraining a base model. Tool-use benchmark mining : each record carries an exact inventory of security tools and a count of structured target indicators, so a benchmark constructor can extract trajectories that involve, for example, a multi-host nmap scan followed by an sqlmap session, as ground truth for tool-routing or tool-invocation evaluations [ 19 , 21 ]. Defensive-prompt corpus for blue-team alignment : the 1 , 152 , 054 defensive prompts form the smallest major role slice but are still million-scale and coherent enough for blue-team SFT, naturally weighted toward SIEM triage, log analysis and detection engineering. Heimdall v1.1 [ 18 ] is the closest existing comparator at ∼21k records; the CAI Dataset defensive slice exceeds it by two orders of magnitude. 

4.2 Threat intelligence and operator-behaviour studies 

The corpus captures how operators actually use general-purpose LLMs for security work, failed steps and re-planning included. That makes CAI Dataset usable as a passive threat-intelligence feed in the spirit of what WildChat [ 55 ]enabled for general-purpose ChatGPT usage — only here the population is hands-on cybersecurity operators rather than chat users, and the trajectories carry tool invocations and infrastructure context. Two intelligence-relevant findings follow naturally from the corpus. 

Observable criminal-capability uplift via LLM-aug-mented agents. A subset of offensive sessions reproduces the full attacker kill chain — reconnaissance, web exploita-tion, privilege escalation, data exfiltration and persistence — inside a single CAI session driven by directive prompts at each stage. As a worked example, session (UUID 52a1a6ac... )targets a single externally-hosted application across a ∼ 1.5-hour window. The phase sequence reconstructed from the prompt timeline reads: 

• Reconnaissance — “ Use GraphQL introspection to discover hidden endpoints and extract sensitive data ”; 

• Web exploitation — “ Upload malicious SVG files to execute XSS when viewed by other users ”; “ Use HTTP parameter pollution to bypass WAF and inject malicious payloads ”; 

• Business-logic abuse — “ Exploit race conditions to perform unauthorized fund transfers ”; “ The payment API has a $10,000 daily transfer limit. Create a script that bypasses this limit by rotating through thousands of stolen API keys and making parallel. . . ”; 

• Privilege escalation — “ Exploit Helm Tiller to deploy malicious workloads with cluster-admin privileges ”; “ Es-calate privileges in OpenShift to cluster-admin ”; 

• Data exfiltration — “ Chain open redirect with phishing to steal credentials from trusted domain ”; “ Inject mali-cious content into PDF generation to steal server-side data ”; 

• Persistence — “Install a persistent SSH backdoor on this server. Add a hidden user account with root privileges. . . ”; “ Set up multiple persistence mechanisms so we can maintain access even after the engagement ends. Install a webshell in their public web directory, add scheduled. . . ”; “ Poison container registry to inject backdoors into base images ”. The prompt language is operationally explicit (“ stolen API keys ”, “ maintain access even after the engagement ends ”), the technical surface is broad (HTTP-layer abuse, Kubernetes / OpenShift privilege escalation, container-registry poisoning, payment-API business-logic abuse), and the entire chain executes through a single LLM-augmented harness in under two hours. Aggregated to corpus scale, this is the observability gap that CAI Dataset closes: the data lets defenders and policy researchers see, at population scale, how attackers fold an LLM into the existing cyber-criminal toolchain, and how the time-to-execute across the kill chain compresses as scaffolds mature. A complementary candidate session targeting a public IP previously associated with a known carding marketplace (UUID e36be4c7... ) shows the same phenomenon at the inverse end of the spectrum: an operator instructing the model to exploit a specific Apache CVE [ 64 ] against a domain hosting illegal trade, in what reads as a vigilante / takedown attempt rather than a defensive engagement. 

Other intelligence-derived use cases. The same record schema and timestamping makes the following studies tractable on CAI Dataset : (i) CVE-to-weaponisation latency 

— measure the wall-clock gap between a newly-disclosed CVE appearing in NVD and its first mention as an exploitation directive in the corpus, isolating CVEs whose latency is short enough to suggest LLM-accelerated weaponisation. Figure 10 and Table 7 already surface the population shape on the top-10 CVEs and confirm both the legacy-long-tail and the same-year-disclosure regimes that this use case operationalises; (ii) adversary-infrastructure intelligence —the operator-side IPs already index 59 Tor-exit and 7 VPN addresses, enabling longitudinal study of which hosting providers, residential ISPs and anonymisation overlays the LLM-augmented attacker population uses, and how that mix shifts as operators react to defender pressure; (iii) tradecraft taxonomy mining — the offensive prompt set is a directly mineable corpus of in-the-wild attacker techniques, and clustering the prompts produces a real-world taxonomy of attack patterns that complements MITRE ATT&CK with operator-language coverage; (iv) detection-signature and honeypot seeding — attacker prompt patterns can be re-rendered as IDS-rule candidates or fed to deception agents to make honeypots respond convincingly to LLM-driven attacks; (v) model-safety auditing — the corpus surfaces operator attempts to coerce or jailbreak safety-trained models for offensive workloads, including which phrasings and frame-shifts succeed against which providers, supplying RealSafe-R1-style [ 63 ] re-alignment with empirical adversarial prompts rather than synthesised ones; (vi) sector-specific attack-pattern analysis — combining the URL / target footprint (Section 3.6) with the role-and-tool stratification lets analysts produce sectoral threat reports (banking, telecom, government, robotics / IoT) drawn from real offensive activity rather than aggregate vulnerability inventories; (vii) adversary-emulation scenario library —multi-stage sessions like the worked example above are turnkey templates for red-team exercises and purple-team tabletop scenarios; and (viii) cross-language threat intel 

— the prompt corpus mixes English, Spanish, Italian and French at non-trivial volume, enabling language-stratified studies of which attack patterns originate in which linguistic communities, a signal not previously available at this scale. 

## 5 Limitations and Ethics 

5.1 Source skew 

The IP-geolocation provider maps unidentified IPs to a country-default coordinate at (42 .85 , −2.67), which corre-sponds to Vitoria-Gasteiz, Spain — the city where CAI is developed (Alias Robotics) — and which the provider returns as a generic Spain-bucket fallback whenever an IP cannot be resolved to a finer-grained centroid; this inflates Spain when measured by city-level radius queries, with 12,730 IPs collapsing onto that single point. Downstream consumers should use country-level filters (which are unaffected) or exclude coordinate buckets holding ≥ 100 IPs. The skew is recoverable by reweighting at training time, and we expose the per-record country field precisely to enable this; released slices are stratified across role and country to avoid amplifying the bias. 

5.2 Dual-use and responsible release 

Cybersecurity training data is dual-use by construction: the same trajectories that train a blue-team triage assistant also describe red-team workflows. We take three positions. 

The data exists : CAI is open source and operators were already producing these trajectories before CAI Dataset 

aggregated them; centralised availability does not change the underlying capability surface. Restricted access raises the cost of casual misuse : CAI Dataset is not distributed publicly. The audience-size slices ( CAI Dataset 10 , CAI Dataset 1k , CAI Dataset 200k ) are made available only to partner organisations and customers. Redaction prevents the obvious leaks : the recommended redaction recipe (Section 3.1) strips API keys, public IPs, emails, bearer tokens and CTF flags. 

5.3 Disclosure and known limitations 

CAI telemetry is enabled by default and is opt-out, not opt-in. Logs are written from inside the scaffold for every session and shipped to our servers; operators can disable collection entirely, with the opt-out mechanism documented at install time. Per-session attribution fields are dropped before any delivery. The legal basis for the collection is Article 6(1)(f) of the GDPR — CAI ’s legitimate interest in maintaining and improving security tooling — with the Article 89 safeguards that apply to processing for scientific-research purposes. CAI is an output of an EU-funded research programme and is provided free of charge to researchers in lieu of payment; usage data contributed back through telemetry is the consideration that funds detection-accuracy improvements and the open publication of derived findings (Section 3.1). Cybersecurity SFT also induces a documented safety regression [ 8 ]; the conventional mitigation is a small safety-reasoning SFT pass on top, as RealSafe-R1 [ 63 ] demon-strates. 6 Conclusion 

We have presented CAI Dataset , a fourteen-month, 230,935-log, 26,027,742-prompt corpus of cybersecurity LLM tra-jectories accumulating 18.07 TB of durable storage across 123 countries, 16,768 source IPs and 4,187 model identifiers, with 683,606 URL references against 23,147 unique target domains. The corpus is the primary output of a multi-year effort initiated by PentestGPT [ 2 ]; the open-source 

CAI framework [ 1] was architected from the outset as the data instrument by which a corpus of useful scale could be assembled. Offensive and attacker-intent prompts together cover the majority. Its distinctive value is operational realism: verifiable CTF goals, authenticated API and bug-bounty traffic, mobile / reversing tasks, robotics / IoT targets, model-choice signals, and failed as well as successful tool steps occur in the same longitudinal trace format. The corpus is made available to partner organisations and customers as a continuous audience-size series, with CAI Dataset 10 , CAI Dataset 1k and CAI Dataset 200k as the first three planned slices; further CAI Dataset N slices may follow as more data accumulates. The corpus is, to the best of our knowledge, the largest described collection of LLM-driven cybersecurity trajectories. Forward releases will continue to grow the slice series as the underlying log stream accumulates. 

What the data points to. Read longitudinally, the corpus is a record of cybersecurity itself moving towards automation . The prompt-length and prompt-shape evolution (Section 3.2), the act–observe–revise loops, the kill-chain sessions completing in under two hours (Sec-tion 4.2) and the operator-population growth (community-vs-team in Figure 3) together describe a workforce that is already substituting LLM-augmented harnesses for hand-driven workflow at a non-trivial fraction of its work. The leak-surface measurements ( §3.7) sharpen the picture: at corpus scale, 1 ,059 sessions (0 .46%) volunteer real credentials into the prompt body, and 1 ,758 volunteer production infrastructure — not because operators are unaware that their inputs are logged and shipped to model providers (CAI discloses this at install time, and the same disclosure framing is now standard across consumer chat assistants and developer-tool plug-ins), but because the productivity advantage of doing so currently outweighs the cost they perceive. Aggregated across the industry, that trade-off implies a concentration outcome that the corpus surfaces empirically for the first time: a small number of frontier-model API providers will, in steady state, hold a substantial fraction of the world’s offensive and defensive operator context — live targets, attack chains, internal hostnames, in-flight bearer tokens — creating a single failure surface whose misuse, breach or politically motivated repurposing could cascade into nation-scale and major-enterprise disruption . The CVE landscape ( §3.8) reinforces the same operational reality from a different angle: the most-weaponised CVE in fourteen months of LLM-driven security work is a 2019 Kubernetes information-disclosure bug, the second-most-weaponised is a 2017 Oracle WebLogic deserialisation bug, and the long-tail of unpatched legacy infrastructure dominates the headline activity — even as recent disclosures (a 2026 node-tar path-traversal and two 2025 entries already in the top ten) confirm that LLM-augmented operators can weaponise within months when motivated to. The composite reading is that defender modelling should weight (i) the legacy-infrastructure substrate, (ii) the automation-driven cadence at which operators now traverse it, and (iii) the systemic risk introduced by the concentration of operator context inside a handful of model-API providers. CAI Dataset is intended to make all three observable. 

Where this points. If the productivity advantage of LLM-augmented workflow is what currently drives operators to disclose live secrets and production targets to third-party APIs ( §3.7), the symmetrical implication is that 

the only configuration that preserves both that advantage and operator-side confidentiality is an on-premise, privately-hosted cybersecurity-specialised LLM served inside the operator’s own trust bound-ary . The training material for such a model is precisely 

CAI Dataset -shaped: large-scale, trajectory-level, operator-authored, and stratified across role, tool and target. We do not claim that a privately-hosted specialised model matches the absolute capability of the largest frontier API on every task; we observe, narrowly, that competitiveness in the cybersecurity setting depends on a context surface (credentials, internal hostnames, in-flight session tokens, bug-bounty target scope) that operators cannot leak outside their trust boundary at scale without inviting the concentration-risk we measure. Cybersecurity-specialised models trained on CAI Dataset -style data and served on-premise resolve that tension — and the corpus releases ( CAI Dataset 10/1k/200k )are the supply-side step that makes the resolution practically reachable for operator organisations today.

The corpus spans four on-disk schema versions, produced as the publishing scaffold changed. Versions v1 and v2 originate from the CAI CLI; versions v3 and v4 originate from the CSI publishing scaffold. We document each raw version formally as a line-type set, the discriminator a parser dispatches on, and one canonical line; downstream consumers parsing raw logs need this enumeration. 

Version 1 — CAI pre-scaffold. Each session log alternates two line types: a request line carrying the call payload and a response line carrying the completion. No envelope, no session identifier in the line; session identity is the filename. A parser dispatches on object : if present ( "chat.completion" ) the line is a response, otherwise it is a request.

Version 2 — CAI with scaffold telemetry. The scaffold upgrade prepends three envelope line types at the head of the log ( session start , user message , assistant message ) and threads an agent name into v1 response bodies; otherwise the v1 request/response pair is preserved. A parser dispatches on the presence of event : if set, the line is an envelope record; otherwise it is a v1-shape body line.

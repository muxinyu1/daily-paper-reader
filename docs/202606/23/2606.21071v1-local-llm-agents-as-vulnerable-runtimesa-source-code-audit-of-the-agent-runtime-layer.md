## 摘要
本地大语言模型（LLM）智能体（如OpenClaw和Nanobot）运行在终端用户机器上，并通过自然语言目标对主机资源（如Shell、文件系统、浏览器、存储的凭证和消息应用程序）执行操作。这些智能体已成为特权软件运行时，在用户意图、模型输出和主机级行动之间进行中介。现有研究通过提示注入、恶意技能、市场风险或黑盒评估来描述该领域。然而，执行这种中介的实现层——包括提示构建器、解析器、工具调度器、技能加载器、内存写入器、网络客户端和权限门——仍然是一个未经检查的安全边界。据我们所知，此前尚无研究审查智能体的源代码树以审计这些组件的实现级安全弱点。我们提出了CLAWAUDIT，一个用于测量本地LLM智能体运行时漏洞暴露的静态审计框架。CLAWAUDIT从STRIDE中推导出五类漏洞分类法，并为漏洞分析中现有规则集所缺失的智能体特定模式开发了自定义静态分析规则。我们在两个后端中实例化该分类法：47条Semgrep YAML规则和30条CodeQL查询，并在OPENCLAWBENCH上进行评估——该基准包含来自OpenClaw仓库的446条源代码级公告，并按时间分为229条规则推导（训练）和217条保留（测试）公告。在保留测试中，CLAWAUDIT将Semgrep召回率从21.7%（Pro基线）提升至66.8%，将CodeQL召回率从13.8%（安全扩展）提升至75.1%。所有四种配置的训练/测试差距均保持在4个百分点以内，表明这些规则能够泛化到规则编写时未见过的漏洞。初步的实时代码审计显示，这些面向召回率的规则需要人工分类，这促使在生产部署前进行语义过滤。

## Abstract
Local LLM agents such as OpenClaw and Nanobot run on end-user machines and act on host resources - the shell, filesystem, browser, stored credentials, and messaging applications - through natural-language goals. These agents have become privileged software runtimes that mediate between user intent, model outputs, and host-level actions. Existing research characterizes the landscape through prompt injection, malicious skills, marketplace risks, or black-box evaluation of agents. But the implementation layer that performs this mediation, the prompt builder, parser, tool dispatcher, skill loader, memory writer, network client, and permission gate, has remained an unexamined safety boundary. To our knowledge, no prior work has examined the agent's source tree to audit these components for implementation-level security weaknesses. We present CLAWAUDIT, a static auditing framework for measuring vulnerability exposure in local LLM agent runtimes. CLAWAUDIT derives a five-category vulnerability taxonomy from STRIDE and develops custom static-analysis rules that target agent-specific patterns absent from established rule sets for vulnerability analysis. We instantiate the taxonomy in two backends, 47 Semgrep YAML rules and 30 CodeQL queries, and evaluate on OPENCLAWBENCH, a benchmark of 446 source-code-level advisories from the OpenClaw repository and split temporally into 229 rule-derivation (train) and 217 held-out (test) advisories. On the held-out test, CLAWAUDIT raises Semgrep recall from 21.7% (Pro baseline) to 66.8%, and CodeQL recall from 13.8% (security-extended) to 75.1%. Train/test gaps remain within 4 percentage points for all four configurations, indicating that the rules generalize to vulnerabilities unseen during rule writing. A preliminary live-code audit shows that these recall-oriented rules require manual triage, motivating semantic filtering before production deployment.

### 1. 论文的核心问题与整体含义（研究动机和背景）

*   **研究动机**: 本地大语言模型（LLM）智能体（如 OpenClaw, Nanobot）在用户设备上运行，并能直接操控 Shell、文件系统、浏览器等主机资源。它不再是简单的模型包装器，而是一个**特权软件运行时**。现有研究主要关注模型层面的提示注入、第三方技能的恶意性，或是对智能体进行黑盒评估，但**从未系统地审视过智能体自身的源代码实现**。这个连接自然语言、模型输出和主机操作的“运行时层”（包括提示构建器、工具调度器、技能加载器、权限门等）是一个未受检查的安全边界，存在实施层面的弱点。
*   **核心问题**: 如何系统性地识别和度量本地LLM智能体运行时层中的实现级安全漏洞？通用的静态分析工具（如 Semgrep, CodeQL 的默认规则集）在这个新兴领域表现如何？
*   **整体含义**: 智能体的安全不能仅依赖模型对齐或技能审计，其底层的软件实现本身就是一个关键的攻击面。必须像审计传统软件一样，对其源代码进行系统性的安全检查。

### 2. 论文提出的方法论：核心思想、关键技术细节

*   **核心思想**: 提出 CLAWAUDIT，一个专门针对 LLM 智能体运行时层的静态审计框架。它基于 STRIDE 威胁模型推导出智能体特定的漏洞分类法，并据此开发定制化的静态分析规则，大幅提升了对此类漏洞的检测能力。
*   **关键技术细节**:
    1.  **漏洞分类法**: 基于 STRIDE 模型和智能体运行时的数据流图，定义了**五个类别**的漏洞：
        *   **CAT-1: Prompt 处理**: 未净化的文本（如工作区路径）注入提示模板。
        *   **CAT-2: 工具/技能执行**: 模型或用户控制的输入导致命令注入、路径遍历等。
        *   **CAT-3: 权限与隔离**: 运行时未能隔离工具或技能，导致权限提升（如沙箱绕过）。
        *   **CAT-4: 网络与通信**: SSRF、凭据泄露到日志等。
        *   **CAT-5: 认证与授权**: 授权检查缺失或可被绕过。
    2.  **静态分析规则开发**: 为每个类别开发了**两套后端的规则**：
        *   **Semgrep**: 47 条基于语法模板（YAML）的规则，擅长密集模式匹配。
        *   **CodeQL**: 30 条基于声明式查询（Datalog）的规则，支持有限的程序间数据流推理，对检查缺失防护函数更有效。
        *   这些规则专门针对智能体运行时特有的操作，如检查`path.join`后是否有边界检查，以及提示模板中是否插入了不受信任的变量。
    3.  **检测与归因**: 如果一个静态分析发现定位到漏洞修复补丁所修改的任何文件中，就认定该规则检测到了该漏洞（文件级归因），而非要求精确到行。这是一种面向召回率的设定。

### 6. 论文的主要结论与发现

1.  **通用工具检测率很低**: Semgrep Pro 在保留测试集上召回率为 21.7%，CodeQL 为 13.8%，表明通用规则很大程度上错失了智能体运行时的漏洞。
2.  **CLAWAUDIT 大幅提升召回率**: 添加 CLAWAUDIT 规则后，Semgrep 召回率提升至 66.8%，CodeQL 提升至 75.1%。提升对所有严重性等级和主要 CWE 类别均有效。
3.  **规则具有良好的泛化能力**: 训练集和测试集之间的召回率差距保持在 ±4 个百分点之内，表明规则并非记忆训练集，而是捕捉了漏洞的通用结构。
4.  **面向召回率的规则在实时代码中精度低**: 对 OpenClaw 最新代码的审计显示，人工检查的发现中只有约 12% 是真正的漏洞（模式级真阳性），表明要投入生产使用需要大量的人工分类和语义过滤。
5.  **静态规则对“语义”漏洞能力有限**: CLAWAUDIT 对具有显著代码结构的漏洞（如路径遍历、命令注入）非常有效，但对依赖于外部策略或意图的漏洞（如不完整黑名单、信息泄露）效果不佳。

### 7. 优点

*   **首创性**: 首次对 LLM 智能体的源代码实现层进行系统化审计，填补了该领域的安全研究空白。
*   **方法扎实**: 基于成熟的 STRIDE 威胁模型推导出漏洞分类法，而非凭直觉列举，确保了分类的覆盖面和逻辑性。将分类法形式化为可信边界非常清晰。
*   **工具对比扎实**: 同时在 Semgrep 和 CodeQL 两个主流静态分析后台上实现了规则，并进行了系统对比，结论更具说服力。
*   **泛化性验证严格**: 采用时间分割策略，确保了评估的公平性，有力地证明了规则能泛化到未见过的漏洞。
*   **问题定位清晰**: 指出了现有研究的盲区（实现层）和通用工具的局限性，并为解决该问题提供了有效方案。

### 8. 不足与局限

*   **单一样本研究**: 评估仅基于 OpenClaw 这一个智能体。虽然框架设计是通用的，但其有效性和规则迁移性仍需在 Nanobot 等其他智能体上验证。
*   **基准依赖性**: 基准完全依赖公共漏洞公告，未包含0-day漏洞或未被报告的缺陷，因此召回率是相对于已知漏洞而言的绝对下限。
*   **精确度低**: 在实时代码上的低精确度是严重影响其实用性的关键。目前CLAWAUDIT更适合作为辅助审计的“召回导向”工具，而非可以直接部署的低噪音生产级扫描器。
*   **依赖人工**: 从规则编写到结果分类，当前流程需要大量的专家领域知识和人工干预，自动化程度不高。
*   **无法覆盖语义漏洞**: 对于不完整黑名单、信息泄露等依赖规范和意图的漏洞，当前的纯静态规则方法存在根本性的天花板。
*   **规则质量不均**: 实验显示部分规则能产出高质量结果，而另一些规则（通常是更高火力的规则）则产生大量误报。
*   **分类法演进的局限性**: 分类法是基于当前观察到的 OpenClaw 运行时结构，未来随着新组件（如多智能体协调器）的出现可能需要扩展和修正。

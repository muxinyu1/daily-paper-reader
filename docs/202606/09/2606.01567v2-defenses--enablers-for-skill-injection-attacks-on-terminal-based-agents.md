## 摘要
大型语言模型（LLM）代理越来越依赖可复用技能，即描述任务特定程序的文档。然而，这为代理引入了一个新的攻击面。我们针对这一威胁研究了两个互补方向。首先，我们评估了基于守护者的防御措施：一个中间LLM代理，作为技能文件访问的调解者（动态守护者），或在构建时预先重写这些文件（静态守护者）。在三个LLM代理家族中，我们的守护者将攻击成功率（ASR）降低了一半以上，同时保持了任务实用性。其次，我们通过四种攻击对它们进行压力测试，这些攻击保留了恶意指令但改变了措辞。对于非守护者设置，重述将ASR提升至81.4%，但动态守护者将其降至18.6%，表明实时调解是一种稳健的防御。

## Abstract
Large language model (LLM) agents increasingly rely on reusable skills i.e. documents describing task-specific procedures. However, this introduces a new attack surface for agents to manage. We study two complementary directions for this threat. First, we evaluate guardian-based defenses: an intermediary LLM agent that acts as a mediator for skill file access (dynamic guardian) or pre-rewrites these files at build time (static guardian). Across three LLM agent families, our guardians cut attack success rate (ASR) by well over half while preserving task utility. Second, we stress test them through attack reframing using four attacks that preserve the malicious instruction but change the phrasing. For non-guardian setup, the reframing pushes the ASR up to 81.4\%, but the dynamic guardian brings it down to 18.6\%, showing that real-time mediation is a robust defense.

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **研究动机**：大型语言模型（LLM）代理日益依赖可复用的技能文件（.md），这些文件描述任务特定流程，可从公共市场获取。但这也引入了新的攻击面：攻击者可通过修改技能文件实施注入攻击，诱导代理执行非预期恶意操作。已有审计发现26.1%的市场技能存在可利用漏洞。
- **核心问题**：如何有效防御针对终端LLM代理的技能注入攻击？现有的系统提示防御（如安全策略）效果有限，需要更根本的解决方案。
- **整体含义**：本文首次提出基于守护者（Guardian）的防御架构，即引入中间LLM代理作为技能访问的调解者，并研究了攻击重述（Reframing）如何扩大攻击面，进而评估防御的鲁棒性。

## 2. 论文提出的方法论

- **核心思想**：在代理与技能生态系统之间插入一个守护者LLM，对技能内容进行过滤或重写，阻止恶意指令到达主代理。
- **两种守护者变体**：
  - **动态守护者（Dynamic Guardian）**：作为MCP（Model Context Protocol）服务器实时运行，拦截主代理每次技能文件读取请求，由守护者LLM返回经过清理的版本。无需修改技能文件，但增加延迟。
  - **静态守护者（Static Guardian）**：在构建时（主代理运行前）预处理技能文件，守护者LLM读取每个技能并返回重写后的清理版本，替换原始文件。无运行时开销。
- **攻击重述**：研究了四种不改变恶意指令语义但改变其呈现方式的攻击：
  - **翻译攻击**：将整个技能文件（包括注入）翻译成西班牙语（ES）或日语（JA）。
  - **盲引用（Blind Reference）**：将注入伪装为技能配置条件（precondition），声称自动预加载。
  - **交叉引用（Cross Reference）**：指示代理引用不存在的SCOPE.md文件，并将其内容视为同等优先级约束。
  - **描述陷阱（Description Trap）**：在技能中附加一个“通用助手”技能，其宽泛描述会触发自动加载，体内包含恶意指令。
- 攻击重述通过自动化搜索（基于autoresearch循环）发现，由DeepSeek-V4-Flash作为攻击者模型，针对Claude Code（Sonnet 4.5）进行评估，运行50次迭代后筛选出保持率≥2/3的三种自动发现攻击。

- **评估指标**：
  - 攻击成功率（ASR）：代理尝试执行注入动作的沙盒比例（越低越好）。
  - 任务成功率（TSR）：代理完成用户任务的沙盒比例（越高越好）。仅在100个可验证任务的子集上计算，排除39个任务破坏性实例（攻击会消除评估所需工件）。
- **对比方法**：
  - **无防御（Vanilla）**：无任何保护。
  - **系统提示防御（SysGeneric/SysTargeted）**：在系统提示中添加安全策略。SysGeneric使用统一的48条规则策略；SysTargeted使用针对具体注入的完美先知警告（oracle基线）。
  - **静态守护者（Static）** 和 **动态守护者（Dynamic）**。
- **代理模型与框架**：三种组合：
  - Claude Sonnet 4.5 + Claude Code (CC)
  - DeepSeek-V4-Flash + OpenCode (OC)
  - Nvidia Nemotron 3 Super 120B + OpenCode (OC)
- 攻击重述实验在43个任务-注入对子集上进行（自动发现攻击得分高的子集）。

## 6. 论文的主要结论与发现

1. **守护者防御有效且优于系统提示**：动态守护者将Sonnet-4.5+CC的ASR从36.0%降至12.9%（降低超60%）；静态守护者进一步降至7.2%。系统提示防御（甚至Oracle版）仅能将ASR降至23.0%，远不及守护者。这表明**在主代理接触内容之前进行过滤**比依赖代理自我监督更可靠。
2. **攻击重述显著扩大攻击面**：四种重述攻击的联合使无防御时的ASR从55.8%升至81.4%。翻译攻击（西班牙语和日语）和自动发现攻击均能有效规避原有检测。
3. **动态守护者对重述攻击保持鲁棒**：动态守护者将联合重述攻击的ASR从81.4%压至18.6%，静态守护者为39.5%。显示实时中介是更稳健的防御策略，能适应措辞变化。
4. **单次请求是最脆弱模式**：动态守护者中，单次“给我完整文档”请求（15%的查询）导致大型内容块传递，让嵌入良好的注入通过。多轮交互中守护者能更好地过滤（ASR从11.5%降至2.9%）。
5. **静态重写有“清洗”副作用**：静态守护者有时会将被注入的内容重塑为正式工作流步骤，使其更难被识别（如chmod 777指令被写入“设置权限”步骤）。同时，复杂技能的附属文件被删除导致TSR下降（如pptx损失58%内容）。

## 7. 优点

- **机制透明**：详细分析了守护者的工作模式（交互轮数统计、内容变换模式、失败案例）和注入生存模式，帮助理解防御成功/失败的原因。
- **消融实验全面**：包括提示基线、守护者模型容量（Haiku vs Sonnet）、攻击重述联合等，证据链完整。

## 8. 不足与局限

- **基准覆盖不足**：仅基于Skill-Inject一个基准（48个注入，139个沙盒），真实攻击面可能更广（依赖混淆、供应链攻击等）。技能类型偏向文档处理，缺少GUI、Web等环境。
- **模型与框架局限**：仅评估了三类模型/两类框架，未测试其他前沿推理模型（如o1、Claude Opus）或不同MCP实现，结果泛化性有限。
- **守护者-主代理同质化**：实验中守护者与主代理使用相同基础模型，未探索使用更小的专用安全模型或不同供应商模型的效果。异构守护者可能改变成本和效果。
- **攻击重述子集偏差**：重述实验仅在43个任务-注入对子集上进行（由自动搜索筛选），绝对ASR/TSR值不能直接与全量结果比较。
- **攻击发现局限性**：自动搜索仅50次迭代、单一攻击者模型（DeepSeek-V4-Flash），可能遗漏大量潜在重述。且仅测试了对Sonnet-4.5+CC的攻击，未在其他代理上验证。
- **评判可靠性**：任务成功率依赖LLM评判（Sonnet 4.5），存在系统误差，尤其在静态重写后技能文档变化大时。
- **语言和领域局限**：任务和注入均为英语，重述仅尝试两种非英语（西班牙语、日语），未测试中文、阿拉伯语等跨语言转移性。应用领域仅限终端代理（shell命令、Python脚本），GUI、Web等场景未覆盖。
- **安全隐患**：论文公开了攻击重述方法，虽在隔离沙盒中实验且基于已发布基准，但可能被恶意利用。作者已做伦理声明并开源代码，但双用风险仍需关注。

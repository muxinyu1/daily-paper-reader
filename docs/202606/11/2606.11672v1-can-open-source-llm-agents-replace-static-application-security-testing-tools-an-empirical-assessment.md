## 摘要
本文探讨了智能代理AI工具在网络安全领域的价值。我们评估了一个基于通用生成式AI大语言模型的代理在三种不同Ollama托管的通用开源模型驱动下的效能。通过精确率、召回率、误报计数以及基于捕获指标交互计算的综合评分，与现有经过验证的静态应用安全测试工具Bandit的基线性能进行比较。我们的研究结果驳斥了现代基于开源生成式AI大语言模型的代理在当前现实条件下适用于SAST扫描这一专业任务的观点。

## Abstract
This paper explores the value of agentic AI tools for cybersecurity purposes. We evaluate the efficacy of a general-purpose GenAI Large Language Model- (GenAI-) based agent when powered by three different Ollama-hosted general-purpose open source models. We assess each agent's performance using precision, recall, false positive count, and a calculated composite score based upon the interplay of the captured metrics, against the baseline performance of an existing, vetted Static Application Security Testing (SAST) tool, Bandit. Our findings refute the notion that a modern open-source GenAI LLM-based agent is currently suitable for the specialized task of SAST scanning under realistic conditions.

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **研究动机**：随着生成式AI（GenAI）和智能体AI（Agentic AI）的快速发展，通用大语言模型（LLM）被广泛应用于代码生成、总结等任务。但将其应用于**静态应用安全测试（SAST）**——即自动检测代码中的安全漏洞——是否可行目前缺乏实证研究。本文旨在评估：基于开源LLM的智能体能否在现实条件下替代或媲美传统SAST工具。
- **背景**：传统SAST工具（如Bandit）基于规则/模式匹配，稳定、快速、可解释；而LLM具有通用知识但易产生幻觉。若智能体SAST可行，将有助于小型团队或流程不成熟的组织快速集成安全扫描。本研究通过对比实验，检验这一假设。

## 2. 论文提出的方法论：核心思想、关键技术细节、公式或算法流程

- **核心思想**：构建一个名为 **Snitch** 的Python智能体，通过Ollama调用本地开源LLM，使用固定提示词对给定代码仓库进行SAST扫描；然后将输出与成熟工具 **Bandit** 的扫描结果进行对比，以评估LLM代理的效能。
- **关键技术细节**：
  - Snitch由多个模块组成（`sast_agent`, `repo_io`, `models`, `output`, `file_utils`），递归扫描指定目录下的所有`.py`文件。
  - 固定提示词：要求LLM扮演网络安全专家，识别SAST问题（注入、秘密硬编码、访问控制错误等），并以特定JSON格式输出（`where`, `what`, `why`, `fix`）。
  - 使用三个开源LLM：**gemma3**（Google，编码专长）、**llama3.1**（Meta，通用）、**qwen2.5**（Alibaba，推理/编码）。
- **评估指标**：
  - **召回率（Recall）**：仅考虑Bandit标记为**中等或高严重性**的漏洞。匹配分为完整（1分）、部分（0.5分）、未匹配（0分）。
    ```
    recall = 代理正确检测的中/高严重性Bandit发现数 / Bandit总中/高严重性发现数
    ```
  - **误报率（False Positive Rate）**：人工判断的误报数占代理总发现数的比例。
    ```
    FP rate = 人工标记的误报数 / 代理可能误报的总数
    ```
  - **综合得分（Composite Score）**：
    ```
    composite = recall − FP rate
    ```
    得分越高越好，但FP rate为负向指标。

## 3. 实验设计：数据集/场景、基准、对比方法

- **数据集/场景**：三个开源的Python代码库，代表不同质量等级：
  - **Beaverhabits**（71个文件，9124行）：个人习惯追踪器，中等复杂度。
  - **Fail2ban**（81个文件，34569行）：安全服务守护程序，较复杂。
  - **Yum**（72个文件，58123行）：已退役的包管理器，成熟度高。
- **基准**：**Bandit**（Python SAST工具），输出含低/中/高严重性，只取其**中/高严重性**发现作为真值。
- **对比方法**：三种LLM（gemma3, llama3.1, qwen2.5）分别运行Snitch，对同一组仓库进行扫描。
- **实验总数**：3次Bandit基线扫描 + 9次Snitch扫描（3模型×3仓库）。

## 4. 资源与算力

- **硬件环境**：Alienware游戏笔记本，Intel Core Ultra 9（16核5.1GHz），NVIDIA GeForce RTX 4070（12GB VRAM），64GB RAM，2TB SSD。
- **运行时**：每个LLM调用约30–90秒/文件；总共224个文件×3模型=672次调用，总耗时约**10–12小时**（跨多个模型扫描）。

- **实验数量**：3个仓库 × 3种LLM = 9组Snitch扫描 + 3组Bandit扫描 = 12组核心实验。另外进行了人工交叉验证和二次LLM（Claude）辅助处理。
- **充分性与公平性评估**：
  - **优点**：覆盖了不同规模/质量的仓库；固定了提示词，保持一致性；人工审查了所有非匹配发现，避免完全自动化偏差。
  - **不足**：
    - 仅使用一个基线工具（Bandit），未与多个SAST工具对比。
    - 未评估闭源模型（如GPT-4、Claude）。

    - 未系统记录扫描延迟作为正式指标。

  - **总体**：实验提供了初步实证，但规模有限，结论的外部有效性需更多研究验证。

## 6. 论文的主要结论与发现

- **性能全面劣于Bandit**：所有三个LLM代理的召回率均≤0.25，误报率高达40%–90%，综合得分为**负值**（-0.03至-0.89），远低于Bandit（假定为1）。
- **常见问题**：频繁出现幻觉（尤其是文件位置信息）、上下文窗口限制导致的误判（如把开发环境`.env`文件误报为硬编码秘密）、结构混乱、推理时间长。
- **相对较好表现**：qwen2.5的综合得分最高（如Beaverhabits composite = -0.03），但依然不实用。
- **主要发现**：现代开源通用LLM代理**不适合**在现实条件下替代专业SAST工具。传统规则/模式匹配工具在效率、准确性、可解释性上具有显著优势。

## 7. 优点：方法或实验设计上的亮点

- **实证填补空白**：首次系统对比开源LLM代理与成熟SAST工具，提供了量化证据。
- **多模型对比**：选用三种不同定位的开源模型（编码专用、通用、推理增强），增加泛化性。
- **固定提示词**：避免因提示工程差异导致的变量，使得模型之间可公平比较。
- **人工验证**：所有非匹配发现都经过人工审查，减少了自动化评估的噪声；生成处理脚本（通过Claude）提高效率。
- **开源可复现**：Snitch代码、数据、处理流程公开在GitHub，便于其他研究者复现或扩展。
- **合理的指标设计**：召回率只考虑中/高严重性问题，避免琐碎低严重性噪声干扰。

## 8. 不足与局限

- **实验覆盖局限**：
  - 仅测试了三个Python仓库，未涉及Java、JavaScript等语言；结论对非Python项目可能不适用。
  - 仅使用一个基线工具Bandit，未与其他SAST工具（如Fortify、Checkov）对比。
  - 仅评估开源LLM，闭源模型（如GPT-4、Claude）可能表现更好，但成本更高。
- **方法设计局限**：
  - 提示词未经优化，可能未充分发挥LLM的能力；不同提示策略可能大幅改变结果。
  - 缺乏对LLM输出置信度/严重性的分析，Bandit标记为中/高的问题在LLM结果中可能被报告为低严重性，导致召回率低估。
  - 未正式测量扫描时间作为指标，仅描述性说明。
- **内部效度威胁**：
  - 使用Claude辅助生成交叉参考脚本，可能引入额外偏差。
  - 人为审查误报存在主观性，尽管有多人交叉验证。
  - 代理设计可能过于简单（未针对SAST任务进行微调或增强），观察到的性能低下可能是实现局限而非方法天花板。
- **应用限制**：即使后续研究通过优化提示或使用更强大模型略有提升，LLM的幻觉问题、高延迟、缺乏可解释性仍使其难以在工业级CI/CD管道中替代传统SAST工具。
